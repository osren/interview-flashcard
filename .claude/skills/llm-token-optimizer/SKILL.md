# LLM Token 优化器

在**效果不明显下降**的前提下，系统性减少大模型应用的无效 Token 消耗。

适用两类场景：
1. **工程审计**：已有 LLM 应用（客服、判责、告警降噪、RAG、Agent），找浪费点并出改造方案
2. **方案设计**：Hackathon / 立项文档，输出「问题理解 → 创新解法 → 创造价值」

## 核心能力

- 识别 6 类 Token 浪费根因并量化
- 按「路由 → 上下文 → Prompt → 模型 → 输出 → 缓存」六层给出改造清单
- 结合 error-triage 等实战经验，输出可迁移的工程模式
- 给出 Before/After 指标与验证计划（eval 驱动，不靠感觉砍 prompt）

## 工作流程

### Step 0: 确认输入（缺了就问）

向用户确认：

| 项 | 说明 |
|---|---|
| **场景** | 工程审计 / Hackathon 提案 / 两者都要 |
| **目标系统** | 应用名、链路描述、或贴代码/Prompt/架构 |
| **约束** | 准确率下限、延迟要求、能否改模型、能否加缓存 |
| **现状数据** | 单次 input/output token、日调用量、月成本（有则精算，无则估） |

若用户只给赛题描述、没有具体系统 → 走 **Hackathon 模式**（Step 5）。
若有具体代码/Prompt → 走 **审计模式**（Step 1–4）。

---

### Step 1: Token 浪费诊断（6 类根因）

逐条扫描当前链路，标注浪费类型与严重度（高/中/低）：

| # | 浪费类型 | 识别信号 | 典型浪费占比 |
|---|---|---|---|
| 1 | **上下文过长** | 整段历史、整库文档、无关字段全塞入 | 30–60% |
| 2 | **Prompt 冗余** | system prompt 重复规则、过长范例、全量加载 | 15–30% |
| 3 | **RAG 不精准** | 召回 chunk 过长、无关文档、未做摘要分层 | 20–40% |
| 4 | **模型路由不当** | 简单分类/抽取也上大模型 | 40–70% 成本 |
| 5 | **输出不受控** | reason 过长、未约束 JSON、多余 markdown | 10–25% |
| 6 | **失败重试** | 工具调用失败、解析失败反复重调 | 视失败率 |

**诊断输出**：每张浪费类型填「现状 → 证据 → 严重度 → 改造方向」。

---

### Step 2: 六层优化框架（Tiered Token Governance）

按优先级从「零风险」到「需 eval 验证」排列改造项：

```
请求 → ①路由层 → ②上下文层 → ③Prompt层 → ④模型层 → ⑤输出层 → ⑥缓存层
```

#### ① 路由层（Router First）— 最高 ROI

**原则**：先分类，再决定「要不要 LLM、用哪套 Prompt、用哪个模型」。

| 手段 | 做法 | 参考模式 |
|---|---|---|
| 规则/前缀路由 | pathname、关键词、错误码先走确定性逻辑 | `error-triage/router.js`：pathname 归一 → 归属判别 → 有序规则 |
| 轻量分类器 | 小模型或 embedding 做意图/复杂度分类 | 简单任务 → 模板/小模型；复杂任务 → 大模型 |
| 兜底策略 | 路由失败时保守处理，而非反复重试 | 解析失败 → fallback 保守上报，不卡队列 |

**省 Token 原理**：大量请求在进大模型前被拦截或降级。

#### ② 上下文层（Context Compression）

| 手段 | 做法 |
|---|---|
| 字段裁剪 | 只传任务必需字段（如告警只传 url/errorCode/errorMsg，不传整段 response） |
| 历史摘要 | 多轮对话保留摘要 + 最近 N 轮，而非全量 history |
| RAG 分层召回 | 先召回标题/摘要 → 命中后再拉正文 chunk |
| 增量/sync | 只处理 delta，不重扫全量（如 discover-triage-urls sync 模式） |

#### ③ Prompt 层（Dynamic Prompt Assembly）

| 手段 | 做法 |
|---|---|
| 模块化 Prompt | `_base.md` + 按场景拼 `<key>.md`，不全量加载 |
| 去冗余 | 删重复规则、缩短范例、用结构化指令代替长段落 |
| 禁止硬编码阈值 | 调松紧改 prompt，但 prompt 本身要精简 |
| 启动时缓存 | prompt 文件启动读入缓存，避免每次 IO；**改完需重启** |

#### ④ 模型层（Model Tiering）

| 任务复杂度 | 推荐 |
|---|---|
| 分类/路由/抽取 | 小模型或规则 |
| 中等推理 | 中等模型 |
| 复杂推理/生成 | 大模型 |

**关键**：用 eval 数据集找 Pareto 点（token ↓ vs 准确率），不凭直觉降级。

#### ⑤ 输出层（Output Constraint）

| 手段 | 做法 |
|---|---|
| JSON Schema | 强制结构化输出，限制字段与 reason 长度 |
| temperature=0 | 判定类任务降低随机性，减少重试 |
| 截断策略 | 超长 reason 截断并记日志 |

#### ⑥ 缓存层（Result Reuse）

| 手段 | 做法 |
|---|---|
| 指纹缓存 | 同 url+errorCode+errorMsg 短 TTL 复用判定 |
| 语义缓存 | 相似 FAQ/告警 embedding 命中直接返回 |
| 批量评测 | `/eval` 批量跑 gold，调优不靠线上试错 |

### Step 3: 输出改造清单

为每个改造项填写：

```markdown
### 改造项 N：[名称]

- **针对浪费类型**：#1 上下文过长
- **现状**：每次请求携带 8k token 历史
- **方案**：保留摘要 200 token + 最近 2 轮
- **预期节省**：input token ↓ 70%
- **风险**：多轮指代可能丢失 → 用 eval 验证
- **验证指标**：准确率下降 < 1%；P95 延迟 ↓
- **落地成本**：低 / 中 / 高
- **优先级**：P0 / P1 / P2
```

按 **P0 先落地、P1 需 eval、P2 长期** 排序。

---

### Step 4: 效果预估与验证计划

#### 4.1 粗算公式

```
月节省 = 日调用量 × 30 × (单次 input 节省 + 单次 output 节省) × 单价
```

无真实数据时用区间估算，并标注假设。

#### 4.2 Eval 驱动验证（必须有）

参考 error-triage eval 闭环：

```
gold 数据集 → 改造前 baseline → 改造后对比 → 归因到具体 prompt/规则 → 迭代
```

| 指标 | 说明 |
|---|---|
| input/output token | 单次 P50/P95 |
| 准确率 / F1 | 任务效果不降 |
| 延迟 P95 | 副作用监控 |
| 月成本 | ROI |

**纪律**：宁可多耗一点 token 保准确率（如告警降噪「宁可误报不可误杀」），降本不能以核心指标为代价。

## 快速检查清单（审计时用）

对任意 LLM 调用点逐条过：

- [ ] 是否可以先路由/规则，不必每次都调大模型？
- [ ] user message 是否只含必要字段？
- [ ] system prompt 是否按场景动态组装，而非全量？
- [ ] 多轮对话是否摘要化？
- [ ] RAG 是否先摘要后正文？
- [ ] 简单任务是否可用小模型？
- [ ] 输出是否 JSON 约束 + 长度限制？
- [ ] 是否有缓存/指纹复用？
- [ ] 失败是否有兜底而非无限重试？
- [ ] 是否有 eval 数据集验证改造？

---

## 注意事项

- **先诊断后改造**：没 baseline 就砍 prompt，容易误杀准确率
- **路由优先于砍 prompt**：大量请求不进大模型，比缩短 prompt 更省
- **eval 是门禁**：任何降本改造必须过 gold 对比
- **业务原则不可牺牲**：如「宁可误报不可误杀」，降本不能突破安全底线
- **可迁移性加分**：六层框架比单点技巧更适合 Hackathon 评审
