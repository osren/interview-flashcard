## 任务：生成服务职责边界文档

你是一名资深后端架构师。基于已有的服务能力地图，提炼一份**极简、高信噪比**的职责边界文档。

## 输入来源

主要读取 `docs/knowledge/domain-model.md`（scan-service 的输出），不重新扫描代码。
若 `docs/knowledge/domain-model.md` 不存在，直接读取代码库进行分析。

## 输出要求

- **主输出位置**：`docs/knowledge/service-boundary.md`
- **可选补充位置**：`docs/knowledge/service-boundary/`（仅当能够按业务域或代码库稳定拆分时生成）
- **格式**：Markdown，总长度控制在 1 页以内

`docs/knowledge/service-boundary.md` 是后续流程读取职责边界的**稳定文档入口**，用于承载边界信息；knowledge 主入口仍是 `docs/knowledge/index.md`。目录版只是可选增强，不得作为唯一产物。

## 分析原则

**拒绝流水账**：不列举具体 API，只提炼业务领域。将"创建、查询、修改订单"合并为"订单全生命周期管理"。

**核心性过滤**：对每个功能问：如果没有它，这个服务还有存在意义吗？
- 是 → 写入核心职责
- 否 → 忽略

**边界判断的代码信号**：

| 判断维度 | 代码信号 | 结论 |
| :--- | :--- | :--- |
| 数据持有 vs 引用 | 有 INSERT/UPDATE → 持有；只有 SELECT → 引用 | 持有的数据才是"我的数据" |
| 编排层 vs 原子层 | 有多个 Feign/RPC 调用组合 → 编排；只操作自己的 DB → 原子 | 影响"核心定位"的描述 |
| 显性"不做"边界 | 存在 Feign 调用但无对应 DB 写操作的业务域 → 该域由下游负责 | 写入"不处理 XX" |
| 显性"不持有"边界 | 只读取其他服务数据（ID 引用或同步副本）→ 不持有该数据 | 写入"不持有 XX" |

**不猜测**：无法从 `domain-model.md` 或代码中确认的边界，标注 `<!-- TODO: 待确认 -->`，不推断。

## 输出格式

```markdown
# [服务名称]：[一句话核心价值]

## 核心定位

`[关键词1]` `[关键词2]` `[关键词3]`

> [100字以内：在架构中的层级（网关/编排/领域/基础）、核心业务对象、最关键的数据归属。]

## 核心职责

- **[职责1]**：[一句话描述]
- **[职责2]**：[一句话描述]
- **[职责3]**：[一句话描述]

## 显性边界

- **不处理 XX**：[明确说明不负责的业务范围，及由谁负责]
- **不持有 XX**：[明确说明不持有的核心数据，及数据归属方]
```

## 注意事项

- 关键词只选 3 个，强迫优先级排序
- 核心职责不超过 5 条，超过说明边界不清晰
- 显性边界至少写 2 条，这是本文档最有价值的部分
- 整篇文档不超过 1 页，超过则说明还在罗列细节，需要继续提炼
