## 三、AI 使用说明（简历 & 面试专用表述）

> 目标：诚实、专业，强调「AI 是杠杆，质量与规范仍由人负责」，避免给人「全靠 AI 写代码」的印象。

### 3.1 你实际用了哪些 AI 能力

| 场景 | 工具/方式 | 具体作用 | 你的把控点 |
|------|-----------|----------|------------|
| 需求理解 | Cooper Skill + MCP | 读取 PRD / 接口文档，提取字段与流程 | 人工核对 PRD 截图与业务口径 |
| 页面开发 | `crud-page` / `service-api` Skill | 生成 Service、列表页、编辑页、路由骨架 | 对齐 Ant Design 规范、项目既有模式、CR Review |
| 缺陷修复 | Cursor Agent 对话 | 定位下拉弹窗、搜索、接口入参等问题 | 本地复现 → 最小 diff 修复 → 回归 |
| 工作总结 | `weekly-report-skill` + Git / DDP | 自动汇总 commit、需求状态、模块关联 | 过滤非 jiazi 范围条目，补充业务解读 |
| 规范约束 | `AGENTS.md` / `ai-spec` | Agent 开发前读取项目规范 | 避免脱离 AntD 原生能力的大面积自定义 |

### 3.2 简历中推荐写法（选 1 句）

- **保守版**：熟悉 Cursor 等 AI 编程工具，用于加速需求文档理解、CRUD 脚手架生成与重复性代码整理，核心逻辑与上线质量仍通过 Code Review 与自测保障。
- **进取版**：在团队规范约束下构建 AI 辅助开发工作流，将 Cooper PRD → Service → 页面骨架的链路部分自动化，缩短新模块冷启动时间，并已应用于机票实验排序等新 feature 从 0 到 1。
- **工程化版**：不仅使用 AI 写代码，还用 Agent Skills + MCP 自动化周报与需求进度追踪，把 AI 用在「交付管理」和「知识对齐」上，降低多分支并行时的上下文切换成本。

### 3.3 面试中如何答「你们怎么用 AI？」

**推荐 30 秒版本：**

> 我们项目里 AI 主要用在三块：一是读 Cooper PRD 和接口文档，快速对齐字段；二是通过项目内的 crud-page、service-api 等 Skill，生成符合甲子规范的页面和 Service 骨架；三是用脚本和 MCP 自动汇总 Git 和望岳进度做周报。我的原则是 AI 负责提效和减少重复劳动，接口入参、交互细节、边界 case 和最终上线还是我自己 Review 和自测，所以 5 个月里 16 个 feature 分支能稳定跟 15 次 release，不是靠 vibe coding，而是 AI 加速 + 规范约束。

**如果被追问「AI 写错了怎么办？」**

> 我会把 AI 输出当第一版 draft，不直接提交。典型流程是：AI 生成 → 我对照 PRD/设计稿 diff → 跑页面自测 → 让同事 CR。比如 HCP 制度下拉的问题，是 AI 辅助定位 Ant Design Select 在 Modal 里的搜索行为，但修复方案是我确认复现路径后做的最小改动。
