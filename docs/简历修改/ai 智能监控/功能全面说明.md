# AI 监控降噪工具（ai-monitor-deposer）功能全面说明

> 分支：`feature/ai-monitor-deposer`  
> 定位：接口错误告警的智能上报判定系统 —— 每条告警走一次 LLM，判断「该不该上报、定什么等级」，把噪音挡在派发之前。  
> 文档版本：基于分支脚手架提交 `c4c4bf9 feat: [AI降噪工具] 脚手架搭建` 整理。

---

## 1. 功能需求背景

### 1.1 痛点

商旅（机票 / 酒店 / 火车票）线上接口错误告警量大、噪音占比高：

| 噪音类型 | 典型表现 | 对值班的影响 |
|---|---|---|
| 业务正常拒绝 | 无票、已退改、重复提交、待核验、超时未支付被取消 | 刷屏，掩盖真故障 |
| 低价值展示类错误 | NPS、公告、分享页、首页配置 | 占用注意力，价值低 |
| 信息不全的原始告警 | 缺 errorCode / 只有模糊 errmsg | 难以人工快速判断 |
| 三方/非自有接口异常 | 营销中台、感知中台、外部域名 | 值班背锅，处置路径不清 |

结果是：**真故障被淹没，值班被刷屏，漏看风险上升**。

### 1.2 目标

在告警到达值班之前，用大模型对**每条**接口错误做降噪判定：

1. **是否值得上报**（`should_report`）
2. **严重等级**（`P0` ~ `P3`）
3. **错误类型**（`error_type`）与**中文依据**（`reason`）

核心原则：

- **宁可误报，不可误杀**（漏报优先压到接近 0）
- **判定交给模型，工程只做路由与编排**
- **引擎领域无关，业务知识全在可插拔 domain**

### 1.3 非目标（当前阶段）

- 不做规则旁路预过滤（前期保证判定质量）
- 不替代故障根因定位（那是 `incident-triage` 工作流的职责）
- 不在引擎内硬编码数值阈值（调阈值 = 改 prompt）

## 2. 功能概述

系统分两大部分：

| 部分 | 路径 | 职责 |
|---|---|---|
| **运行时引擎** | `src/plugins/error-triage/` | 常驻服务：收告警 → 路由 → LLM 判定 → 派发 |
| **接入工具链** | `.claude/skills/` | 两个 Skill：从业务代码摘接口清单 → 生成 domain 配置 |

### 2.1 对外能力一览

| 能力 | 形态 | 说明 |
|---|---|---|
| **判定 + 派发** | `POST /triage` | 生产主链路，每条走一次 LLM |
| **只判定不派发** | `POST /judge` | 回放 / 攒 gold 草稿，不会触发真实告警 |
| **批量评测** | `POST /eval` | 批量判定 + 对比 gold 打分 |
| **摘接口清单** | Skill `discover-triage-urls` | 从业务仓库广召回核心链路 url |
| **生成 domain** | Skill `onboard-triage-domain` | 生成 `rules.js` + `prompts/` |
| **漂移哨兵** | `drift-check.sh` | 纯 grep、零 LLM，可挂 CI |

### 2.2 端到端链路

```
业务代码库
    │  discover-triage-urls（探针）
    ▼
<team>-urls.js（人工确认）
    │  onboard-triage-domain（接入生成器）
    ▼
domains/<team>/{rules.js, prompts/}
    │  DOMAIN=<team> 启动 error-triage
    ▼
监控/业务方投递单条告警 ──▶ route → triage(LLM) → dispatch
                              │
                              ├─ should_report=true  → IM/电话/工单（按 P 级）
                              └─ should_report=false → 丢弃（噪音）
```

### 2.3 当前落地状态

| 项 | 状态 |
|---|---|
| 通用引擎（HTTP / Kafka 双入口） | ✅ 就绪 |
| 火车票 domain（`domains/train/`） | ✅ 已接入，business_reject 已回源核验 |
| 机票 / 酒店 url 清单 | ✅ 已摘取（`flight-urls.js` / `hotel-urls.js`），domain 待 onboard |
| 真实派发通道 | ⏳ `dispatch` 现为日志占位 |
| 频次统计 / 聚合 / 结果缓存 | ⏳ 接缝已留，待建设 |
| 中心化配置 + eval-as-a-service | ⏳ 阶段二规划 |

    ├── router.js                      # url → category（领域无关）
    ├── triage.js                      # 拼 prompt / 调 LLM / 解析
    ├── llm.js                         # 公司大模型网关适配
    ├── pipeline.js                    # 按 DOMAIN 装配 + handleAlarm
    ├── server.js                      # HTTP：/triage /judge /eval /health
    ├── consumer.js                    # Kafka 入口

    │   ├── rules.js
    │   └── prompts/{_base,non-core,<key>}.md
    └── eval/                          # 本地评测闭环

### 3.2 分层架构

```
┌─────────────────────────────────────────────────────────┐
│ 入口层（领域无关）                                         │
│   server.js (HTTP)  ·  consumer.js (Kafka)               │
│   唯一职责：收包 → handleAlarm(alarm) / triage(alarm)     │
└───────────────────────────┬─────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────┐
│ 处理核 pipeline.js                                        │
│   按 DOMAIN 加载 domains/<team> → 拼出 triage + dispatch  │
└───────────────────────────┬─────────────────────────────┘
          ┌─────────────────┼─────────────────┐
          ▼                 ▼                 ▼
     router.js         triage.js          llm.js
   pathname 归一      拼 system/user      网关一次调用
   归属判别            解析 JSON           temperature=0
   有序规则匹配         失败兜底            超时约 10s
          │                 │
          └────────┬────────┘
                   ▼
         domains/<team>/（可插拔）
           rules.js + prompts/
```

**设计要点：**

1. **入口无关**：HTTP / Kafka 都只调 `handleAlarm`（或评测用的 `triage`）。
2. **领域可插拔**：换团队 = 换 `DOMAIN` + 加一个 `domains/<team>/`，引擎零改动。
3. **引擎领域无关**：`router` / `triage` / `llm` 不含业务知识。
4. **全程兜底**：超时 / 网关报错 / 解析失败 → 一律 `should_report=true, severity=P2`，永不吞告警、不卡队列。

### 3.3 单条告警处理流程

| 步骤 | 模块 | 行为 |
|---|---|---|
| 1 收包 | server / consumer | 解析 JSON；**不校验字段是否齐全** |
| 2 编排 | pipeline | `handleAlarm` → `triage` → `dispatch` |
| 3 路由 | router | url → pathname → 归属（ownedPrefixes）→ 有序规则 → key |
| 4 拼 prompt | triage | system = `_base.md` + `<key>.md`；user = 序列化 alarm |
| 5 调模型 | llm | 网关一次调用，`temperature=0` |
| 6 解析 | triage | 抽 JSON、校验字段；失败 → fallback 保守上报 |
| 7 派发 | pipeline | `should_report=true` 则按 severity 推送，否则丢弃 |

### 3.4 路由引擎细节

`createRouter(rules, { fallbackKey, ownedPrefixes })`：

1. **pathname 归一化**：去掉 host 与 query，避免 token/sig 误匹配。
2. **归属判别优先**：配了 `ownedPrefixes` 且路径不在前缀内 → 直接 `non-core`（规则救不回）。
3. **有序匹配**：命中靠前优先；特例必须排在宽松规则前。
4. **未命中兜底**：自有域内未覆盖接口 → `fallbackKey`（保守上报，保证不漏报）。

火车票示例：`ownedPrefixes = ['/esapp/']`，`fallbackKey = 'order'`，分类含 `payment / order / refund-change / query / user-auth / order-detail / display / non-core`。

### 3.5 入参 / 出参契约

**入参**（唯一硬依赖 `url`，其余缺失则倒向上报）：

```jsonc
{
  "url": "/esapp/railway/train/transferLeftTicket",
  "env": "prod",
  "level": "error",
  "errorCode": "1",
  "errorMsg": "参数错误",
  "count": 12,
  "traceid": "...",
  "httpStatus": 200,
  "response": { "errcode": "1", "errmsg": "参数错误" },
  "timestamp": 1751356800000
}
```

**出参**：

```json
{
  "should_report": true,
  "severity": "P1",
  "error_type": "param_validate_failed",
  "reason": "transferLeftTicket 属下单主流程入口，参数错误判 P1",
  "confidence": 0.86,
  "_category": "query"
}
```

| severity | 含义（火车票基线） |
|---|---|
| P0 / P1 | 阻断主流程链（查询→填单→下单→支付→出票→退改） |
| P2 | 影响单用户但有兜底；信息不足保守上报的下限 |
| P3 | 纯展示 / 弱业务；或 `should_report=false` 时的占位级 |

### 3.6 关键设计决策

| 决策 | 理由 |
|---|---|
| 每条告警都走 LLM | 保证判定质量；性能靠缓存/异步，而非减少调用 |
| 信息不足默认上报 | 宁可误报不可误杀 |
| 余票查询异常可判 P1 | 查询是下单主流程入口，挂了等于阻塞下单 |
| 只判正式环境 | 上报侧源头已过滤，`env` 恒为 `prod` |
| 三方不背锅 | 非自有前缀落 `non-core`，基线 P3、单点默认不上报 |
| 阻塞主流程的非自有路径可「提升」进 ownedPrefixes | 避免被归属判别直接压成 non-core 导致漏报 |

### 3.7 部署演进

```
阶段一 · 试点内嵌（当前）              阶段二 · 中心化（推广）
┌──────────────────┐                 ┌────────────────────┐
│ 项目内引擎 + domain│  ──演进──▶     │ 配置中心统一 domain  │
│ 机酒火试点维护     │                 │ 团队只填配置自助接入  │
└──────────────────┘                 │ eval-as-a-service   │
                                     └────────────────────┘
```

## 4. Skill 设计

接入新业务 = 「散落接口 → 可用 domain」，由两个 Skill 串成流水线，中间人工确认一次。

### 4.1 总体流水线

```
首次：项目源码 ─[discover 全量]→ url 清单 ─人工确认→ [onboard]→ domain 骨架 → 启动服务
维护：新增接口 ─[drift-check]→ [discover sync 仅增量]→ 确认 → 更新 rules/prompt
```

**同步不紧迫**：新接口即使未进清单，自有前缀内未命中 → `fallbackKey` 保守上报；非自有 → `non-core`。清单过期只降 severity 精度，**不造成漏报**。

### 4.2 Skill A：`discover-triage-urls`（探针）

| 项 | 说明 |
|---|---|
| 路径 | `.claude/skills/discover-triage-urls/` |
| 输入 | 团队名 + 业务项目路径（+ 可选 ownedPrefixes） |
| 输出 | `<team>-urls.js`（带中文注释、按链路分组） |
| 不做 | 不生成 rules/prompts，不碰引擎 |

**核心能力：**

1. **先探技术栈再套 playbook**  
   JS/TS 前端、原生 App、后端、OpenAPI/proto 各有不同 url 落法；有权威源（网关注册表 / OpenAPI）优先于 grep。

2. **广召回摘取**  
   grep 广撒网 + Explore agent 补读内联/动态拼接；同时捕获链路级语义与「正常业务拒绝」证据（`码 → 原文 (文件:行)`）。

3. **两维度正交打标**  
   - **归属**（是否 non-core）：纯前缀判定。  
   - **核心 vs 边缘**（severity 基线）：反查「挂了用户能否走完主流程」；注意业务专属高危轴（如网约车安全类）。

4. **增量 / sync 模式**  
   已有清单时自动走 sync：`drift-check.sh` 算 delta（零 LLM）→ 只对新增 url 做语义分类 → token ∝ delta。

5. **emit 前自检**  
   业务码标签回行号重读，核验前移，避免下游再起核验债。

**人工拍板点：** 核心链路划定、`ownedPrefixes`、疑似死代码、动态路径归一、主动放弃的覆盖盲区。

### 4.3 Skill B：`onboard-triage-domain`（接入生成器）

| 项 | 说明 |
|---|---|
| 路径 | `.claude/skills/onboard-triage-domain/` |
| 输入 | 团队名 + 已确认的 `<team>-urls.js` |
| 输出 | `domains/<team>/{rules.js, prompts/...}` |
| 不做 | 不改引擎代码 |

**生成物结构：**

```
domains/<team>/
├── rules.js                 # fallbackKey + ownedPrefixes + 有序 rules
└── prompts/
    ├── _base.md             # 通用核（schema/枚举/全局规则）+ 业务槽
    ├── non-core.md          # 非自有接口（基线 P3）
    └── <key>.md             # 每个 category 的判定要点
```

**`_base.md` 生成策略：**

- **通用核原样复用**：输出 schema、`error_type` 枚举、信息不足上报、高频升级、business_reject 不上报。
- **业务槽按目标业务填**：主流程链、severity 举例、正常拒绝语义；禁止照抄火车票预订例子。
- **部署槽按接入方写**：prod-only 则声明不变量，不写永不触发的 test 降级死逻辑。

**人工拍板点：** ownedPrefixes、各类 severity 基线、business_reject 清单、rules 特例排序。生成物是草稿，上线前需回源码核验。

### 4.4 漂移哨兵 `drift-check.sh`

```bash
bash .claude/skills/discover-triage-urls/scripts/drift-check.sh \
  <项目/src> <team>-urls.js [关注前缀...]
```

- 纯 grep、零 LLM，退出码 `1` = 有漂移。
- 可挂 CI / pre-commit，日常盯「新增未接入 / 疑似删除」。
- 它是哨兵不是真相：最终归类仍由 sync 的 agent 分类步确认。

### 4.5 Skill 与引擎的边界

| 层级 | 可变 | 不可变 |
|---|---|---|
| Skill | url 清单、domain 配置草稿 | — |
| Domain 配置 | rules / prompts（调判定松紧） | 引擎代码 |
| 引擎 | — | 换团队也不改 |

## 5. 效果指标

评测闭环位于 `src/plugins/error-triage/eval/`，也可通过 `POST /eval` 服务化调用。

### 5.1 指标体系（分开看，不看笼统准确率）

| 指标 | 含义 | 优先级 |
|---|---|---|
| **漏报率 FN** | gold 该报、判成不报 | **头号指标**，先压到 0 / 接近 0 |
| **误报率 FP** | gold 不报、判成报 | 在 FN 达标后下降噪 |
| **severity 完全一致** | 与 gold 同级 | 主精度 |
| **severity 差一级 / 差多级** | 临界偏差 vs 严重偏差 | 差多级进失败清单 |
| **error_type 准确率** | 类型是否命中 gold | 辅助归因 |
| **missing** | 未跑出判定的 case | 工程健康度 |

CLI 报告字段与 `/eval` 的 `report.train|holdout` 同源（`score-lib.js`）。

### 5.2 评测闭环

```
MCP 拉异常 → 指纹去重成 case → 标 gold（模型草标 + 人工纠正）
    → run.js 或 POST /eval（真实模型，只判定不派发）
    → score 出指标 + 失败清单
    → 归因到具体 prompt 规则 → 改 prompt
    → 数据集冻结重跑对比（回归）
```

### 5.3 调优纪律

1. **漏报优先**：不为降噪牺牲 recall。  
2. **数据集冻结**：一轮调优期间不动 dataset，否则指标不可比。  
3. **train 调 / holdout 验**：holdout 不涨或下跌 = 过拟合，回退。  
4. **最终数字用生产模型**：更强模型只用于标 gold 草稿与错误分析。  
5. **正负样本平衡**：真实数据里 business_reject 占多数，必须保留足够「真故障」正样本，防止「一律不报」刷高分。

### 5.4 人机协作调 prompt

| 环节 | 主力 |
|---|---|
| 错误分析、出 prompt diff | AI |
| 守 gold、severity / business_reject 业务语义 | 人 |
| 上线终审 | 人 |

全自动闭环若启用，四道护栏缺一不可：holdout 不参与调、gold 冻结 AI 碰不到、FN 与 FP 一起卡、上线前人闸门。

### 5.5 效果验收建议（试点）

| 验收项 | 建议标准 |
|---|---|
| holdout FN | ≈ 0 |
| holdout FP | 相对基线明显下降（业务自定目标） |
| severity 差多级 | 趋近 0 |
| 工程兜底 | 网关故障时仍能保守上报，Kafka 位移可提交 |
| 人工抽检 | 主流程链（查/下/付/退）抽样与业务共识一致 |

> 判定方向在相同模型 + 相同 prompt 下可复现；临界 severity（P2↔P1）以生产模型实跑为准。

## 6. 使用方法

### 6.1 运行前提

- Node **18+**（内置 `fetch`、ESM）
- `package.json` 中 `"type": "module"`
- Kafka 入口需 `npm i kafkajs`
- 接网关前按 `llm.js` 内 TODO 改鉴权头 / 请求体 / 响应取文本路径

### 6.2 环境变量

| 变量 | 用途 | 默认 |
|---|---|---|
| `DOMAIN` | 加载 `domains/<DOMAIN>/` | `train` |
| `LLM_GATEWAY_ENDPOINT` | 大模型网关（必填） | — |
| `LLM_GATEWAY_KEY` | 鉴权凭证 | — |
| `LLM_MODEL` | 模型名 | `gpt-4o-mini` |
| `PORT` | HTTP 端口 | `8080` |
| `KAFKA_BROKERS` / `KAFKA_TOPIC` / `KAFKA_GROUP` | Kafka 消费 | topic=`alerts` |
| `KAFKA_CONCURRENCY` | 跨分区并发（≈ LLM 并发） | `4` |
| `EVAL_CONCURRENCY` | `/eval` 批量并发 | `4` |

### 6.3 启动服务

**HTTP：**

```bash
cd mpx/skills/tools/ai-monitor-deposer
DOMAIN=train \
LLM_GATEWAY_ENDPOINT=https://your-gateway/v1/chat \
LLM_GATEWAY_KEY=xxx \
node src/plugins/error-triage/server.js
```

**Kafka：**

```bash
DOMAIN=train \
LLM_GATEWAY_ENDPOINT=... LLM_GATEWAY_KEY=xxx \
KAFKA_BROKERS=broker1:9092 KAFKA_TOPIC=alerts \
node src/plugins/error-triage/consumer.js
```

### 6.4 API 调用示例

**生产判定（会派发）：**

```bash
curl -XPOST localhost:8080/triage \
  -H 'Content-Type: application/json' \
  -d '{"url":"/esapp/railway/train/queryLeftTicket","env":"prod","level":"error","errorMsg":"validate failed","count":12}'
```

**只判定不派发（评测 / 回放）：**

```bash
curl -s localhost:8080/judge -H 'Content-Type: application/json' -d '{
  "alarm": {
    "url": "/esapp/railway/train/transferLeftTicket",
    "env": "prod",
    "errorCode": "1",
    "errorMsg": "参数错误"
  }
}'
```

**批量评测：** 见 [`eval-api.md`](./eval-api.md)。

### 6.5 新业务首次接入

```text
1. /discover-triage-urls   团队名=<team>，项目路径=<业务代码库>
     → 产出 <team>-urls.js，人工确认

2. /onboard-triage-domain  团队名=<team>，url 清单=<team>-urls.js
     → 产出 domains/<team>/{rules.js, prompts/...}

3. 人工核验（上线前必做）
     - business_reject 码回源码逐条核对（含自有封装库 import 链）
     - ownedPrefixes、severity 基线、rules 特例排序

4. DOMAIN=<team> LLM_GATEWAY_ENDPOINT=... node .../server.js
```

### 6.6 增量维护

```bash
# 1) 漂移哨兵（可挂 CI）
bash .claude/skills/discover-triage-urls/scripts/drift-check.sh \
  <项目/src> <team>-urls.js /esapp/

# 2) 再调一次 discover（检测到已有清单 → 自动 sync，只分类 delta）

# 3) 新 category / 新核心接口时再跑 onboard；否则常只需在 rules.js 补 pattern
```

### 6.7 调整判定松紧

引擎无硬编码数值阈值，**调阈值 = 改 prompt**：

| 想调什么 | 改哪里 |
|---|---|
| 全局置信度 / 高频升级 / severity 总纲 | `prompts/_base.md` |
| 某类基线 / business_reject 清单 | `prompts/<key>.md` |
| 只派发 P0/P1 | 派发层 `pipeline.js` 的 `dispatch` 加 severity 闸门 |

⚠️ prompt 启动时读入缓存，**改完需重启服务**。  
上报侧尽量填全 `url + errorCode + count`，否则对应阈值形同虚设。

## 7. 后续演进与风险

### 7.1 待建设项

| 项 | 说明 |
|---|---|
| 真实派发通道 | 接 IM / 电话 / 工单，替换 `console.log` |
| 频次统计（count） | 服务侧按指纹滑动窗口，驱动高频升级 |
| 告警聚合 | 同指纹时间窗内合并，防风暴 |
| 结果缓存 | 同指纹短 TTL 复用判定，降模型成本 |
| 探针下探组件库 | 接口封装在共享组件内时，当前扫业务仓会漏 |
| BFF 统一封装 | BFF 捕错 → 调 triage，业务侧接入更轻 |
| 中心化配置 | domain 抽到配置中心，多租户按请求 `domain` 路由 |

### 7.2 已知风险与缓解

| 风险 | 缓解 |
|---|---|
| prompt 草稿业务码标错 | discover 强制带证据 + emit 自检 + 上线前人工回源 |
| 清单过期 | fallback / non-core 兜底不漏报；drift-check 盯漂移 |
| 模型临界抖动 | temperature=0；生产模型实跑验收；eval 回归 |
| LLM 故障 | 永不吞告警，统一 P2 保守上报 |
| 为降噪牺牲漏报 | FN 头号指标 + 正负样本平衡纪律 |
