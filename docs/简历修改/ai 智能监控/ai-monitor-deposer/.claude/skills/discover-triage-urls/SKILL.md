# 从项目里摘取核心链路 url 清单

把一个**接口散落在很多地方**的项目，转成一份 `onboard-triage-domain` 能直接吃的 url 清单：
`<team>-urls.js`（urls.js 形态：按业务链路分组的注释 + `export const NAME = '/path'`）。

**本 skill 只产出「中间物 = url 清单」，不生成 rules.js / prompts，也不碰 error-triage 引擎。**
清单必须经用户人工确认，才交给下游 `onboard-triage-domain`。切分清楚，坏味道拦在中间。

设计参照 `src/plugins/error-triage/domains/train/rules.js` 与项目根的 `urls.js`（人工维护的清单范本，注意它每条 url 都带一句中文注释——那正是我们要还原的语义）。

## 0. 先确认输入 + 判定模式（缺了就问用户）

- **团队/业务名**（kebab-case，如 `hotel`、`flight`；用于输出文件名和下游 DOMAIN）。
- **项目路径**：要分析的代码库根目录。
- （可选）**已知的自有前缀**：用户若知道（如 `/esapp/`）先记下，否则第 1 步侦察时推断。

**判定模式**：
- 若 `<team>-urls.js` **不存在** → **全量模式**（走第 1–5 步，首次接入）。
- 若 `<team>-urls.js` **已存在**（项目后续新增接口、要同步）→ **增量/sync 模式**（跳到「增量 / sync 模式」一节，只处理 delta，省 token）。

## 1. 侦察：先探技术栈，再套对应 playbook（不能跳）

**不要假设项目是 JS 前端。** 先花几分钟判断技术栈，再选匹配的抽取 playbook——不同栈里 url 的落法天差地别，套错 playbook 会大面积漏。

**第一步先判栈**（看 `package.json` / `pom.xml` / `go.mod` / `build.gradle` / `Podfile` / `pubspec.yaml` / 目录结构）：

| 技术栈 | url 主要落在哪 | 首选来源 |
|---|---|---|
| **JS/TS 前端**（含 mpx/小程序/RN） | 请求调用内联 `$http/axios/fetch(...)`、集中常量 `urls.js`、`base+相对路径`拼接 | grep 字符串字面量 + Explore 补读 |
| **原生 App**（Android/iOS/Flutter） | Retrofit/OkHttp 注解、`URLSession`/Alamofire、Dart 常量；也可能从远端配置下发 | 注解/常量 grep；下发式则找配置源 |
| **后端服务**（Java/Go/Node） | 路由注册/注解 `@RequestMapping`/`router.post('/x')` | **优先读网关注册表 / OpenAPI，而非 grep 代码** |
| **有 OpenAPI/Swagger/proto** | `openapi.yaml` / `.proto` / 接口文档 | **直接解析规范文件——最权威、最全，优先于一切 grep** |

> **原则：有权威接口清单（网关注册表、OpenAPI、proto）就先用它**，grep 代码只作补充/交叉核对。只有前端内联散落这种没有权威源的，才以 grep + Explore 为主。

**选定 playbook 后，各栈都要覆盖的通用落法**（別漏）：
- **集中常量文件**：`export const X = '/path'`（`urls.js` / `constants` / `endpoints`）。
- **请求调用内联**：`mpx.$http.post('/x')`、`axios.get('/x')`、`fetch('/x')`、`request({ url })`、`$api('/x')`。**前端散落重灾区**。
- **base + 相对路径 / 模板串**：`base + '/x'`、`` `/order/${id}` ``（动态段需归一）、`` `${mockOrBase}/x` `` 前缀常量拼接。
- **绝对域名三方接口**：完整 `https://host/path`。**只按自有前缀 grep 会整类漏掉——必须单独扫 `https?://` 请求目标**；其 pathname 多不在自有前缀下 → non-core。

**做法**：先广谱 grep 摸清主模式与规模，再据此定精确抽取模式。grep 示例按栈调整，以下为 JS/前端起手式：

```bash
# 路径字面量（有 host 就带 host，业务通常只关心 pathname）
grep -rEoh "['\"](/[a-zA-Z][a-zA-Z0-9/_-]+)['\"]" <项目> | sort | uniq -c | sort -rn | head -40
# 调用姿势 / 分布目录
grep -rEn "\.(post|get|put|delete)\(|url\s*[:=]|request\(|fetch\(" <项目>/src | head
grep -rEl "['\"]/[a-zA-Z]" <项目>/src | sed -E 's#(src/[^/]+)/.*#\1#' | sort | uniq -c | sort -rn
# 绝对域名请求目标（必扫，防三方整类漏）
grep -rEoh "['\"\`]https?://[a-zA-Z0-9.-]+/[a-zA-Z0-9/_-]+['\"\`]" <项目>/src | sort -u
# 后端/注解栈改用（示例）：grep -rEn "@(Get|Post|Request)Mapping|router\.(get|post)" <项目>
```

**产出一段 recon 小结**：技术栈、选用的 playbook、几种声明模式、自有前缀、url 规模、主要分布目录。写给用户看，作为后续抽取依据。

## 2. 广召回摘取（召回优先，连语义上下文一起抓）

目标是**先不漏**，宁可多抓后面再剪。纯 grep 会漏内联调用和动态拼接，所以 **grep 广撒网 + `Explore` agent 补读**：

- 用 `Explore` agent 扫第 1 步认定的重灾目录（如内联调用密集的 `src/api`、`src/pages`、`src/store`），任务写明：**「找出所有对外接口 url，并对每个 url 记录：完整路径、它是干嘛的（取常量名/上方注释/所在函数名/调用组件作为语义）、声明位置文件」**。
- 对每个候选 url 记：`{ path, 语义说明, 来源文件, 是否核心链路(待第3步定) }`。
- **同时捕获链路级语义**（供下游生成专属 prompt，不要只停在单条 url）：每条业务链路记一句「这条链路是干嘛的」，以及从代码/文案里能看出的「这条链路里什么算业务正常拒绝」（如预订的无票/已退改、网约车的无司机应答/超服务区）。这是让下游 prompt 贴该业务、而非抄模板的关键原料。
  - **业务码/正常拒绝必须带逐字证据，标签从原文推、禁止转述/猜**：抓到 `code==='2023'` 这类分支时，连它的 `message`/弹窗文案/动作一起记（如 `message:'距发车太近，请换车次'`），标签**直接由这句原文得出**——不许凭码号或经验编（本项目就吃过亏：把"距发车太近换车次"猜成"无票"、把"人脸核验"写成"人证核验"、把"儿童票拦截"写成"代客下单"）。记成 `码 → 原文片段 (文件:行)` 的形态。
  - **emit 前自检（把核验前移，省掉下游单独核验阶段）**：清单写完后，对每个业务码回它标注的行号**重读一次**，确认 (a) 码字面还在那行、(b) 写的标签与分支 message/动作一致。**只读被引用的那几行，token ∝ 码数量而非项目大小**，很便宜。对不上就改标签或标「待人工确认」，绝不把没核过的码留给下游。

**必须处理的坑（照做）**：
- **死代码**：注释掉的 url（`// '/x'`）、定义了但全项目无人调用的常量 → 不进清单，或单列"疑似死代码"待用户裁。
- **动态段归一**：`/order/${id}`、`/user/:id` → 归一成路径片段（`/order/`）或正则形态，并标注，供下游 rules 写 pattern。
- **去重**：同一 url 多处调用只保留一条，语义取信息量最大的那处。
- **只取 pathname**：带 host 的去掉 host 和 query（下游引擎按 pathname 路由）。

## 3. 链路归并 + 两维度打标（归属 / 核心度）

先按**业务链路**聚类（搜索/查询 → 下单 → 支付 → 出票 → 退改 → 订单详情 → 用户鉴权 → 展示/弱业务），
再沿**两个相互独立、正交**的维度打标。别把两者混成一件事。

**维度 A · 归属（决定路由到 non-core 与否）—— 纯前缀判定**
- 归属只看**路径前缀**，与业务重要性无关。取自有接口的公共前缀作为 `ownedPrefixes`（如 `/esapp/`）。
- **凡以 `ownedPrefixes` 开头 = 自有接口**，走正常业务规则——哪怕它调的是内部中台（如 `/esapp/mars`、`/esapp/crius`、`/esapp/laputa`），只要在 `/esapp/` 下就是自有，**不是 non-core**。
- **不以 `ownedPrefixes` 开头 = non-core**（外部营销/感知中台、绝对域名三方服务等），路由到 `non-core.md`。绝对域名（`https://host/...`）取其 pathname 判定，通常不在 `/esapp/` 下 → non-core。
- **⚠️ 两维度冲突的解法（非自有前缀但阻塞主流程）**：引擎的归属判别**在规则匹配之前**——非自有前缀会**直接落 non-core**，规则救不回；而 `non-core.md` 基线低（P3、单点不上报），会**把一个阻塞主流程的接口漏报**。所以当一个接口前缀不在 `ownedPrefixes` 内、但维度 B 判定它阻塞主流程（如清风拦截 `/qingfeng/.../intercept` 失败即拦下单）时，**不能放任它落 non-core**：把**该具体路径**补进 `ownedPrefixes`（`startsWith` 精确到整条路径即可，不必放开整个前缀），让它路由到对应核心 category。在清单里把它归到对应核心链路分组，并注明"已提升进 ownedPrefixes"。核心度决定要不要提升，前缀机制只是落地手段。

**维度 B · 核心 vs 边缘（只影响下游 severity 基线，不影响路由归属）**
- **先识别这个业务的"主流程链"，不要套预订的**。从代码入口/主用户旅程反推该业务的核心链路是什么：
  - 预订类（机票/酒店/火车）：搜索→下单→支付→出票→退改。
  - 其它业务自有一条，如网约车：发单→派单→接驾→行程中→支付→结束（并注意其**安全类**接口天然高危，见下）。
  - 拿不准就把推断出的主流程链写给用户确认，再据此判核心。
- **核心 = 阻塞主流程**：接口异常会卡住用户走完"该业务主流程链"的。下游 severity 基线高（P0/P1）。
- **边缘 = 弱业务**：挂了不阻塞主流程的（展示/公告/日志/AB 实验/分享等），下游 severity 基线低（P2/P3）。
- 判"是否阻塞"用**反查法**：这个接口挂了，用户还能不能走完该业务的主流程链？不能 → 核心。
- **注意业务专属的高危轴**：某些接口不"阻塞交易"但天然 P0（如网约车的安全/一键报警/行程分享）。识别主流程时一并标出这类，别因为"不阻塞下单"就误判成边缘。
- **标核心前必须验调用方**：回到业务层（pages/logic/store）找到真实调用点，确认 (a) 它确实有人调用（无调用方 = 死代码，别标核心，归入疑似死代码），(b) 失败是**卡住流程**而非**失败放行**（fail-open）。**中台/拦截/风控/异常上报类接口尤其要查**——它们常常是死代码或失败放行，光看接口名（如"订单拦截"）会误判成核心。中台归属仍是自有（维度 A 按前缀），但核心度要按真实调用行为定。

> 两维度正交：一个 `/esapp/` 自有接口可能是边缘弱业务（如 NPS）——归属仍是**自有**，只是 severity 基线低。
> **non-core 只由维度 A（前缀）决定，不要因为"是中台/不重要"就标成 non-core。**
> 分类只是给下游一个好起点；category 的最终切法由 `onboard-triage-domain` 定。

## 4. 产出清单文件 + 待确认清单

写出 `<team>-urls.js`（放在 ai-monitordeposer 项目下，供下游读取），形态对齐项目根 `urls.js`：

分组标题带上**链路级语义**（这条链路干嘛的 + 什么算正常拒绝），这是下游生成专属 prompt 的原料，别省。**其中每个业务码的正常拒绝都要附证据**——写成 `码=语义 (文件:行)`，语义取自分支 message 原文，让下游和复核者能一眼溯源；这样码的存在性后续还能纯 grep 校验（零 LLM），只剩"标签贴不贴切"需要瞄一眼：

在文件头记下**源码 commit SHA**（`git -C <项目> rev-parse --short HEAD`），供后续增量 `git diff` 精确定位改动：

```js
/* <team> 核心链路接口清单 —— discover-triage-urls 自动摘取，需人工确认后交给 onboard-triage-domain
 * 源码: <项目路径> @ <commit SHA>    （sync 时用 git diff <SHA>..HEAD 缩小范围）
 * 建议 ownedPrefixes: ['/esapp/']
 * 推断的主流程链：搜索 → 下单 → 支付 → 出票 → 退改   （业务专属高危轴：无）
 */

// ============ 支付链路（核心） ============
// 链路语义：完成订单支付；正常拒绝（带证据）：'2032'=无效订单(order-detail/index.js:33)、支付超时被取消/重复支付等业务态。
export const PAY_CREATE = '/esapp/flight/pay/create/v1';   // 下单-创建支付
// ...

// ============ 展示/弱业务（边缘） ============
export const NPS_REWARD = '/esapp/flight/nps/reward';      // NPS 问卷奖励
// ...
```

同时给用户列出**必须人工拍板**的点（别假装已定好）：
1. **核心链路划定**是否正确（有没有把边缘接口当核心、或漏了核心接口）。
2. `ownedPrefixes`（自有 vs 三方边界）是否对。
3. **疑似死代码**清单（注释掉的 / 无调用的）——留还是删。
4. **动态路径**归一得对不对（`/order/${id}` → `/order/`）。
5. 摘取过程中**主动放弃/未覆盖**的部分（如某种声明模式没扫、动态拼接无法静态还原的）——如实说明，别让"清单看起来很全"掩盖漏掉的接口。

## 增量 / sync 模式（已有 `<team>-urls.js`，只处理新增，省 token）

全量重扫最贵的是「agent 通读全项目语义」。增量的原则：**把 agent 的输入从"整个项目"缩到"新增的几条 url"，token 消耗 ∝ delta，而非项目大小**。

**先摆一个前提，降低同步紧迫性**：新增接口即使没进清单/rules，引擎也**不会漏报**——自有前缀内未命中规则 → 落 `fallbackKey` 保守上报；非自有 → non-core 兜底。所以清单过期只是 **severity 精度下降，不是漏报**，同步可批量、低频做。

**sync 步骤**：

1. **廉价算 delta（纯 grep，零 LLM）**：跑本 skill 目录下的 `scripts/drift-check.sh`：
   ```bash
   bash scripts/drift-check.sh <项目/src> <team-urls.js> [关注前缀...]
   # 前缀默认 /esapp/；有非自有关注前缀就追加，如： /esapp/ /qingfeng/ /eos/ /perceive/
   ```
   它输出「新增未接入」和「疑似删除/改名」两组，退出码 1 表示有漂移。可挂 CI/pre-commit 平时零成本盯着。
   （若清单头记了 commit SHA，还可 `git -C <项目> diff <SHA>..HEAD -- src/api` 进一步缩小要读的文件。）

2. **只对「新增」少数 url 跑语义分类**：用 `Explore` agent，任务只针对这几条——给出语义/核心边缘/所属链路/正常拒绝，并**照第 3 步维度 A 判归属、检查是否有新的非自有前缀阻塞接口**（触发提升规则）。不要重述已有 url。

3. **「疑似删除」→ 标记待确认**（可能删了或改名/换前缀），不擅自删。**「不变」→ 原样保留**语义，不重读。

4. **合并回 `<team>-urls.js`**：新增项按链路归到对应分组；更新文件头的 commit SHA。列出本次「新增/删除/需人工确认」清单给用户。

5. **交棒**：清单变化涉及新 category 或新核心接口时，再跑 `onboard-triage-domain` 更新 rules/prompts；若新增项都落进已有 category、且 severity 语义无变化，往往只需在 `rules.js` 补几条 pattern，prompt 不动。

> 注意：drift-check 是**哨兵**不是**真相**——它按前缀 grep，可能带出边缘/异常接口或形态变体（实测能兜出全量遗漏的边缘接口，这正是它的价值）；最终归类仍由 sync 的 agent 分类步确认。

## 注意（守则）
- **只产出 url 清单**这一个中间物；不生成 rules/prompts，不碰引擎。
- **召回优先**：宁可多抓让用户剪，不要静默漏。任何有意的截断/放弃都要 `log` 给用户，不能让清单显得"已覆盖全部"。
- **语义来自代码，业务码更要逐字取证**：每条 url 的中文说明必须来自常量名/注释/函数名/调用位置；business_reject 业务码的标签必须由分支 message 原文推出，并在 emit 前回行号自检，绝不凭码号猜编。核验前移到摘取当下，下游不再另起核验阶段。
- 侦察结论要写给用户，抽取模式随项目而变，别套用某一个项目的固定假设。
