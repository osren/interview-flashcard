# 接口错误告警智能降噪判定服务（error-triage）

## 能力总览

**用大模型给每条接口错误告警做降噪分级判定，把噪音挡在派发前，只把真故障按 P1~P3 送给值班。** 四类能力对外都是 **API / 工具**，团队自带数据与配置、借中心的引擎和模型，不需集成项目代码。

| 能力 | 对外形态 | 说明 |
|---|---|---|
| **判定** | `POST /triage` | 判定 + 派发（生产主链路），每条走一次 LLM |
| **回放/调优** | `POST /judge` | 只判定、不派发，跑评测不会触发真实告警 |
| **调优评测** | `POST /eval` | 批量判定 + 对比 gold 打分，量化 prompt 好坏 |
| **接入** | 两个 Skill | `discover-triage-urls`（摘接口，含 sync 增量）+ `onboard-triage-domain`（生成配置） |

**核心设计**：引擎领域无关、业务知识全在可插拔的 `domains/<team>/`（路由表 + prompt）；换团队只改配置不改代码。
**推广路线**：机票/酒店/火车票试点**内嵌**，跑通后统一走**中心化配置 + eval-as-a-service**，其他团队自助接入。

---

## 一、背景与目标

线上接口错误告警量大、噪音多：大量"业务正常拒绝"（无票、已退改、重复提交等）和低价值展示类错误会淹没真正阻塞主流程的故障，值班被刷屏、真告警被漏看。

**目标**：在告警到达值班之前，用大模型对每条接口错误做**降噪判定** —— 判断"是否值得上报 + 严重等级 + 原因"，把噪音过滤掉、把真故障按 P1~P3 分级推送。

## 二、方案概述

对**每一条**接口错误告警：

1. 按 `url` 路由到对应业务分类（查询/下单/支付/退改/…）；
2. 取该分类的判定 prompt（公共规则 `_base` + 分类细则），连同告警内容一起调**一次**大模型；
3. 模型输出结构化判定（是否上报 / severity / 错误类型 / 原因 / 置信度）；
4. 需上报的按等级派发到告警通道（IM/电话/工单）。

核心理念：**判定交给模型，工程只做路由与编排**。规则沉淀在 prompt 里，改判定策略 = 改 prompt，不改代码。

## 三、架构设计

```
   业务方 / 监控平台
   （单条接口错误事件）
          │
          ▼
┌─────────────────────┐
│  入口层（领域无关）    │   server.js (HTTP /triage)
│                     │   consumer.js (Kafka)
└─────────┬───────────┘   职责：收包 → handleAlarm(alarm)
          │
          ▼
┌───────────────────────────────────────────────────────┐
│  处理核 pipeline.js（handleAlarm：编排 + 派发）           │
│                                                       │
│   ┌──────────┐   ┌──────────────┐   ┌─────────────┐    │
│   │ router.js│──▶│  triage.js   │──▶│   llm.js    │    │
│   │ 路由引擎  │   │  编排引擎     │   │ 网关适配     │    │
│   │ url→分类  │   │ 拼prompt/解析 │   │ 调模型一次   │    │
│   └────┬─────┘   └──────┬───────┘   └──────┬──────┘    │
│        │                │                  │           │
│        │ 读取            │ 读取              ▼           │
│        ▼                ▼            ┌─────────────┐    │
│   ┌───────────────────────────┐     │  LLM 网关    │    │
│   │ domains/<team>/（可插拔）    │     │  (DCC api)  │    │
│   │  ├─ rules.js  路由表        │     └─────────────┘    │
│   │  └─ prompts/  判定规则       │                       │
│   │      ├─ _base.md 公共        │   换团队=换DOMAIN       │
│   │      └─ <分类>.md 细则       │   + 加一个文件夹         │
│   └───────────────────────────┘   引擎零改动             │
│                                                       │
│        判定结果 → dispatch（should_report 则派发）        │
└───────────────────────────┬───────────────────────────┘
                            │
                            ▼
                告警通道（IM / 电话 / 工单，按 P1~P3 分级）
```

**要点**：入口层与三个引擎（router/triage/llm）**领域无关**；业务知识全部下沉到可插拔的 `domains/<team>/`（路由表 + prompt）。换业务只改配置目录，不动引擎。

**分层要点：**

- **入口无关**：HTTP（`server.js`）和 Kafka（`consumer.js`）两个入口，唯一职责是"拿到一条 alarm → `handleAlarm(alarm)`"，判定逻辑全在 `pipeline.js`。
- **领域可插拔**：换团队/业务 = 换 `DOMAIN` 环境变量 + 加一个 `domains/<team>/` 文件夹（路由表 `rules.js` + 判定 prompt），引擎代码零改动。
- **引擎领域无关**：`router.js`（路由）、`triage.js`（编排）、`llm.js`（模型网关适配）都不含业务知识。
- **全程兜底**：判定链路 try/catch，任何异常（超时/网关报错/解析失败）都**保守上报**，永不抛错、不卡队列。

## 四、单条告警处理流程（分步）

| 步骤 | 模块 | 做什么 |
|---|---|---|
| 1 收包 | server/consumer | 解析 JSON 成 alarm，**不校验字段是否齐全**（缺字段交给兜底） |
| 2 编排 | pipeline | `handleAlarm` → `triage(alarm)` → `dispatch` |
| 3 路由 | router | `url` 去 host/query 取 pathname → 归属判别（是否自有接口）→ 有序规则匹配得分类 key |
| 4 拼 prompt | triage | system = `_base.md` + 分类 `.md`；user = 序列化后的 alarm |
| 5 调模型 | llm | 调网关一次，temperature=0（判定要稳定），10s 超时 |
| 6 解析 | triage | 抽 JSON、校验字段；失败 → fallback 保守上报 |
| 7 派发 | pipeline | `should_report=true` → 按 severity 推送；否则丢弃 |

## 五、服务入参契约

**唯一硬依赖是 `url`**（用于路由）；其余字段是喂给模型的判定材料，缺失时走"信息不足默认上报"。业务方只需发**单条错误事件**：

```jsonc
{
  "url": "/esapp/railway/train/transferLeftTicket", // 必填，路由用
  "env": "prod",            // 恒为 prod（源头已过滤）
  "level": "error",         // 监控侧原始等级
  "errorCode": "1",         // 由响应体 errcode 归一化
  "errorMsg": "参数错误",    // 由响应体 errmsg 归一化
  "traceid": "0ac5...9602", // 定位用
  "httpStatus": 200,        // 区分 5xx / 业务码（可选）
  "response": { "data": null, "errcode": "1", "errmsg": "参数错误" }, // 原始响应体（可选留证）
  "timestamp": 1751356800000
}
```

## 六、判定输出与分级

**输出（严格 JSON）：**

```json
{ "should_report": true, "severity": "P1", "error_type": "param_validate_failed",
  "reason": "中文一句话依据", "confidence": 0.86 }
```

**分级基线**：主流程链 = 查询(余票) → 填单/校验 → 下单占座/抢票 → 支付 → 出票 → 退改。
阻断该链的真异常 = **P1**；影响单用户但有兜底 = **P2**；纯展示/弱业务 = **P3**。

## 七、实例演示

**输入**：接口 `/railway/train/transferLeftTicket` 返回
```json
{ "data": null, "errcode": "1", "errmsg": "参数错误", "traceid": "0ac501b8...9602" }
```

**判定过程**：
- 路由：url 含 `LeftTicket` → 分类 `query`（余票查询）。
- 规则：transferLeftTicket 是下单主流程必经入口，参数错误也阻塞下单 → 按 `query` 细则"余票查询异常一律 P1"。

**输出**：
```json
{ "should_report": true, "severity": "P1", "error_type": "param_validate_failed",
  "reason": "transferLeftTicket 中转余票查询参数错误，属下单主流程入口，判 P1", "confidence": 0.86 }
```

## 八、关键设计决策

1. **每条告警都走 LLM**：不做规则旁路预过滤，保证判定质量；性能优化靠缓存/异步，而非减少调用。
2. **信息不足默认上报**：缺关键字段或置信度低 → 倒向上报（`unknown`，≥P2），宁可误报不可误杀。
3. **余票查询异常判 P1**：查询是下单主流程入口，挂了等于阻塞下单，不因名义上是"查询"而降级。
4. **只判正式环境**：源头已过滤，`env` 恒为 `prod`。

## 十、部署演进：试点内嵌 → 中心化配置

**分两阶段推广，降低早期落地成本、又不牺牲后续规模化：**

- **阶段一 · 试点内嵌（当前）**：机票、酒店、火车票三条业务先试点。框架与各业务的 domain 配置（路由表 + prompt）**先放在项目内**随项目一起维护，快速验证判定效果、打磨 prompt，不引入额外基础设施。
- **阶段二 · 中心化配置（推广期）**：试点跑通后，把 domain 配置抽到**统一的配置中心**。后续其他团队接入时，直接在这一个地方读取/维护自己的配置即可拥有降噪能力，**无需再把配置塞进各自项目、也不用各自维护一份框架**。引擎与配置彻底解耦，接入成本从"改代码"降到"填配置"。

```
阶段一（试点）                     阶段二（推广）
┌───────────────┐               ┌──────────────────┐
│ 项目内         │               │  配置中心（统一）   │
│  ├─ 引擎        │               │  ├─ 机票 domain    │
│  └─ domain配置  │  ──演进──▶    │  ├─ 酒店 domain    │
│    (随项目维护) │               │  ├─ 火车票 domain  │
└───────────────┘               │  └─ 新团队 domain… │
  机票/酒店/火车票                 └──────────────────┘
                                  各团队只在此读/写配置，
                                  项目内不再维护
```

## 十一、快速接入：两个 Skill

新团队接入 = 从"项目里散落的接口"到"可用的 domain 配置"，由两个 Skill 串成一条流水线，中间人工确认一次：

```
项目源码 ──[discover-triage-urls]──▶ url 清单 ──人工确认──▶ [onboard-triage-domain] ──▶ domain 骨架
                                  (带中文注释)                              (rules.js + prompts/)
```

**① `discover-triage-urls` —— 摘接口清单**
- **输入**：一个项目路径 + 团队名。
- **做什么**：从散落着接口调用的前端/后端项目里，自动发现并摘取「核心业务链路」的接口 url，产出一份**带中文注释的 url 清单**（供人工确认）。

**② `onboard-triage-domain` —— 生成 domain 骨架**
- **输入**：上一步确认后的 url 清单。
- **做什么**：为 error-triage 生成该团队的完整 domain 骨架 —— 路由表 `rules.js` + 判定 prompt（`_base.md` + 各分类 `.md`），落到 `domains/<team>/` 下，即可接入判定服务。

**③ 增量同步（`discover-triage-urls` 的 sync 模式）—— 接入后的持续维护**
- **场景**：项目后续迭代新增了接口，url 清单需要跟着更新。
- **价值**：接入不是一次性的 —— 业务在长，配置也能低成本跟着长；只处理增量，token 消耗 ∝ 新增数量而非项目大小。

**怎么用（触发方式）**：无需手动指定"增量"。之前跑过全量、项目里已有 `<team>-urls.js` 时，**再调一次同一个 team + 项目路径**（如"同步/更新 flight 清单"），skill 检测到清单已存在 → **自动走 sync 模式**；不存在才走全量。

**底层四步**：

1. **纯 grep 算 delta（零 LLM）**：跑自带脚本 `scripts/drift-check.sh <项目/src> <team-urls.js> [关注前缀…]`，输出「新增未接入」+「疑似删除/改名」两组，有漂移退出码 1。
2. **只对新增的少数 url 跑语义分类**（Explore agent），不重读全项目 —— 这是省 token 的关键。
3. **疑似删除 → 标待确认**（不擅自删，可能是改名/换前缀）；不变的原样保留。
4. **合并回清单 + 更新 commit SHA**，列出「新增/删除/待确认」。新增项若都落进已有分类，往往只在 `rules.js` 补几条 pattern、prompt 不动；涉及新分类/新核心接口才再跑 `onboard-triage-domain`。

```
首次接入： 项目源码 ─[discover 全量]→ url清单 ─确认→ [onboard]→ domain 骨架
后续维护： 新增接口 ─[drift-check 哨兵]→[discover sync 仅增量]→ 确认 → 更新 rules/prompt
```

**两个关键认知**：
- **同步不紧迫、可低频批量做**：新接口即使还没进清单，引擎也**不会漏报** —— 自有前缀内未命中规则 → 落 `fallbackKey` 保守上报，非自有 → non-core 兜底。清单过期只是 **severity 精度下降，不是漏报**。
- **`drift-check.sh` 是哨兵**：纯 grep、零成本，**可挂 CI / pre-commit** 自动盯接口漂移，不用人惦记。

> 这条流水线正是"可配置化 + 快速接入 + 持续维护"的落地方式：从零接入主要工作是**确认 url 清单**和**微调 prompt 分级基线**；日常迭代靠 sync 增量跟进，都不需要碰引擎代码。

## 十二、判定质量调优（eval 评测闭环）

现状是"人肉 few-shot"调 prompt，样本量不足、改好改坏靠感觉。下一步用**数据驱动的评测闭环**把它规模化：真实异常拉进来 → 变成带标准答案的评测集 → 每改一版 prompt 就跑分对比。

```
① MCP 拉异常 → ② 按指纹去重成 case → ③ 标 gold（模型草标+人工纠正）
      → ④ run.js 跑判定（接生产模型，只判定不派发）→ ⑤ score.js 打分
      → ⑥ 错误分析归因到具体 prompt 规则 → ⑦ 改 prompt
      → 数据集冻结重跑（回归测试）↺
```

**已落地的脚手架**（`src/plugins/error-triage/eval/`）：

| 文件 | 作用 |
|---|---|
| `datasets/<domain>.jsonl` | 评测集：每行一个 case（`alarm` + `gold` 标准答案 + `split`）|
| `run.js` | 读数据集 → 逐条真实判定（同路由/prompt/模型，**只判定不派发**）→ outputs |
| `score.js` | outputs vs gold → 指标 + 失败清单，train / holdout 分开出 |
| `reports/vN.md` | 每版指标快照，版本对比即回归测试 |

**指标分开看（不看总准确率）**：**漏报率 FN 为头号指标**（呼应"宁可误报不可误杀"，必须压到极低）、误报率 FP、severity 一致率、error_type 准确率。

**关键纪律**：漏报优先；数据集冻结保证可比；train 调 / holdout 验防过拟合；最终数字用 DCC 生产模型跑；正负样本平衡（避免"一律不报"刷高分）。

> 价值：prompt 从"凭经验改"升级为"有考卷、有分数、有回归"，每次调整都能量化是否变好、有没有引入退化。

**prompt 谁来调 —— 人机协作**：不是纯人工、也不一上来全自动，而是 **AI 做错误分析 + 出具体改稿，人 review 采纳并守住 gold**。分工：AI 擅长「归因 + 出 prompt diff」；人握住「gold 标注、severity 基线/business_reject 语义、上线终审」。后续可升级为**有护栏的自动调优循环**，四道护栏缺一不可：holdout 不参与调（防过拟合）、gold 冻结 AI 碰不到（防刷分作弊）、FN 与 FP 一起卡（防"一律上报"钻空子）、上线前留人闸门。

> 一句话：**改 prompt 的"手"可以是 AI，但"标准答案(gold)"和"最终拍板"必须在人手里。**

**对外服务化（其他团队自助调优，不集成项目）**：跑判定本质就是批量调判定服务，所以 eval 也对外做成 API，`server.js` 已加两个 endpoint：

| Endpoint | 作用 |
|---|---|
| `POST /judge` | 只判定、不派发 —— 回放/调优用，不会触发真实告警 |
| `POST /eval` | 批量判定 + 对比 gold 打分 —— `{dataset} → {report, outputs}` |

分工：**中心侧**持有引擎、模型凭证、统一打分口径；**团队侧**用自己的 MCP 拉数据、拥有自己的数据集与 gold、在配置中心改自己的 prompt。团队循环 = 拉数据 → 标 gold → 调 `/eval` → 改 prompt → 再调，全程不碰项目代码。演进路线与 domain 配置一致：**阶段一试点内嵌（我们代跑）→ 阶段二 eval-as-a-service**（多团队按 `domain` 路由、数据集按团队存中心）。

## 十三、关于结果一致性的说明

判定质量取决于 **模型 + prompt**。相同模型、相同 prompt下，判定方向**稳定可复现**；但严重级别处于临界点（P2↔P1）时，建议以生产实际调用的模型（DCC 接口）实跑结果为准。上线前会用真实接口对齐一批样例，验证 prompt 在目标模型上的表现。
