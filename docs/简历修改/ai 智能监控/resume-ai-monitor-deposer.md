### 版本 A：精简版（1 段，空间有限时用）

**商旅接口错误告警智能降噪系统** | Node.js + LLM

面向机酒火线上接口错误告警噪音高、真故障易被淹没的问题，参与设计并落地「每条告警走一次 LLM」的降噪判定服务：按 url 路由到业务分类后，用可插拔 Domain（路由表 + Prompt）输出是否上报、P0–P3 等级与中文依据；引擎领域无关，换团队只换配置。配套 discover / onboard 两个 Skill 完成接口清单摘取与 Domain 生成，并以漏报率 FN 为头号指标搭建评测闭环；LLM 故障时统一保守上报，保证不漏告警。

---

### 版本 B：标准版（3–4 条 bullet，推荐）

**商旅接口错误告警智能降噪（error-triage）** | Node.js / LLM / Prompt 工程

- 针对商旅接口告警中「业务正常拒绝、展示类噪音、三方异常」刷屏问题，落地 **告警到达值班前** 的智能判定：`should_report` + `severity(P0–P3)` + `error_type` + 中文 `reason`
- 设计 **引擎领域无关 + Domain 可插拔** 架构：HTTP/Kafka 双入口 → 路由 → Prompt 编排 → LLM 网关；业务知识全部下沉到 `domains/<team>/`（`rules.js` + `prompts/`），换团队零改引擎
- 落地 **宁可误报不可误杀**：信息不足默认上报；超时/解析失败统一 `should_report=true, P2`；并以 **漏报率 FN** 为评测头号指标，配合 train/holdout 与数据集冻结做 Prompt 回归
- 建设接入流水线：`discover-triage-urls`（广召回摘接口 + drift-check 增量）→ 人工确认 → `onboard-triage-domain`（生成路由表与 Prompt 骨架）；火车票 Domain 已接入，机酒 url 清单已摘取

---

### 版本 C：突出 AI 工程版（AI / Agent 向岗位）

**AI 告警降噪判定服务** | LLM 编排 · Agent Skills · Eval 闭环

- 核心理念：**判定交给模型，工程只做路由与编排**；调阈值 = 改 Prompt，不做规则旁路预过滤，保证判定质量
- 用 Claude Code Skills 把「散落业务代码 → 可用 Domain」产品化：探针 Skill 广召回核心链路 url，接入生成器 Skill 产出 `_base` + 分类 Prompt；`drift-check.sh` 纯 grep 挂 CI 盯接口漂移
- 搭建数据驱动调优：`/judge`（只判定不派发）+ `/eval`（批量 vs gold 打分）；人机协作调 Prompt（AI 归因出 diff，人守 gold 与上线终审）
- 规划试点内嵌 → 中心化配置 + eval-as-a-service，使其他团队自助接入而不必集成引擎代码

## 二、一句话 / 电梯演讲（30 秒）

> 商旅线上接口错误告警噪音很大，无票、已退改这类业务正常拒绝会淹没真故障。我参与做的 error-triage，是在告警派发前用大模型对每条错误做一次判定——该不该报、定什么 P 级。工程侧只做路由和编排，业务知识全在可插拔的 Domain 配置里；并且明确「宁可误报不可误杀」，模型挂了也要保守上报。
