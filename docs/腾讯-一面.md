## 1. 实习中，重构动态表单引擎数据结构设计？耦合的表单数据怎么处理？

## 答案

### 原有痛点

- 57 个页面，每个页面表单逻辑硬编码
- 字段变更需要前后端同时修改，改动成本高
- 代码重复率高，维护困难

### 重构方案：JSON Schema 驱动

**数据结构设计：**

```json
{
  "type": "object",
  "title": "特殊价政策配置",
  "required": ["policyName", "airline"],
  "properties": {
    "policyName": {
      "type": "string",
      "title": "政策名称",
      "ui:placeholder": "请输入政策名称"
    },
    "airline": {
      "type": "string",
      "title": "航空公司",
      "enum": ["CA", "MU", "CZ"],
      "enumNames": ["国航", "东航", "南航"]
    },
    "refundRule": {
      "type": "object",
      "title": "退票规则",
      "properties": {
        "beforeDeparture": { "type": "number", "title": "起飞前(天)" },
        "refundPercent": { "type": "number", "title": "退款比例(%)" }
      }
    }
  }
}
```

### 耦合表单数据处理

**1. Schema 与 UI 分离**
- Schema 定义字段结构、校验规则
- `ui:` 前缀扩展字段定义 UI 展示（widget、placeholder、disabled 等）
- 完全解耦字段逻辑和 UI 渲染

**2. 动态表单引擎核心实现**

```typescript
// 字段路径扁平化，解决嵌套数据的耦合
function flattenPath(schema: JsonSchema, prefix = ''): Record<string, FieldConfig> {
  const result = {};
  for (const [key, value] of Object.entries(schema.properties || {})) {
    const path = prefix ? `${prefix}.${key}` : key;
    result[key] = {
      path,
      schema: value,
      widget: value['ui:widget'] || getDefaultWidget(value.type),
      rules: value['ui:rules'] || []
    };
    if (value.type === 'object') {
      Object.assign(result, flattenPath(value, path));
    }
  }
  return result;
}
```

**3. 数据聚合与回填**
- 表单数据以扁平 path 为 key 存储
- 提交时自动聚合成嵌套结构，发送给后端
- 编辑时根据 path 回填到对应字段

**4. 与 Ant Design 组件映射**

| JSON Schema type | Ant Design 组件 |
|-------------------|-----------------|
| string            | Input           |
| string (enum)     | Select          |
| number            | InputNumber     |
| boolean           | Switch          |
| string (date)     | DatePicker      |
| array             | Upload          |

**重构收益：**
- 新增字段：2-3 天 → 配置即生效
- 代码复用率：20% → 80%+
- 57 个页面全面覆盖

## 2. JS闭包 使用不当出现的问题

## 答案

### 闭包形成原理

函数能访问其创建时作用域中的变量，即使函数在其作用域外部执行。

```javascript
function outer() {
  const a = 1;
  return function inner() {
    console.log(a); // 闭包引用了 outer 的变量
  };
}
```

### 使用不当造成的问题

**1. 内存泄漏**

```javascript
// 闭包持有大对象引用，无法被 GC
function leak() {
  const bigData = new Array(1000000);
  const cache = {};

  return function() {
    cache.big = bigData; // cache 持续增长，永不释放
    console.log(cache);
  };
}
```

**2. 循环引用导致的内存泄漏**

```javascript
// 事件处理器中的循环引用
function attachHandler() {
  const element = document.getElementById('btn');
  const data = new Array(10000);

  element.addEventListener('click', function() {
    // 闭包引用了 element，element 也引用了这个函数
    // 形成循环：如果未移除，GC 无法回收
    console.log(data);
  });
  // 正确的做法：
  // element.removeEventListener('click', handler);
  // 或使用 { once: true }
}
```

**3. 循环中闭包值固定**

```javascript
// 错误：所有 setTimeout 输出相同值
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100); // 输出 3, 3, 3
}

// 原因：var 是函数作用域，i 是同一个变量
// 解决1：用 let（块级作用域）
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100); // 输出 0, 1, 2
}

// 解决2：IIFE 捕获
for (var i = 0; i < 3; i++) {
  (function(j) {
    setTimeout(() => console.log(j), 100);
  })(i);
}
```

**4. 多个闭包共享同一变量（意外修改）**

```javascript
function createCounter() {
  let count = 0;

  return {
    increment: () => ++count,
    decrement: () => --count,
    getCount: () => count,
    // 错误：如果直接返回 count，所有操作都会修改同一个值
  };
}
```

**5. this 指向问题**

```javascript
const obj = {
  name: 'obj',
  // 错误：setTimeout 回调中的 this 指向 window
  delayed: function() {
    setTimeout(function() {
      console.log(this.name); // undefined（严格模式下）
    }, 100);
  },
  // 正确：用箭头函数或 bind
  delayedFixed: function() {
    setTimeout(() => console.log(this.name), 100); // obj
  }
};
```

### 避免策略

- 谨慎持有外部引用，尽量弱引用
- 及时清理事件监听器和定时器
- 用 let/const 替代 var，避免作用域问题
- 大型对象使用 WeakMap/WeakSet 存储

## 3. JS垃圾回收机制是什么？需要人工操作吗？

## 答案

### JavaScript 垃圾回收（GC）机制

JavaScript 引擎（如 V8、SpiderMonkey）自动管理内存，开发者无需手动分配和释放。

### 核心算法

**1. 引用计数（早期方案）**

```javascript
let obj = { a: 1 }; // obj 引用对象，计数 = 1
let ref = obj;      // 计数 = 2
ref = null;         // 计数 = 1（ref 不再引用）
obj = null;         // 计数 = 0，可被回收
```

**问题：循环引用无法回收**
```javascript
function cycle() {
  let a = {};
  let b = {};
  a.ref = b; // b 计数 +1
  b.ref = a; // a 计数 +1
  // 函数结束后 a、b 计数都是 1，无法回收
}
```

**2. 标记-清除（V8 主用）**

```
GC 启动
    ↓
从根对象（window、global）出发，标记所有可达对象
    ↓
遍历堆，回收未被标记的对象
    ↓
内存整理（减少碎片）
```

**可达性原则**：从根对象出发，能通过引用链访问到的对象是"活对象"，否则回收。

**3. V8 的分代回收**

| 分代 | 对象年龄 | 回收频率 | 算法 |
|------|----------|----------|------|
| 年轻代 | < 1 次 GC | 高（毫秒级） | Scavenge（复制） |
| 老生代 | 经历多次 GC | 低 | 标记-清除 + 标记-压缩 |
| 大对象区 | > 数百KB | 极低 | 标记-清除 |
| 代码区 | 存 JIT 编译代码 | 极低 | — |

**增量标记 + 懒清理**：将 GC 过程拆分成小增量，避免长停顿（Stop The World）。

### 需要人工操作吗？

**不需要手动触发 GC**，但开发者需要：
- 及时解除不需要的引用（`obj = null`）
- 清理事件监听（`removeEventListener`）
- 避免意外的全局变量
- 谨慎使用闭包持有大对象

### 排查内存问题

```javascript
// 1. Chrome DevTools Memory Profiler
// 2. performance.memory（Chrome 提供）
console.log(performance.memory);

// 3. heap snapshot 对比
// 4. performanceonitor 监控实时内存
```

## 4. Promise.all、Promise.race、Promise.allSettled 的区别和应用场景？

## 答案

### 核心区别

| 方法 | 成功条件 | 失败行为 | 返回值 |
|------|----------|----------|--------|
| `Promise.all` | 全部成功 | 任一失败，整体失败 | 结果数组（按顺序） |
| `Promise.race` | 任一先完成（无论成功/失败） | 无（总会成功） | 首个结果 |
| `Promise.allSettled` | 全部完成 | 不会失败 | 结果数组（每项包含 status） |
| `Promise.any` | 任一成功 | 全部失败才失败 | 首个成功结果 |

### Promise.all

```typescript
Promise.all([
  fetchUser(),
  fetchPermissions(),
  fetchSettings()
])
  .then(([user, permissions, settings]) => {
    // 全部成功后，按顺序获取结果
  })
  .catch(err => {
    // 任一失败，整体 reject
  });
```

**场景**：并行获取多个相关数据，所有数据都就绪才继续（如页面初始化需要同时获取用户信息、权限、配置）。

### Promise.race

```typescript
Promise.race([
  fetchData(),
  new Promise((_, reject) =>
    setTimeout(() => reject(new Error('Timeout')), 5000)
  )
]).then(data => console.log(data));
```

**场景**：超时控制、图片加载（哪个先到用哪个）。

### Promise.allSettled

```typescript
const results = await Promise.allSettled([
  fetchUser(),
  fetchOrders(),
  fetchRecommendations()
]);

results.forEach((result, i) => {
  if (result.status === 'fulfilled') {
    console.log(`任务${i}成功:`, result.value);
  } else {
    console.log(`任务${i}失败:`, result.reason);
  }
});
```

**场景**：并行执行多个独立任务，不管成功失败都要收集结果（如发送多个埋点请求、批量导入数据）。

### Promise.any（ES2021）

```typescript
Promise.any([
  fetchFromCDN(),
  fetchFromServer1(),
  fetchFromServer2()
]).then(firstSuccess => console.log(firstSuccess));
```

**场景**：从多个数据源并行获取，优先使用最快返回的（如多路 CDN 获取资源）。

## 5. React 相关原理以及生命周期

## 答案

### React 核心原理

**1. 虚拟 DOM（Virtual DOM）**

用 JavaScript 对象描述真实 DOM 树：

```javascript
// 虚拟 DOM 节点
{
  type: 'div',
  props: { className: 'container', id: 'app' },
  children: [
    { type: 'h1', props: {}, children: ['Hello'] }
  ]
}
```

**为什么用 VDOM**：直接操作 DOM 昂贵，VDOM 通过 Diff 算法批量更新，减少 DOM 操作。

**2. 调和（Reconciliation）**

React 15 及以前：同步递归调和，递归深度优先，无法中断。

React 16 Fiber：**可中断的异步渲染**。

```
旧架构（Stack Reconciler）：
render() → 递归构建 VDOM 树 → 同步完成，无法中断

新架构（Fiber Reconciler）：
render() → 构建 Fiber 链表（可中断）→ requestIdleCallback 让出主线程
           → 高优先级任务（用户输入）可打断低优先级（渲染）
```

**Fiber 双缓存**：
```
current Fiber Tree (正在显示)
        ↕ 切换
workInProgress Fiber Tree (内存中构建)
```

**3. Diff 算法**

三大策略：
- **同层比较**：只比较同一层级，不跨层级
- **类型不同直接替换**：`div` → `span`，整个子树替换
- **列表比较用 Key**：通过 Key 识别相同元素，减少移动

### Class 组件生命周期

```
挂载阶段：
constructor() → static getDerivedStateFromProps → render → componentDidMount

更新阶段：
static getDerivedStateFromProps → shouldComponentUpdate → render
  → getSnapshotBeforeUpdate → componentDidUpdate

卸载阶段：
componentWillUnmount
```

**16.3 后 lifecycle 改动**：
- `componentWillMount/Unmount/Update` 标记 UNSAFE_
- 新增 `getDerivedStateFromProps`（替代旧的派生状态逻辑）
- 新增 `getSnapshotBeforeUpdate`（替代 `componentWillUpdate`）

### Hooks 生命周期（函数组件）

```javascript
useEffect(() => {
  // componentDidMount + componentDidUpdate
  // 副作用逻辑

  return () => {
    // componentWillUnmount
    // 清理逻辑
  };
}, [deps]); // 依赖数组：控制何时触发
```

| Class 生命周期 | Hook 等价 |
|----------------|-----------|
| componentDidMount | useEffect([], ...) |
| componentDidUpdate | useEffect([deps], ...) |
| componentWillUnmount | useEffect return |
| shouldComponentUpdate | React.memo / useMemo |

## 6. React Hooks 设计是为了解决什么问题？

## 答案

### Class 组件的问题

**1. 逻辑复用困难**

- HOC 和 Render Props 产生「包装地狱」
- Props 层层传递，复杂度增加
- 逻辑与 UI 强耦合

**2. 状态逻辑分散**

相关代码分散在不同的生命周期方法中：

```javascript
class UserProfile extends React.Component {
  componentDidMount() { /* 获取用户 */ }
  componentDidUpdate(prevProps) { /* 响应 ID 变化 */ }
  componentWillUnmount() { /* 清理订阅 */ }

  // 用户相关的逻辑散落在 3 个生命周期中
}
```

**3. this 指向复杂**

```javascript
class Button extends React.Component {
  this.state = { clicked: false };
  // 需要 bind this 或用箭头函数
  handleClick = () => this.setState({ clicked: true });
}
```

### Hooks 解决的问题

**1. 逻辑复用：自定义 Hook**

```javascript
// 把相关逻辑提取到自定义 Hook
function useUser(userId) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetchUser(userId)
      .then(setUser)
      .finally(() => setLoading(false));
  }, [userId]);

  return { user, loading };
}

// 在组件中使用，逻辑清晰、可复用
function Profile() {
  const { user, loading } = useUser('123');
}
```

**2. 状态逻辑聚合**

```javascript
function UserProfile({ userId }) {
  // 相关逻辑集中在一起
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // 获取用户
    fetchUser(userId).then(setUser).finally(() => setLoading(false));

    // 订阅更新
    const unsub = subscribe(userId, setUser);
    return () => unsub(); // 清理
  }, [userId]);
}
```

**3. 去掉 this**

```javascript
// 函数组件无需处理 this
function Button() {
  const [clicked, setClicked] = useState(false);
  const handleClick = () => setClicked(true);
}
```

### Hooks 规则

- 只能在函数组件顶层调用（不能放在条件、循环中）
- 只能用 React 函数中（不能在普通函数中调用）
- `useEffect` 依赖数组必须正确，否则可能死循环或陈旧闭包

## 7. key 对于 Diff 的作用

## 答案

### key 的作用

React 的 Diff 算法在比较列表时，**key 用于判断元素是否在变化前后是同一个**。

### 无 key 的问题

```jsx
// 没有 key，React 按索引比较
<div>{items.map((item, i) => <div>{item.name}</div>)}</div>

// items = [A, B, C] → [A, B, C, D]（末尾追加）
// React 会比较：0-A vs 0-A，1-B vs 1-B，2-C vs 2-C，D 是新节点 ✓

// items = [A, B, C] → [A, X, B, C]（中间插入 X）
// React 比较：0-A vs 0-A（匹配），1-B vs 1-X（不匹配，更新）
//          2-C vs 2-B（不匹配，更新），3-C vs 3-C（新增）
// 问题：B 和 C 被错误地更新了（实际只是位置变化）
```

### 有 key 的作用

```jsx
// 有 key，React 按 key 比较
<div>{items.map(item => <div key={item.id}>{item.name}</div>)}</div>

// items = [A, B, C] → [A, X, B, C]
// React 比较：
// key=A → key=A（匹配，位置变了但元素相同，复用）
// key=X → 全新节点，新增
// key=B → key=B（匹配，复用）
// key=C → key=C（匹配，复用）
// 结果：只新增 X，其他节点复用 ✓
```

### 最佳实践

**1. 用稳定且唯一的 ID**

```jsx
// ✓ 好：稳定的业务 ID
items.map(item => <div key={item.id}>)

// ✗ 差：用数组索引
items.map((item, i) => <div key={i}>)

// ✗ 差：用随机数（每次渲染 key 都变）
items.map(item => <div key={Math.random()}>)
```

**2. 索引作为 key 的适用场景**

- 静态列表，不会增删
- 列表不会重新排序
- 列表项不会被删除或新增中间项

**3. key 相同但位置变化**

```jsx
// 列表反转，key 相同 → React 会识别为「只是位置变化」，复用节点
// ✓ 性能好
[1,2,3].map(id => <div key={id}>)

// 不用 key 或用索引 → React 认为是「删除旧节点 + 新增节点」
// ✗ 性能差，DOM 操作翻倍
[1,2,3].map((id, i) => <div key={i}>)
```

### key 不影响组件内部状态

key 改变时，React 认为是**新组件**，会卸载旧组件并挂载新组件，状态会重置。

## 8. 浏览器输入网址到渲染的完整过程

## 答案

### 完整流程

```
用户输入 URL
    ↓
1. DNS 解析
    ↓
2. TCP 连接（三次握手）
    ↓
3. TLS 握手（HTTPS）
    ↓
4. HTTP 请求发送
    ↓
5. 服务器处理
    ↓
6. HTTP 响应
    ↓
7-15. 浏览器渲染 pipeline
    ↓
16. JavaScript 框架渲染（React/Vue）
```

### 1. DNS 解析

```
浏览器缓存 → 系统缓存（hosts） → 路由器缓存 → ISP DNS 缓存
    → 递归查询 DNS 服务器
```

### 2-3. TCP + TLS

```
TCP 三次握手：
Client: SYN=1, seq=x          → Server
Server: SYN=1, ACK=1, seq=y   → Client
Client: ACK=1                  → Server
(连接建立)

TLS 握手（HTTPS）：
Client: 支持的加密套件 + Random1
Server: 证书 + Random2 + 选择加密套件
Client: 验证证书 + PreMasterSecret（用公钥加密）
两者: 用 Random1 + Random2 + PMS 生成对称密钥
```

### 4-6. HTTP 请求/响应

```
浏览器发送请求头、cookies
服务器处理（负载均衡、Web 服务器、应用服务器）
返回 HTTP 响应（含 HTML/CSS/JS 等资源）
```

### 7-15. 浏览器渲染 Pipeline

```
7.  解析 HTML → DOM 树
8.  解析 CSS → CSSOM 树
9.  合成 DOM + CSSOM → Render Tree（渲染树）
10. Layout（布局/回流）：计算每个节点几何信息
11. Paint（绘制）：填充像素
12. Composite（合成）：多层合成（transform/opacity 可跳过 Layout/Paint）
```

**关键优化点**：
- `defer`/`async` 脚本：不阻塞 HTML 解析
- CSS `<link>` 在 `<head>`：避免 FOUC（无样式内容闪烁）
- 图片懒加载：`loading="lazy"`
- 预解析 DNS：`<link rel="dns-prefetch">`

### 16. JS 框架渲染（React 为例）

```
JS Bundle 下载 → JavaScript 执行 → React 组件渲染
    → Virtual DOM 构建 → Diff 比较 → 最小化 DOM 更新
```

## 9. 重绘和重排（回流）触发条件

## 答案

### 定义

| 类型 | 定义 | 代价 |
|------|------|------|
| **重绘（Repaint）** | 外观改变（颜色、背景），不影响布局 | 较低 |
| **重排/回流（Reflow）** | 几何属性变化（尺寸、位置），需重新计算布局 | 高 |

**重排必定触发重绘，重绘不一定重排**。

### 触发重排（Reflow）的条件

```javascript
// 几何属性变化
element.style.width = '100px';
element.style.height = '200px';
element.style.padding = '10px';
element.style.margin = '5px';
element.style.position = 'fixed'; // 定位变化

// 读取布局属性（强制触发回流）
const width = element.offsetWidth;  // 读取 → 回流
const height = element.scrollHeight;
const top = element.getBoundingClientRect().top;

// 滚动滚动条（viewport 滚动）
window.scrollY = 100;

// 添加/移除 DOM 元素
document.body.appendChild(newDiv);

// 字体变化
document.body.style.fontSize = '16px';
```

### 触发重绘（Repaint）的条件

```javascript
// 外观属性变化，不影响布局
element.style.color = 'red';
element.style.backgroundColor = '#f0f0f0';
element.style.border = '1px solid blue';
element.style.visibility = 'hidden'; // visibility 不触发 reflow
element.style.opacity = '0.5'; // opacity 不触发 reflow
```

### 优化策略

**1. 批量修改样式**

```javascript
// 错误：多次触发 reflow
element.style.width = '100px';
element.style.height = '100px';
element.style.margin = '10px';

// 正确：用 class 批量应用
element.classList.add('active');
```

**2. 用 transform 代替 top/left**

```javascript
// 错误：触发 reflow
element.style.top = '100px';
element.style.left = '100px';

// 正确：只触发 composite，不触发 reflow
element.style.transform = 'translate(100px, 100px)';
```

**3. 避免频繁读取布局属性**

```javascript
// 错误：每次读取都触发 reflow
const width = element.offsetWidth;
element.style.width = (width + 1) + 'px';
const height = element.offsetHeight;

// 正确：缓存值
const { offsetWidth, offsetHeight } = element;
element.style.width = (offsetWidth + 1) + 'px';
```

**4. 使用 DocumentFragment**

```javascript
// 错误：多次 reflow
for (let i = 0; i < 100; i++) {
  const div = document.createElement('div');
  container.appendChild(div);
}

// 正确：一次性添加，只触发一次 reflow
const fragment = document.createDocumentFragment();
for (let i = 0; i < 100; i++) {
  fragment.appendChild(document.createElement('div'));
}
container.appendChild(fragment);
```

**5. 动画元素用 fixed/absolute**

```javascript
// 让动画元素脱离文档流，避免影响其他元素布局
element.style.position = 'absolute';
```

## 10. 跨域中的同源策略？解决跨域的手段有哪些？

## 答案

### 同源策略（Same-Origin Policy）

浏览器的安全机制，限制「源 A」的资源访问「源 B」的数据。

**同源判断**：`协议 + 域名 + 端口` 三者一致。

| URL A | URL B | 是否同源 |
|-------|-------|----------|
| `https://a.com` | `https://a.com/page` | ✓ |
| `https://a.com` | `http://a.com` | ✗ 协议不同 |
| `https://a.com` | `https://b.com` | ✗ 域名不同 |
| `https://a.com` | `https://a.com:8080` | ✗ 端口不同 |
| `https://a.com` | `https://sub.a.com` | ✗ 子域名不同 |

### 限制范围

- Cookie、LocalStorage、IndexedDB 无法跨域读写
- DOM 无法跨域访问（iframe 间限制）
- AJAX 请求（XMLHttpRequest/fetch）被限制

### 解决跨域的手段

**1. CORS（跨域资源共享）—— 主流方案**

服务端设置响应头：

```javascript
// Node.js express
app.use((req, res, next) => {
  res.header('Access-Control-Allow-Origin', 'https://a.com'); // 允许的源
  res.header('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE');
  res.header('Access-Control-Allow-Headers', 'Content-Type, Authorization');
  res.header('Access-Control-Allow-Credentials', 'true'); // 允许携带 cookie
  next();
});
```

**2. JSONP（已过时，仅历史场景）**

利用 `<script>` 标签不受同源限制：

```html
<script src="https://b.com/api?callback=handleData"></script>
<script>
  function handleData(res) { console.log(res); }
</script>
```

局限：只支持 GET，不支持 POST，且有安全风险。

**3. 代理服务器**

同源策略是浏览器限制，服务器之间无此限制：

```
浏览器 → 自己的代理服务器(/api) → 代理转发请求 → 目标服务器
```

前端请求同域代理，避免跨域，Nginx/Node 代理：

```nginx
# Nginx 反向代理
location /api {
  proxy_pass https://b.com/api;
}
```

**4. WebSocket（不受同源限制）**

```javascript
const ws = new WebSocket('wss://b.com/socket');
ws.onopen = () => ws.send('Hello');
```

**5. postMessage**

```javascript
// A 页面发送消息到 iframe
iframe.contentWindow.postMessage('data', 'https://b.com');

// B 页面监听
window.addEventListener('message', event => {
  if (event.origin === 'https://a.com') {
    console.log(event.data);
  }
});
```

**6. 跨域 POST MESSAGE 场景（RN / 小程序）**

- `postMessage` API
- `window.location.href` 跳转 + `window.name`
- 短轮询 / 长轮训

## 11. 预检请求是什么？为了处理什么问题？

## 答案

### 预检请求（Preflight Request）

跨域 AJAX 请求前，浏览器自动发送的 `OPTIONS` 请求，询问服务器是否允许该请求。

### 触发条件

以下任一条件满足，浏览器发送预检请求：

- 请求方法是 `PUT`、`DELETE`、`CONNECT`、`OPTIONS`、`TRACE`、`PATCH`
- `Content-Type` 不是：`application/x-www-form-urlencoded`、`multipart/form-data`、`text/plain`
- 请求头包含自定义字段（如 `Authorization`、`X-Custom-Header`）

### 预检流程

```
浏览器: 发送 OPTIONS 请求（预检）
  ↓
服务器: 检查 Origin、Access-Control-Request-Method、Access-Control-Request-Headers
  ↓
服务器: 返回允许的方法和头
  ↓
浏览器: 预检通过后，发送实际请求
  ↓
服务器: 返回实际响应
```

### 预检请求示例

**实际请求：**

```javascript
fetch('https://api.b.com/users', {
  method: 'PUT',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': 'Bearer token123'
  },
  body: JSON.stringify({ name: 'Tom' })
});
```

**浏览器发送的预检（OPTIONS）：**

```
OPTIONS /users HTTP/1.1
Origin: https://a.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Content-Type, Authorization
```

**服务器响应：**

```
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://a.com
Access-Control-Allow-Methods: PUT, GET, POST
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400  // 预检结果缓存 86400 秒
```

### 预检解决的问题

**安全考虑**：实际请求中可能包含敏感操作（PUT/DELETE），浏览器先确认服务器「认识这个请求」再发送。

**减少风险**：恶意网页无法在用户不知情的情况下发送复杂请求到其他域。

### 避免预检的优化

- 尽量用 `GET/POST`
- `Content-Type` 用 `application/x-www-form-urlencoded`
- 避免自定义请求头（合并到 body 或 query string）
- 合理设置 `Access-Control-Max-Age`，减少预检次数

## 12. 浏览器缓存机制

## 答案

### 缓存分类总览

```
请求资源
    ↓
强缓存（本地）→ 命中 → 直接使用 → 200 (from cache)
    ↓ 未命中
协商缓存（服务端）→ 命中 → 返回 304（未修改）
    ↓ 未命中
返回新资源 → 更新缓存 → 200
```

### 1. 强缓存（Strong Cache）

不发请求到服务器，直接使用本地缓存。

**Cache-Control（优先级高）：**

| 指令 | 作用 |
|------|------|
| `max-age=3600` | 3600 秒后过期 |
| `no-cache` | 不直接使用缓存，需协商 |
| `no-store` | 完全不使用缓存 |
| `private` | 只能浏览器缓存，CDN 不能 |
| `public` | 浏览器和 CDN 都能缓存 |

**Expires（HTTP/1.0，低优先级）：**

```
Expires: Wed, 21 Oct 2026 07:28:00 GMT
```

### 2. 协商缓存（Conditional Cache）

发请求到服务器，由服务器判断是否使用缓存。

**ETag / If-None-Match（优先级高，内容hash）：**

```
响应头: ETag: "33a64df551425fcc55e4d42a148795d9"
请求头: If-None-Match: "33a64df551425fcc55e4d42a148795d9"
```

服务器对比资源的 hash，变化才返回新内容。

**Last-Modified / If-Modified-Since（时间戳）：**

```
响应头: Last-Modified: Wed, 21 Oct 2026 07:28:00 GMT
请求头: If-Modified-Since: Wed, 21 Oct 2026 07:28:00 GMT
```

### 3. 缓存策略实践

| 资源类型 | 策略 | 原因 |
|---------|------|------|
| HTML | `Cache-Control: no-cache` | 必须及时获取最新版本 |
| 静态资源 (CSS/JS/图片) | `max-age=31536000` + 文件指纹 | 长期缓存，变更后 URL 变 |
| API 接口 | `Cache-Control: no-store` | 数据敏感，不缓存 |

### 4. Service Worker 缓存

```javascript
// 缓存优先策略
self.addEventListener('fetch', event => {
  event.respondWith(
    caches.match(event.request).then(cached => {
      return cached || fetch(event.request);
    })
  );
});
```

### 5. 缓存流程图

```
浏览器请求
    ↓
查 Service Worker 缓存
    ↓
查 memory cache（内存）
    ↓
查 disk cache（磁盘）
    ↓
发送网络请求（带 ETag/Last-Modified）
    ↓
服务器判断 → 304（用缓存）或 200（新资源）
```

## 13. CSRF 攻击原理

## 答案

### CSRF（跨站请求伪造）

攻击者诱导受害者在已登录的网站上执行非本意的操作。

### 攻击原理

```
1. 用户登录银行网站 A，获取 Session Cookie
2. 攻击者构造恶意页面 B
3. 用户访问页面 B
4. 页面 B 自动发起转账请求到银行 A：
   <img src="https://bank.com/transfer?to=attacker&amount=10000">
5. 浏览器自动带上 bank.com 的 Cookie
6. 服务器验证 Cookie 有效，执行转账
```

### CSRF 攻击的必要条件

- 用户在目标网站已登录（持有有效 Session/Cookie）
- 目标网站无严格的 CSRF 防护
- 用户在登录状态下访问了攻击者的恶意页面

### 防护手段

**1. CSRF Token**

```html
<form method="POST" action="/transfer">
  <input type="hidden" name="csrf_token" value="abc123xyz">
  <button type="submit">转账</button>
</form>
```

服务器验证 Token，攻击者页面无法获取（同源策略限制）。

**2. SameSite Cookie**

```http
Set-Cookie: sessionId=xxx; SameSite=Strict; Secure
```

| 值 | 行为 |
|----|------|
| `Strict` | 仅在同站请求时携带 |
| `Lax` | 导航请求（点击链接）携带，POST 图片等不携带 |
| `None` | 任意请求都携带（需 Secure） |

**3. 验证 Origin / Referer**

```javascript
app.use((req, res, next) => {
  const origin = req.headers.origin;
  if (origin !== 'https://a.com') {
    return res.status(403).json({ error: '禁止跨域' });
  }
  next();
});
```

**4. 双重提交 Cookie**

```javascript
// 攻击页面无法读写目标域的 Cookie
// 但仍会携带原 Cookie
// 服务端验证 Cookie 和请求体中的 CSRF Token 是否一致
```

### XSS vs CSRF

| | XSS | CSRF |
|--|-----|------|
| 攻击原理 | 执行恶意脚本 | 伪造用户请求 |
| 防护 | CSP、输入转义 | CSRF Token、SameSite |
| 攻击位置 | 目标网站执行 | 用户浏览器发送请求 |

## 14. JWT 原理

## 答案

### JWT（JSON Web Token）结构

```
Header.Payload.Signature
xxxxx.yyyyy.zzzzz
```

**示例：**

```
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwiZXhwIjoxNzA0MTY3ODAwfQ.xcrE
```

**Header（头部）：**

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
// Base64 编码
```

**Payload（负载）：**

```json
{
  "sub": "1234567890",    // 用户 ID（subject）
  "name": "Tom",         // 自定义字段
  "roles": ["admin"],
  "iat": 1704167800,     // 签发时间
  "exp": 1704254200      // 过期时间
}
// Base64 编码
```

**Signature（签名）：**

```javascript
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret_key
)
```

### JWT 验证流程

```
1. 用户登录 → 服务器验证密码
2. 服务器签发 JWT → 返回给客户端
3. 客户端存储 JWT（localStorage 或 httpOnly Cookie）
4. 后续请求：Authorization: Bearer <token>
5. 服务器验证 JWT 签名 → 解析 payload → 验证 exp
6. 通过则处理请求
```

### JWT vs Session

| | JWT | Session |
|--|-----|---------|
| 存储位置 | 客户端 | 服务端（内存/Redis） |
| 状态 | 无状态 | 有状态 |
| 扩展性 | 天然分布式 | 需共享 Session |
| 主动失效 | 不能直接失效（需黑名单） | 可直接删除 Session |
| 体积 | 较小 | 极小（只存 Session ID） |
| 安全性 | Token 泄漏风险 | Cookie + httpOnly 更安全 |

### Spring Boot + JWT 示例

```java
// 登录时生成
String token = Jwts.builder()
    .setSubject(user.getUsername())
    .claim("roles", user.getRoles())
    .setExpiration(new Date(System.currentTimeMillis() + 86400000))
    .signWith(key, SignatureAlgorithm.HS256)
    .compact();

// 过滤器验证
Claims claims = Jwts.parserBuilder()
    .setSigningKey(key)
    .build()
    .parseClaimsJws(token)
    .getBody();
```

### JWT 安全注意事项

- Token 存在 httpOnly Cookie 中，避免 XSS 窃取
- 过期时间不宜过长（access_token 建议 1h 内）
- 用 Refresh Token 续期
- Token 泄漏后加入黑名单（Redis）
- 不要在 Token 中存敏感信息（payload 是 Base64，可直接解码）

## 15. Node.js 原理

## 答案

### Node.js 核心特点

- **单线程**：主线程是单线程，但底层用 libuv 处理 I/O
- **非阻塞 I/O**：异步 I/O 操作，不阻塞主线程
- **事件驱动**：基于事件循环处理并发

### 架构

```
┌──────────────────────────────────────┐
│           JavaScript 代码              │
├──────────────────────────────────────┤
│         V8 JavaScript 引擎            │
├──────────────────────────────────────┤
│         Node.js API（ bindings）       │
├──────────────────────────────────────┤
│    libuv（线程池 + 事件循环 + I/O）     │
├──────────────────────────────────────┤
│  系统内核（ epoll/kqueue/IOCP）       │
└──────────────────────────────────────┘
```

### 事件循环（Event Loop）

```javascript
┌─────────────────────────────────────────┐
│                 timers                   │
│    setTimeout / setInterval 的回调        │
├─────────────────────────────────────────┤
│             pending callbacks            │
│       延迟到下一个循环迭代的 I/O 回调     │
├─────────────────────────────────────────┤
│           idle, prepare                  │
│              内部使用                    │
├─────────────────────────────────────────┤
│                 poll                     │
│    检索新的 I/O 事件（文件/网络等）        │
│    执行与 I/O 相关的回调                 │
│    → 若无可用 → 等待可用 → 回到 check    │
├─────────────────────────────────────────┤
│                check                     │
│         setImmediate 的回调              │
├─────────────────────────────────────────┤
│           close callbacks               │
│           socket.on('close')            │
└─────────────────────────────────────────┘
```

### Node.js 与浏览器的 Event Loop 区别

浏览器：任务队列（MacroTask）执行完 → 所有 MicroTask 执行 → 渲染 → 下一个 MacroTask

Node.js： timers → pending → poll → check → close → timers...

**Node.js 特有的 setImmediate**：`setTimeout(fn, 0)` 在 timers 阶段，`setImmediate` 在 check 阶段，顺序不确定（I/O 回调中则 immediate 先执行）。

### 单线程优缺点

**优点：**
- 不存在多线程死锁、竞争问题
- 适合 I/O 密集型（网络请求、文件读写）
- 前后端用同语言，共享代码方便

**缺点：**
- CPU 密集型任务会阻塞主线程
- 无法利用多核 CPU（需用 cluster/worker_threads）
- 错误未捕获会导致整个进程崩溃

### CPU 密集型解决方案

**1. cluster 模块：**

```javascript
const cluster = require('cluster');
const os = require('os');

if (cluster.isMaster) {
  for (let i = 0; i < os.cpus().length; i++) {
    cluster.fork(); // 复制主进程，利用多核
  }
} else {
  // 子进程运行 HTTP 服务器
  http.createServer((req, res) => {
    res.end('Hello from ' + process.pid);
  }).listen(3000);
}
```

**2. worker_threads：**

```javascript
const { Worker } = require('worker_threads');

const worker = new Worker('./compute.js', {
  workerData: { input: 1000000 }
});
worker.on('message', result => console.log(result));
```

### CommonJS 模块机制

```javascript
// module.js
module.exports = { name: 'module' };
// 或
exports.name = 'module';

// index.js
const m = require('./module');
```

模块缓存：根据文件路径缓存，同一路径多次 require 返回同一对象。

## 16. 前端 Skill 怎么设计？

## 答案

### 什么是前端 Skill

Skill 是 AI Agent（如 Claude Code）的扩展能力，让 Agent 能调用外部工具完成任务。

### 核心组成

```
Skill = 描述（Description）+ 指令（Instructions）+ 工具（Tools）
```

### 设计模式

**1. MCP（Model Context Protocol）格式**

Anthropic 提出的标准 Skill 定义格式：

```json
{
  "name": "read_file",
  "description": "读取指定路径的文件内容",
  "input_schema": {
    "type": "object",
    "properties": {
      "file_path": {
        "type": "string",
        "description": "文件的绝对路径"
      }
    },
    "required": ["file_path"]
  },
  "output_schema": {
    "type": "object",
    "properties": {
      "content": { "type": "string" },
      "lines": { "type": "number" }
    }
  }
}
```

**2. Claude Code 的 Skill 结构**

```
.claude/
├── skills/
│   ├── skill-name/
│   │   ├── description.md     # Skill 的说明
│   │   ├── instructions.md   # 详细指令
│   │   └── tools/             # 可选：自定义工具脚本
│   └── settings.json          # Skill 注册配置
```

**3. 设计原则**

**a. 单一职责**：每个 Skill 只做一件事
```
✓ read_file：读取文件
✗ read_and_analyze_and_modify_file：职责过多
```

**b. 清晰的描述和边界**：
```markdown
# Read File Skill

## Description
读取指定路径的文件内容，返回文件文本。

## When to Use
- 用户要求查看某个文件
- 需要了解现有代码结构

## Limitations
- 不能读取二进制文件
- 文件超过 1MB 时返回错误
```

**c. 传递上下文**：
```markdown
## Input Format
用户会传递：
- file_path: string（文件路径）
- options?: { encoding?: string; maxLines?: number }

## Output Format
返回：
- content: string（文件内容）
- metadata: { lines: number; size: number }
```

### Skill 实现示例

```typescript
// .claude/skills/analyze-code/tools/analyze.ts
import { Tool } from '@anthropic-ai/claude-code';

export const analyzeCodeTool: Tool = {
  name: 'analyze_code',
  description: '分析代码文件的复杂度、依赖和问题',
  inputSchema: {
    type: 'object',
    properties: {
      file_path: { type: 'string' },
      depth: { type: 'string', enum: ['basic', 'full'], default: 'basic' }
    },
    required: ['file_path']
  },
  async execute({ file_path, depth }) {
    const content = await readFile(file_path);
    const result = analyze(content, depth);
    return formatResult(result);
  }
};
```

### Skill 编排

多个 Skill 可以组合成工作流：

```json
{
  "workflow": "code_review",
  "steps": [
```

## 17. MCP 格式是什么？

## 答案

### MCP（Model Context Protocol）

Anthropic 提出的**标准化协议**，用于 AI 模型与外部工具、数据源、文件系统交互。

### 核心概念

```
┌─────────────┐         ┌──────────────┐
│   Host      │  ←──→   │   MCP Server │
│ (Claude等)  │  MCP    │ (工具提供方)  │
└─────────────┘         └──────────────┘
```

- **Host**：AI 客户端（如 Claude Desktop、Claude Code）
- **MCP Server**：提供工具、数据源的服务程序

### MCP 四大核心能力

```
1. Tools（工具）：让 AI 调用外部功能
2. Resources（资源）：让 AI 读取数据/文件
3. Prompts（提示模板）：预定义的提示词
4. Sampling（采样）：AI 主动请求生成
```

### MCP 传输协议

MCP 服务通过 **stdio** 或 **HTTP/SSE** 与 Host 通信。

**stdio（本地）：**

```json
// MCP Server 输出（JSON-RPC 格式）
{ "jsonrpc": "2.0", "method": "tools/list", "id": 1 }

// Host 响应
{ "jsonrpc": "2.0", "result": { "tools": [...] }, "id": 1 }
```

### Tools 接口定义

```typescript
// MCP Tool 定义格式
interface Tool {
  name: string;           // 工具名（小写+下划线）
  description: string;    // 描述，AI 根据此决定何时调用
  inputSchema: {          // JSON Schema 输入参数
    type: 'object',
    properties: {
      file_path: { type: 'string' },
      // ...
    },
    required: ['file_path']
  };
}

// 调用请求
{
  "method": "tools/call",
  "params": {
    "name": "read_file",
    "arguments": { "file_path": "/path/to/file" }
  }
}

// 响应
{
  "method": "tools/call/result",
  "params": {
    "content": [
      { "type": "text", "text": "文件内容..." }
    ]
  }
}
```

### MCP Server 示例

```typescript
// 使用 @modelcontextprotocol/sdk
import { Server } from '@modelcontextprotocol/sdk/server';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio';
import { CallToolRequestSchema, ListToolsRequestSchema } from '@modelcontextprotocol/sdk/types';

const server = new Server(
  { name: 'filesystem-server', version: '1.0.0' },
  { capabilities: { tools: {} } }
);

server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [
    {
      name: 'read_file',
      description: '读取文件内容',
      inputSchema: {
        type: 'object',
        properties: {
          path: { type: 'string' }
        },
        required: ['path']
      }
    }
  ]
}));

server.setRequestHandler(CallToolRequestSchema, async (request) => {
  if (request.params.name === 'read_file') {
    const content = await readFile(request.params.arguments.path);
    return { content: [{ type: 'text', text: content }] };
  }
});

const transport = new StdioServerTransport();
await server.connect(transport);
```

### MCP vs OpenAI Function Calling

| | MCP | OpenAI Function Calling |
|--|-----|-------------------------|
| 生态 | 开放标准，多厂商支持 | OpenAI 专有 |
| 传输 | stdio/HTTP/SSE | OpenAI API |
| 范围 | 工具+资源+提示 | 仅工具调用 |
| 客户端 | Claude Desktop/Code | OpenAI 模型 |
| 可扩展性 | 任何工具可接入 | 需 OpenAI 集成 |

## 18. 简历平台 IndexedDB 的存储方式，存的什么数据？

## 答案

### IndexedDB 核心概念

- **数据库（Database）**：最大容器
- **对象仓库（Object Store）**：类似表
- **索引（Index）**：加速查询
- **事务（Transaction）**：保证原子性

### GResume 的存储架构

**1. 简历数据（主数据）**

```javascript
// 对象仓库：resumes
{
  id: 'uuid-xxx',           // 简历唯一 ID
  userId: 'user-123',       // 用户 ID
  name: '张三的简历',
  version: 12,              // 版本号
  createdAt: Date,
  updatedAt: Date,
  // CRDT 文档内容
  doc: {
    profile: { name: '张三', email: '...', phone: '...' },
    education: [...],
    experience: [...],
    skills: [...]
  }
}
```

**2. 版本历史**

```javascript
// 对象仓库：resume_versions
{
  id: 'version-xxx',
  resumeId: 'resume-xxx',   // 关联简历 ID
  version: 12,
  docSnapshot: { ... },     // 当时快照
  createdAt: Date,
  note: 'AI 优化前版本'      // 备注
}
```

**3. 离线操作队列**

```javascript
// 对象仓库：pending_operations
{
  id: 'op-xxx',
  resumeId: 'resume-xxx',
  operation: 'update',
  path: 'experience[0].company',
  value: '滴滴出行',
  timestamp: Date,
  synced: false             // 是否已同步到云端
}
```

**4. 用户信息**

```javascript
// 对象仓库：user
{
  id: 'user-xxx',
  email: 'zhangsan@example.com',
  lastSyncedAt: Date
}
```

### 使用的库

```typescript
import Dexie from 'dexie';

class GResumeDB extends Dexie {
  resumes!: Table<Resume>;
  resumeVersions!: Table<ResumeVersion>;
  pendingOps!: Table<PendingOp>;

  constructor() {
    super('GResumeDB');
    this.version(1).stores({
      resumes: 'id, userId, updatedAt',
      resumeVersions: 'id, resumeId, version',
      pendingOps: 'id, resumeId, synced, timestamp'
    });
  }
}
```

### 数据流

```
用户编辑简历
    ↓
写入 IndexedDB（本地优先，立即响应）
    ↓
写入 CRDT 文档内存状态（状态立即更新 UI）
    ↓
同时记录 pending_operations（离线队列）
    ↓
网络恢复后 → 从 pending_operations 取出 → 同步到 Supabase
    ↓
同步成功后 → 标记 synced: true
    ↓
定期清理已同步的操作记录
```

### IndexedDB 容量

- 存储配额：浏览器通常提供 50MB - 无限制（取决于磁盘空间）
- `navigator.storage.estimate()` 可查询当前使用量

### 为什么要用 IndexedDB 而不是 localStorage

| | IndexedDB | localStorage |
|--|-----------|--------------|
| 容量 |数百MB-数GB | ~5MB |
| 数据类型 | 结构化对象、Blob | 仅字符串 |
| 事务 | 支持原子操作 | 不支持 |
| 索引 | 支持多种索引查询 | 不支持 |
| 异步 | 全异步 API | 同步（阻塞主线程） |
| 适用场景 | 大数据离线存储 | 小数据配置 |

## 答案

### Supabase 服务端需要同构吗？

**不需要传统意义上的「同构渲染」（SSR）**，但 Supabase 本身充当后端服务。

### Supabase 架构

```
┌─────────────────────────────────────────────────┐
│                 前端（React/Vue）                  │
└────────────────────┬────────────────────────────┘
                     │ HTTPS
┌────────────────────▼────────────────────────────┐
│                  Supabase                       │
│  ┌─────────────┐  ┌──────────┐  ┌───────────┐ │
│  │ PostgreSQL  │  │  Auth    │  │ Edge Fn   │ │

│  └─────────────┘  └──────────┘  └───────────┘ │
└─────────────────────────────────────────────────┘
```

### 分布式支持

**Supabase 底层是 PostgreSQL，分布式能力有限：**

| 维度 | 支持情况 |
|------|----------|
| **水平扩展（读写分离）** | 需额外配置（Supabase 使用 PostgresBouncer 连接池） |
| **分库分表** | 不支持原生（需应用层或 pg 插件） |
| **多区域部署** | Supabase Pro 支持多区域实例 |
| **高可用** | 自带 HA（PostgreSQL 流复制 + 备份） |

**Supabase 的分布式局限**：
- PostgreSQL 原生是单机数据库
- 多租户场景需在应用层做数据隔离
- 超大规模数据（TB 级）需考虑专用 PostgreSQL 集群（如 AWS RDS Aurora）

## 20. PostgreSQL 和 MySQL 区别

## 答案

### 核心区别

| 维度 | PostgreSQL | MySQL |
|------|-----------|-------|
| **类型** | 对象关系型（ORDBMS） | 纯关系型（RDBMS） |
| **诞生年份** | 1996 | 1995 |
| **开发公司** | PostgreSQL Global Development Group | Oracle（收购Sun） |
| **开源协议** | PostgreSQL License（MIT-like） | GPL / 商业版 |
| **标准遵循** | SQL 标准支持更好 | 部分标准扩展 |

### 功能对比

**1. 数据类型**

| | PostgreSQL | MySQL |
|--|-----------|-------|
| 数组 | ✅ 原生支持 `TEXT[]` | ❌ 需 JSON 列模拟 |
| JSON/JSONB | ✅ 原生支持，JSONB 索引 | ✅ JSON 支持（5.7+），但不如 PG |
| 范围类型 | ✅ `DATERANGE`、`INT4RANGE` | ❌ 不支持 |
| IP 地址 | ✅ `INET`、`CIDR` | ❌ 需 VARCHAR |
| 几何类型 | ✅ PostGIS（强大） | ✅ 基础几何类型 |
| 自定义类型 | ✅ `CREATE TYPE` | ❌ 不支持 |

**2. 事务与隔离级别**

| | PostgreSQL | MySQL |
|--|-----------|-------|
| ACID | 完整支持 | 完整支持（InnoDB） |
| 隔离级别 | 4 级（READ COMMITTED 默认） | 4 级（REPEATABLE READ 默认） |
| MVCC | ✅（并发读写互不阻塞） | ✅（InnoDB MVCC） |
| 嵌套事务 | ✅（SAVEPOINT） | ✅（SAVEPOINT） |

**3. 性能与优化**

| | PostgreSQL | MySQL |
|--|-----------|-------|
| 复杂查询 | 更强（优化器更先进） | 一般 |
| 写入性能 | 稍弱（MVCC 开销） | 更好（InnoDB 优化） |
| 全文本搜索 | ✅ 原生 | ✅ MyISAM 时期已有，InnoDB 5.6+ 支持 |
| 分区 | 支持（不如 MySQL 灵活） | 多种分区策略 |
| 并行执行 | ✅（并行查询） | ✅（MySQL 8.0+） |

**4. 索引类型**

| 索引 | PostgreSQL | MySQL |
|------|-----------|-------|
| B-Tree | ✅ | ✅（默认） |
| Hash | ✅ | ✅（Memory 引擎） |
| GiST/GIN | ✅（全文、JSON、范围） | ❌ |
| 表达式索引 | ✅ | ❌ |
| 部分索引 | ✅ | ❌ |
| 覆盖索引 | ✅ | ✅ |

**5. 复制与集群**

| | PostgreSQL | MySQL |
|--|-----------|-------|
| 主从复制 | ✅ 流复制 | ✅ binlog 复制 |
| 同步复制 | ✅（逻辑复制） | ✅（Group Replication） |
| 分布式 | 需插件（ Citus） | MySQL Fabric |
| 分片 | 需应用层/Citus | 应用层/ShardingSphere |

**6. 扩展性**

PostgreSQL：
```sql
-- 扩展生态
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS "postgis";
CREATE EXTENSION IF NOT EXISTS "pg_trgm";  -- 模糊搜索
```

MySQL：
```sql
-- 插件生态
INSTALL PLUGIN soname 'ha_rocksdb.so';
```

### 选择建议

| 场景 | 推荐 |
|------|------|
| 复杂查询、OLAP、数据分析 | **PostgreSQL** |
| 高并发写入、简单查询 | **MySQL** |
| JSON 数据多、需要全文搜索 | **PostgreSQL** |
| 需要 GIS 地理信息 | **PostgreSQL + PostGIS** |
| 需要 MyISAM 特性（全文搜索）| **MySQL** |
| 成熟生态、Oracle 兼容 | **MySQL** |
| 严格 SQL 标准遵循 | **PostgreSQL** |
| AI/向量数据 | **PostgreSQL + pgvector** |

### GResume 项目选择 PostgreSQL 的原因

- JSONB 存储 CRDT 文档结构
- 全栈 JS/TS（Supabase + React）生态好
- 实时订阅（Realtime）基于 PostgreSQL 逻辑复制
- RLS（行级安全）策略强大
