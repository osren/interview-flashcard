## 1. 在滴滴商旅平台业务中遇到过最复杂的业务场景是什么？是怎么解决的？

## 答案

**最复杂的业务场景：多供应商机票政策动态配置系统**

背景：
- 商旅平台对接了国航、东航、南航等多家航空公司
- 每家航空公司的退票、改签、特殊价政策规则各不相同
- 政策配置需要支持动态生效，且不能影响线上现有订单

**解决方案：**

1. **业务抽象层**
   - 将所有航空公司政策抽象为统一的数据模型
   - 定义政策规则的"条件-动作"映射结构

2. **动态配置引擎**
   - 使用 JSON Schema 定义政策配置规范
   - 实现配置版本管理，支持灰度生效和快速回滚

3. **数据隔离**
   - 不同供应商的数据完全隔离
   - 配置变更不影响已有订单的政策结算

**技术难点突破：**

| 难点 | 解决方案 |
|------|----------|
| 规则复杂度高 | 引入规则引擎（如 JSON Schema + 自定义校验） |
| 性能要求高 | 配置预编译 + 缓存策略 |
| 实时性要求 | 消息队列通知 + 热更新 |

## 3. GResume项目中离线功能是如何保证数据一致性的？

## 答案

### 离线数据一致性保证机制

**1. 本地优先策略（Local-First）**

```
用户编辑 → 本地 CRDT 变更 → 写入 IndexedDB
                     ↓
           同时写入内存（状态立即更新）
                     ↓
           网络恢复后自动同步到云端
```

**2. CRDT 保证最终一致性**

- 每个操作都是"可交换"的
- 离线期间的编辑不会丢失
- 冲突自动合并，无需人工介入

**3. 版本管理机制**

```
本地版本
   ↓ (自动合并)
云端版本 (Supabase)
   ↓
版本冲突 → 自动 LWW (Last-Writer-Wins)
   ↓
版本历史保留 → 可回滚
```

**4. 同步状态可视化**

| 状态 | 含义 | 用户反馈 |
|------|------|----------|
| Synced | 已同步 | 绿色勾 |
| Syncing | 同步中 | 蓝色旋转 |
| Offline | 离线中 | 灰色提示 |
| Conflict | 有冲突 | 红色标记 |

**5. 断点续传**

- 使用增量同步（Delta Sync）
- 网络中断后从上次同步点继续
- 不需要重新传输整个文档

## 6. 在简历项目中遇到的最大技术挑战是什么？如何突破的？

## 答案

### 最大技术挑战：GResume 多人实时协作的光标同步

**挑战背景：**

- 多个用户同时编辑同一份简历
- 需要实时显示每个人的光标位置
- 网络延迟、离线场景下的用户体验

**解决思路：**

```
问题拆解：
1. 光标位置如何表征？（相对位置 vs 绝对位置）
2. 如何处理光标冲突？（最后写入优先 vs 协商）
3. 如何在低带宽下保持实时性？（节流 + 压缩）
```

**具体解决方案：**

**1. 相对位置表示**
```typescript
// 绝对位置："第 100 个字符"
// 相对位置："contact-info 模块之后，第 3 个段落"

const cursorPosition = {
  moduleId: 'contact-info',  // 所属模块
  blockIndex: 2,            // 块内索引
  offset: 15               // 字符偏移
}
```

**2. 节流与批量**
```typescript
// 不节流：每个字符变动都发消息 → 网络阻塞
// 节流后：每 50ms 最多发送一次 → 节省 90% 带宽

const throttledCursor = throttle((pos) => {
  channel.send('cursor-update', pos)
}, 50)
```

**3. 离线优雅降级**
```typescript
if (!isOnline) {
  // 离线时：不显示其他用户光标
  // 本地操作正常进行
  // 网络恢复后：重新同步状态
}
```

**突破结果：**

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 光标同步延迟 | 500ms+ | < 100ms |
| 带宽占用 | 100KB/分钟 | 10KB/分钟 |
| 离线可用性 | 不可用 | 完全支持 |

## 14. GResume项目中遇到过的最难解决的Bug是什么？怎么解决的？

## 答案

### 最难解决的 Bug：CRDT 同步导致的内存泄漏

**问题描述：**

- 用户长时间编辑简历（2+ 小时）后，页面变得卡顿
- Chrome DevTools 显示内存持续增长
- 刷新页面后内存释放，但再次编辑又会出现

**排查过程：**

```
1. 初步判断：React 组件未正确卸载
   - 检查 useEffect cleanup
   - 结果：未发现问题

2. 怀疑：Automerge 文档对象持续增长
   - 检查 doc 对象大小
   - 发现：doc 的 changeHistory 无限增长

3. 深入分析：
   - Automerge 的每个编辑都会记录 change
   - 旧 change 未被清理，导致内存膨胀

4. 根因确认：
   - Automerge 默认保留所有 change 历史
   - 长时间编辑 = 大量 change 累积
```

**解决方案：**

```typescript
// 1. 定期清理历史（保留最近 100 个 change）
function pruneHistory(doc: Doc, maxHistory = 100) {
  const history = doc.getChangeHistory()
  if (history.length > maxHistory) {
    // 压缩旧 history
    const pruned = history.slice(-maxHistory)
    doc.setChangeHistory(pruned)
  }
}

// 2. 定时触发清理
useEffect(() => {
  const interval = setInterval(() => {
    pruneHistory(docRef.current)
  }, 5 * 60 * 1000) // 每 5 分钟

  return () => clearInterval(interval)
}, [])

// 3. 页面切换时强制 GC
useEffect(() => {
  return () => {
    // 卸载时清理
    docRef.current = null
  }
}, [])
```

**修复效果：**

| 指标 | 修复前 | 修复后 |
|------|--------|--------|
| 2 小时后内存 | 800MB+ | 150MB |
| 页面卡顿 | 明显 | 无感知 |
| 内存增长 | 线性增长 | 稳定 |

## 17. 讲讲你对TypeScript的理解？在项目中是如何落地的？

**本质：JavaScript 的超集，添加了类型系统**

```
const name = 'Tom'

```

**核心价值：**

| 价值 | 说明 |
|------|------|
| 类型安全 | 编译期发现错误，减少线上 Bug |
| 代码提示 | IDE 智能提示，提升开发效率 |
| 可读性 | 类型即文档，理解代码意图 |
| 重构安全 | 重构时自动检查影响范围 |

### 项目落地实践

**1. 严格模式配置**

```json

{

    "noImplicitAny": true,
    "strictNullChecks": true,

  }
}
```

**2. 类型定义规范**

## 20. 你对微前端有什么了解？GResume项目是否适合使用微前端？

## 答案

### 微前端理解

**定义：将大型应用拆分为独立部署的小应用**

```
传统单体：
┌─────────────────────────────────┐
│         应用 A (100人维护)      │
│   页面1 + 页面2 + 页面3 + ...   │
└─────────────────────────────────┘

微前端：
┌─────────────────────────────────┐
│         容器应用 (主应用)        │
│  ┌─────┐ ┌─────┐ ┌─────┐       │
│  │子应用│ │子应用│ │子应用│  ... │
│  │ A   │ │ B   │ │ C   │       │
│  └─────┘ └─────┘ └─────┘       │
└─────────────────────────────────┘
```

**适用场景：**

| 场景 | 适合度 |
|------|--------|
| 多团队维护同一个大应用 | ✅ 非常适合 |
| 多个独立系统需要统一入口 | ✅ 适合 |
| 新旧技术栈共存 | ✅ 适合 |
| 单个小团队维护简单应用 | ❌ 不适合 |

**主流方案对比：**

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| qiankun | 路由注册 | 蚂蚁背书、文档完善 | 依赖 single-spa |
| Garfish | 路由注册 | 阿里更替方案 | 相对新 |
| iframe | 嵌套页面 | 完全隔离 | 体验差、通信难 |
| Web Component | 自定义元素 | 标准规范 | 生态弱 |

### GResume 是否适合微前端？

**结论：不适合**

**原因分析：**

```
1. 项目规模小
   - GResume 目前是单个应用
   - 代码量 < 10 万行
   - 一个团队维护足够

2. 不需要独立部署
   - 目前的迭代节奏不需要单独部署
   - Vercel 自动部署足够

3. 技术复杂度增加
   - 微前端带来：通信、样式、依赖管理复杂性
   - 对于小项目是过度设计

4. 用户体验
   - 微前端会带来加载延迟
   - GResume 强调"快"
```

**未来扩展场景（可能适合）：**

```
如果 GResume 发展为：
- 多团队分别维护：简历模板、AI 功能、协作功能
- 需要独立发布：某个功能可以单独迭代
- 技术栈演进：新功能用新框架，老功能保持现状

→ ，那时候可以引入微前端
```

## 26. 在项目中遇到接口轮询导致的性能问题，该如何解决？

## 答案

### 问题分析

接口轮询常见场景：
- 实时数据更新（订单状态、消息通知、股票行情）
- 长连接不可用时的降级方案
- 轮询检测状态变化

**常见问题：**
- 轮询频率过高 → 服务器压力大、客户端卡顿
- 多个轮询同时存在 → 并发请求过多
- 网络波动 → 轮询断开后持续重试
- 页面隐藏时仍轮询 → 无效资源消耗

### 解决方案

**1. 智能轮询策略**

```typescript
// 指数退避轮询
class SmartPolling {
  private baseInterval = 1000  // 基础间隔 1s
  private maxInterval = 30000 // 最大间隔 30s
  private currentInterval = this.baseInterval
  private errorCount = 0

  async fetch() {
    try {
      const data = await api.getData()
      this.onSuccess()
      return data
    } catch (error) {
      this.onError()
      throw error
    }
  }

  private onSuccess() {
    // 成功后逐渐恢复正常间隔
    this.currentInterval = this.baseInterval
    this.errorCount = 0
  }

  private onError() {
    this.errorCount++
    // 指数退避：1s → 2s → 4s → 8s → 16s → 30s
    this.currentInterval = Math.min(
      this.baseInterval * Math.pow(2, this.errorCount),
      this.maxInterval
    )
  }
}
```

**2. 请求合并与防抖**

```typescript
// 合并短时间内多次轮询请求
const pollingRequest = debounce(async (key: string) => {
  const cached = cache.get(key)
  if (cached && Date.now() - cached.timestamp < 1000) {
    return cached.data
  }
  const data = await fetch(key)
  cache.set(key, { data, timestamp: Date.now() })
  return data
}, 100)
```

**3. 页面可见性控制**

```typescript
useEffect(() => {
  let pollingId: number

  const startPolling = () => {
    pollingId = window.setInterval(fetchData, interval)
  }

  const stopPolling = () => {
    if (pollingId) clearInterval(pollingId)
  }

  // 页面可见时启动，隐藏时停止
  const handleVisibilityChange = () => {
    if (document.hidden) {
      stopPolling()
    } else {
      // 立即执行一次，避免等待
      fetchData()
      startPolling()
    }
  }

  document.addEventListener('visibilitychange', handleVisibilityChange)
  startPolling()

  return () => {
    stopPolling()
    document.removeEventListener('visibilitychange', handleVisibilityChange)
  }
}, [])
```

**4. 资源优先级控制**

```typescript
// 使用 AbortController 管理多个轮询
class PollingManager {
  private controllers = new Map<string, AbortController>()

  start(key: string, fetchFn: () => Promise<any>, interval: number) {
    // 先停止已有的
    this.stop(key)

    const controller = new AbortController()
    this.controllers.set(key, controller)

    const loop = async () => {
      if (controller.signal.aborted) return
      try {
        await fetchFn()
      } catch (e) {
        if (!controller.signal.aborted) {
          // 处理错误，可能降级或停止
        }
      }
      if (!controller.signal.aborted) {
        setTimeout(loop, interval)
      }
    }
    loop()
  }

  stop(key: string) {
    const controller = this.controllers.get(key)
    if (controller) {
      controller.abort()
      this.controllers.delete(key)
    }
  }

  stopAll() {
    this.controllers.forEach(c => c.abort())
    this.controllers.clear()
  }
}
```

**5. 最佳实践总结**

| 场景 | 方案 | 收益 |
|------|------|------|
| 高频轮询 | 指数退避 | 减少服务端压力 |
| 多标签页 | 单例模式 | 避免重复请求 |
| 页面不可见 | visibilitychange | 节省资源 |
| 网络波动 | 请求合并 | 减少并发 |
| 后台运行 | requestIdleCallback | 不阻塞主线程 |

|------|------|----------|

| WebSocket | 实时 | 高实时性、频繁交互 |
| SSE | 实时 | 单向数据流 |
