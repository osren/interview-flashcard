## 一、问题理解与方案构想

### 1.1 真实业务背景：AI 已落地，但 Token 成本成为规模化瓶颈

在滴滴企业版商旅业务中，机票、酒店、火车票线上接口错误告警量大、噪音占比高。典型噪音包括：业务正常拒绝（无票、已退改、重复提交）、低价值展示类错误（NPS、公告、分享页配置）、信息不全的原始告警，以及三方/非自有接口异常。其直接后果是：**真故障被淹没，值班被刷屏，漏看风险上升**。

为应对这一痛点，我们已在商旅场景落地 **AI 监控降噪系统（error-triage / ai-monitor-deposer）**：在告警到达值班之前，由大模型对每条接口错误做智能判定——是否值得上报（`should_report`）、严重等级（P0~P3）、错误类型与中文依据。系统遵循「宁可误报，不可误杀」原则，判定质量优先，因此**每条告警仍需走一次 LLM**。

然而，随着机票、酒店 domain 陆续接入，以及未来向出行侧客服判责、需求理解等场景扩展，**Token 消耗正从「可接受试点成本」演变为「规模化部署瓶颈」**。赛题所指的「不必要 Token 支出」，在我们的真实链路中已有清晰对应——问题不在「任务太难」，而在「调用方式低效」。

### 1.2 问题本质：六类 Token 浪费根因

结合 error-triage 现有架构与待接入场景，我们归纳六类浪费：

| 浪费类型 | 在商旅告警场景的具体表现 | 严重度 |
|---------|------------------------|--------|
| **上下文过长** | 告警 payload 携带完整 response 大段 JSON、无关 trace 字段 | 高 |
| **Prompt 冗余** | 10 个 category 的 prompt 规则被全量拼入 system，而非按路由 key 按需加载 | 高 |
| **RAG 不精准** | 接入阶段 discover 全量扫描项目，token ∝ 项目大小 | 中 |
| **模型路由不当** | 简单归属判别（自有/三方）本可走规则，却与大模型判定混在同一条链路 | 中 |
| **输出不受控** | reason 字段偶发超长 markdown，增加 output token 与解析失败率 | 中 |
| **失败重试** | JSON 解析失败时反复原样重调，而非 fallback 保守上报 | 低~中 |

**核心洞察**：我们的业务不允许「少调模型」来降本（漏报 ≈ 0），因此降本主战场是 **「单次调用瘦身 + 工程链路零 LLM 化 + 结果复用」**，而非简单粗暴减少调用次数。

### 1.3 方案构想：分层 Token 治理框架（Tiered Token Governance）

我们提出一套可跨场景迁移的六层治理框架，将 Token 优化从「靠感觉砍 Prompt」升级为「可度量、可验证、可复用的 AI 工程能力」：

```
告警/请求入站
    → ① 路由层（Router First）      — 零 LLM 完成分类与归属
    → ② 上下文层（Context Trim）    — 字段裁剪、增量处理
    → ③ Prompt 层（Dynamic Assembly）— 按场景模块化拼装
    → ④ 模型层（Model Tiering）    — 复杂度分级选模型
    → ⑤ 输出层（Output Constraint）— JSON Schema + 长度约束
    → ⑥ 缓存层（Result Reuse）     — 指纹/语义缓存复用判定
    → LLM 调用（仅在必要时、用最小上下文）
```

**三大创新点：**

**创新点 1：Router First — 先分类，再决定「用什么 Prompt、调什么模型」**

借鉴 error-triage 已有 `router.js` 设计：pathname 归一 → 自有前缀归属判别 → 有序规则匹配 → 得到 prompt key。非自有前缀（营销中台、外部域名）直接路由至 `non-core`，无需加载 8 个核心业务 category 的 prompt。这是 **零 Token 的路由决策**，已在火车票 domain 验证。

**创新点 2：Dynamic Prompt Assembly — 按需拼装，不全量加载**

system prompt = `_base.md`（全局原则，约 200 token）+ `<key>.md`（当前 category 规则，约 100~300 token）。改造前若全量加载 10 个 category，system 可达 3000+ token；改造后单次仅 800 token 量级。**调阈值 = 改 prompt，但 prompt 本身要精简**——这与现有工程纪律一致。

**创新点 3：Eval 驱动降本 — 降本不能牺牲「宁可误报不可误杀」**

任何 Token 优化必须过 gold 数据集对比（`/eval` 批量评测）。指标门禁：input token ↓ ≥ 40%，准确率/F1 下降 ≤ 1%，**漏报率不升**。没有 baseline 就砍 prompt，在告警场景等于制造生产事故。

### 1.4 与滴滴更大 AI 版图的关系

本框架虽以商旅告警为首个落地场景，但六层结构可直接复用到 Hackathon 其他赛题：

- **赛题 01 需求理解**：路由层识别场景（机场/医院/宠物），Prompt 层按场景加载标签匹配规则
- **赛题 02 路线信任**：上下文层只注入当前路段关键信息，输出层约束解释长度
- **赛题 03 纠纷判责**：模型层对简单节点抽取用小模型，复杂推理才上大模型

**一套框架，多个 AI 场景复用**——这是本方案相对「单点 Prompt 技巧」的核心竞争力。

## 二、实现路径与 Demo 规划

### 2.1 现状与改造基线

**已有资产（可直接 Demo）：**

| 资产 | 状态 | Token 相关价值 |
|-----|------|--------------|
| error-triage 通用引擎 | ✅ HTTP/Kafka 双入口就绪 | 路由 + 动态 prompt 骨架已有 |
| 火车票 domain | ✅ 已接入，business_reject 已回源核验 | 可跑真实告警 replay |
| 机票/酒店 url 清单 | ✅ 已摘取，domain 待 onboard | 可演示多 domain 复用 |
| `/judge` `/eval` API | ✅ 就绪 | 可量化 before/after |
| discover-triage-urls sync 模式 | ✅ 支持增量 | 接入阶段 token ∝ delta |
| drift-check.sh | ✅ 零 LLM | 维护阶段零 token |

**待建设（Hackathon MVP 范围）：**

- 告警字段裁剪 middleware（P0）
- 全链路 Token 埋点与成本看板（P0）
- 指纹缓存层（P1）
- 模型分级路由（P1，需 eval 验证）

### 2.2 四阶段实现路径

#### 阶段一：Baseline 与埋点（第 1 周）

**目标**：建立可对比的 Token 基线，拒绝「拍脑袋降本」。

| 任务 | 交付物 | 验证标准 |
|-----|--------|---------|
| 接入 Token 埋点 | 每次 LLM 调用记录 input/output token、路由 key、耗时 | 100% 采样 |
| 攒 gold 数据集 | 火车票 domain 50+ 条标注（should_report + severity） | 覆盖 8 个 category |
| 跑 baseline | `/eval` 输出 v0 报告：token P50/P95、准确率、F1 | 报告可复现 |

#### 阶段二：P0 改造 — 路由 + 上下文 + Prompt（第 2~3 周）

| 改造项 | 做法 | 预期节省 |
|-------|------|---------|
| 字段裁剪 | user message 只传 url、errorCode、errorMsg、count、traceid | input ↓ 40~60% |
| 动态 Prompt 强制化 | 引擎层禁止全量加载，按 router key 拼 `_base + key` | system ↓ 60~70% |
| 输出约束强化 | reason ≤ 80 字，temperature=0，解析失败 fallback P2 | output ↓ 20%，重试 ↓ |
| discover sync 推广 | 机票/酒店 onboard 走增量模式 | 接入 token ↓ 80%+ |

#### 阶段三：P1 改造 — 缓存 + 模型分级（第 4 周）

| 改造项 | 做法 | 风险管控 |
|-------|------|---------|
| 指纹缓存 | hash(url+errorCode+errorMsg) 短 TTL 5~15min | 仅缓存 should_report=false 的噪音判定 |
| 模型分级 | 规则可覆盖的归属判别零 LLM；判定走统一网关，未来按 complexity 分流 | eval 对比后灰度 |
| 语义缓存 | 相似 FAQ 式告警 embedding 命中 | 二期 |

#### 阶段四：灰度验证与推广（第 4 周末 ~ 第 5 周）

- 火车票 domain 10% 流量灰度 → 50% → 全量
- 机票 domain 接入并复用同一引擎
- 输出《Token 治理手册》供其他团队自助接入

### 2.3 Demo 规划（Hackathon 现场可演示）

**Demo 主题**：「同一条告警，Token 从 4700 → 1350，判定结论一致」

**Demo 脚本（约 5 分钟）：**

1. **场景引入（30s）**  
   展示一条真实火车票告警：「transferLeftTicket 参数错误，count=12」。说明当前商旅值班面临的噪音困境。

2. **Before 回放（1min）**  
   调用 `/judge`，展示改造前 payload：全量 response、全量 system prompt。屏幕显示：**input 4500 token，output 200 token，合计 4700**。

3. **六层改造讲解（1.5min）**  
   动画/架构图展示：router 命中 `query` → 只加载 `_base + query.md` → user 只含 5 个字段 → JSON 输出约束。

4. **After 回放（1min）**  
   同一条告警再调 `/judge`：**input 1200 token，output 150 token，合计 1350**。对比 should_report、severity、reason——**结论一致**。

5. **Eval 报告（1min）**  
   展示 `/eval` 批量 50 条对比：token ↓ 71%，准确率 96%→95.5%（下降 0.5%，门禁内），**漏报 0 条**。

6. **迁移展望（30s）**  
   说明同一框架如何复用到出行侧客服/判责，强调「不是 Prompt 技巧，是 AI 工程能力」。

**Demo 技术栈**：error-triage 引擎 + 火车票 domain + `/judge`/`/eval` API + 简单成本看板页面（Token 对比柱状图）。

## 三、预期价值与业务收益

### 3.1 量化收益预估（商旅告警场景）

以 **日告警 10 万次** 为估算基准（机票+酒店+火车接入后量级）：

| 指标 | 改造前 | 改造后 | 变化 |
|-----|--------|--------|------|
| 单次 input token | ~4500 | ~1200 | ↓ 73% |
| 单次 output token | ~200 | ~150 | ↓ 25% |
| 单次合计 token | ~4700 | ~1350 | **↓ 71%** |
| 日总 token | 4.7 亿 | 1.35 亿 | 节省 3.35 亿/天 |
| 日 LLM 成本（按 ¥0.07/1K token 粗算） | ¥32,900 | ¥9,450 | **节省 ¥23,450/天** |
| 月节省 | — | — | **约 ¥70 万/月** |

> 注：单价为估算，实际以公司大模型网关账单为准；Hackathon 后会用真实埋点数据替换。

### 3.2 业务价值

| 维度 | 价值描述 |
|-----|---------|
| **成本** | 直接降低 LLM 调用费用，使告警降噪从「试点可承受」变为「全业务线可规模化部署」 |
| **性能** | 上下文更小 → P95 延迟下降，告警判定链路更快，值班体验更好 |
| **稳定性** | Eval 门禁确保降本不牺牲「宁可误报不可误杀」，漏报率保持 ≈ 0 |
| **研发效率** | domain 可插拔 + Skill 工具链（discover/onboard），新业务接入从「周」缩短到「天」 |
| **组织复用** | 六层框架可迁移至出行侧需求理解、路线解释、纠纷判责等 AI 场景，避免各团队重复造轮子 |

### 3.3 战略价值：从「堆大模型」到「精细化 AI 工程」

当前不少 AI 应用的默认路径是：任务复杂了 → 换更大模型 → 塞更多上下文。这条路径在 POC 阶段可行，但在滴滴这种**高频、大规模、多场景**的 AI 落地环境中不可持续。

本方案推动组织建立三项能力：

1. **Token 可观测**：每次调用有埋点，成本可归因到 domain/category/prompt 版本
2. **Eval 可门禁**：任何优化必须过 gold 对比，不靠线上试错
3. **框架可复用**：引擎领域无关，换 domain 不改引擎，降低 AI 应用边际成本

这与滴滴产品技术体系「让 AI 真正服务业务」的方向一致——**不是让模型少干活，而是让模型少干无效活**。

### 3.4 风险与对策

| 风险 | 对策 |
|-----|------|
| 砍太多导致准确率/漏报恶化 | Eval 硬门禁；漏报 > 0 则回滚 |
| 缓存导致误判复用 | 仅缓存噪音判定（should_report=false）；真故障不缓存 |
| 各业务 prompt 差异大 | domain 可插拔，`_base + key` 模块化 |
| 模型分级误路由 | 保守策略：不确定时走大模型 + 全量 prompt |
