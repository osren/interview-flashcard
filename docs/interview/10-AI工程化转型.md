# AI 工程化转型与实践

## 一、问题类型

| 问题类型 | 具体表现 | 占比 |
|---------|---------|-----|
| 代码一致性 | 57个页面组件代码风格不统一，命名规范差异大 | 30% |
| 知识传承 | 业务逻辑依赖口口相传，新人上手周期长（2-3周） | 25% |
| 重复基建 | 每个页面重复实现列表、表格、分页、筛选等基础逻辑 | 20% |
| 维护效率 | 依赖升级风险高，升级可能导致功能异常（如 dajax 版本问题） | 15% |
| 自动化程度 | 大量重复劳动（CRUD 模板代码），缺乏智能辅助 | 10% |

## 二、解决方案

### 1. 代码一致性治理

- **规范文档**：建立 CLAUDE.md、code-style.md 等开发规范，约束 AI 和开发者的行为
- **ESLint 规则**：配置 airbnb 规则，部分规则定制化以适配项目（如禁用 camelcase、prop-types 等）
- **Prettier 格式化**：VS Code 保存时自动格式化，统一代码风格
- **Code Review**：合并前人工/自动化审查，确保规范落地

### 2. 知识传承体系

- **文档结构化**：每个业务模块目录下的 README 说明业务背景和关键逻辑
- **注释规范**：关键业务逻辑要求注释说明"为什么这样做"
- **类型系统**：虽然项目未使用 TypeScript，但通过 PropTypes 或注释明确接口

### 3. 消除重复基建

- **公共组件库**：
  - `src/components/UploadModal.jsx` - 统一文件上传
  - `src/components/LogModal.jsx` - 统一操作日志
  - `src/components/SelectBank.jsx` - 统一银行选择
- **统一请求层**：
  - `src/utils/request.js` 封装所有 API 调用，统一处理错误、loading、签名
- **工具函数库**：
  - `src/utils/util.js` 封装日期格式化、参数处理等通用逻辑

### 4. 依赖管理

- **版本锁定**：关键依赖锁定版本，升级需评估风险
- **变更日志**：记录依赖升级原因、影响范围、解决方案
- **CI 自动化**：自动化测试覆盖，降低升级风险

### 5. AI 辅助实践

- **渐进式引入**：从简单任务开始（生成 CRUD 模板、代码审查、文档生成）
- **场景化提示**：针对本项目特点设计提示词（如组件命名规范、目录结构要求）
- **反馈机制**：AI 生成代码需人工审核，逐步优化生成质量
- **知识库建设**：积累常见业务场景的 AI 解决方案，形成最佳实践

## 三、转型收益

| 维度 | 转型前 | 转型后 | 提升 |
|-----|-------|-------|-----|
| 新人上手 | 2-3周 | 1周 | 50%+ |
| CRUD 效率 | 2天/页面 | 0.5天/页面 | 75%+ |
| 代码规范率 | 60% | 90% | 50% |
| 知识传承 | 口头/文档 | 规范+AI 辅助 | 系统化 |

---

*注：以上内容基于甲子（Jiazi）商旅运营管理平台的实际架构总结，涵盖 57 个业务页面、40+ 个 service 文件、React 17 + Redux + Ant Design + Vite 技术栈。*
