# 高内聚业务组件体系设计

## 一、设计思路

高内聚业务组件体系的核心是将相关功能组织在一起，让模块内部元素彼此紧密关联，而模块间保持低耦合。在本项目中的设计实践包括：

### 1. 分层目录结构设计

- `src/pages/` - 按业务模块划分页面，如 `travel_config`、`plane_policy`、`hotel_supplier_price`
- `src/pages/{模块}/page/` - 业务子模块，如 `travel_config/page/BonusRule`
- `src/components/` - 公共组件层，如 `UploadModal`、`LogModal`、`SelectBank`
- `src/services/` - API 服务层，按业务领域划分，每个 service 文件对应一个业务域

### 2. 组件分层策略

- **基础组件**：Ant Design 原子组件（如 Button、Table、Form）
- **业务组件**：封装特定业务逻辑的组件（如本项目的 `SelectBank`、`LogModal`）
- **页面组件**：完整的业务页面，整合业务组件和服务

### 3. 路由级别的分割

- 使用 `@loadable/component` 实现按需加载，每个路由对应独立的业务模块
- Routes.jsx 中配置了57个页面的懒加载，实现首屏性能优化

## 二、评判标准

1. **单一职责原则**：每个组件、服务只负责一个明确的业务功能，不做职责之外的事
2. **内聚性度量**：相关功能在同一目录下组织，模块内引用多于跨模块引用
3. **可维护性**：新增需求时能快速定位到对应模块，修改影响范围可控
4. **可测试性**：业务逻辑可通过 service 层独立单元测试，不依赖 UI
5. **依赖清晰度**：业务组件间通过 props 或服务层通信，避免循环依赖
6. **接口稳定性**：对外暴露的接口（props、API）保持向后兼容
