## 一、协同编辑相关

### 1. 协同 OT 算法？

**本项目使用的是 CRDT，不是 OT（Operational Transform）。**

具体实现为 **Automerge**（`@automerge/automerge-repo`），这是一套基于 CRDT 的协作引擎。相比 OT，CRDT 的优势在于：

- **无冲突合并**：基于数学上可证明的合并性质，不需要中心服务器来转换操作
- **离线友好**：离线修改可以自动合并，不依赖在线顺序
- **去中心化**：不需要锁或协调者

> 源码：`src/lib/automerge/` 目录下完整实现了 Automerge 文档的创建、同步和持久化。

---

### 2. CRDT 的实现原理？

Automerge CRDT 的核心原理：

```
每次修改 → 生成 Change（含操作 + 因果顺序）→ 广播给所有节点
                                              ↓
各节点独立合并：按因果顺序应用所有 Change → 自动解决冲突
```

**三个关键机制：**

1. **因果顺序（Partial Order）**：每个操作带 `context`，记录"发生在哪些操作之后"
2. **无冲突复制数据类型**：每种数据结构（list、map、text）都有数学证明的 CRDT 算法
3. **自动合并**：如果两人同时修改同一字段，最后写入胜出；如果修改不同字段，则全部保留

**项目中的实现：**

```typescript
// 文档结构（src/lib/automerge/document/schema.ts）

  _metadata: {
    resumeId: string
    userId: string

    updatedAt: string

  }
}
```

每次 `handle.change()` 时调用 `touchDocumentMetadata()` 递增 version：

```typescript

  metadata.updatedAt = new Date().toISOString()
  metadata.version = typeof metadata.version === 'number' ? metadata.version + 1 : 1
```

### 3. 两三个人实时在线编辑是怎么实现的？

**通过 Supabase Realtime 广播频道实现 P2P 式的多用户同步。**

架构如下：

```
用户 A ──┐
用户 B ──┼── Supabase Realtime Channel ──→ 所有人收到消息
用户 C ──┘         │
                   └── broadcast（广播）+ presence（在线状态）
```

**源码实现（`src/lib/automerge/collaboration/supabase-network-adapter.ts`）：**

```typescript
// 创建 Supabase Realtime 频道
this.channelName = `automerge:resume:${resumeId}:${sessionId}`
this.channel = supabase.channel(this.channelName)

// 注册 CRDT 消息广播
this.registerSyncBroadcast()

// 注册用户在线状态（presence）
this.registerPresenceEvents()

// 订阅频道
this.channel.subscribe(async (status) => {
  if (status !== 'SUBSCRIBED') return
  await this.channel?.track({
    peerId: String(this.peerId),
    metadata: { ...this.presenceMetadata, peerId: String(this.peerId) },
    online_at: new Date().toISOString(),
    sessionId: this.sessionId,
  })
  this.ready = true
})
```

**协作会话管理（`src/lib/automerge/collaboration/session-manager.ts`）：**

```typescript
async enable(sessionId: string, callbacks: CollaborationCallbacks = {}) {
  const adapter = new SupabaseNetworkAdapter(this.options.resumeId, sessionId, callbacks)
  this.options.repo.networkSubsystem.addNetworkAdapter(adapter)
  // 触发自动同步
  this.syncHandle(this.options.getHandle())
}
```

当 `handle` 发生变化时，Automerge 会自动通过 `send()` 方法将 CRDT change 编码为 bytes，通过 Supabase 频道广播给所有在线协作者。

### 4. WebSocket 怎样建立连接？

**本项目不直接使用 WebSocket，而是通过 Supabase Realtime 间接实现。**

Supabase Realtime 底层基于 WebSocket 协议，建立连接的过程：

```
supabase.channel(name)  →  创建 RealtimeChannel 实例
       ↓
channel.subscribe()      →  建立 WebSocket 长连接（自动重连）
       ↓
channel.track()         →  注册用户在线状态（Presence）
       ↓
ready = true            →  连接建立完成，触发回调
```

**源码（`src/lib/automerge/collaboration/supabase-network-adapter.ts:208`）：**

```typescript
private subscribeToChannel(peerMetadata?: PeerMetadata) {
  this.channel?.subscribe(async (status) => {
    if (status !== 'SUBSCRIBED') return

    await this.channel?.track({
      peerId: String(this.peerId),
      metadata: { ...peerMetadata, ...this.presenceMetadata, peerId: String(this.peerId) },
      online_at: new Date().toISOString(),
      sessionId: this.sessionId,
    })

    this.ready = true
    this.callbacks.onChannelReady?.(this.channelName)
  })
}
```

Supabase Realtime 自动处理：
- WebSocket 握手和心跳
- 自动重连（网络波动时）
- 多路复用（同一个连接承载多个频道）

### 5. 光标同步的时机是什么？

**两个维度：发送时机 + 接收时机。**

#### 发送端（节流 + rAF 调度）：

```typescript
// src/lib/collaboration/cursor/hook.ts
const scheduleOutgoingCursorFlush = useCallback(() => {
  if (sendFrameRef.current !== null) return  // 已有待发送帧，跳过
  sendFrameRef.current = requestAnimationFrame((frameTime) => {
    flushOutgoingCursorRef.current(frameTime)  // 下一帧发送
  })
}, [])

// 实际发送时做节流
const flushOutgoingCursor = useCallback((frameTime: number) => {
  if (frameTime - lastSentAtRef.current < throttleMs) {
    // 节流：未到时间，调度到下一帧
    sendFrameRef.current = requestAnimationFrame((nextFrameTime) => {
      flushOutgoingCursorRef.current(nextFrameTime)
    })
    return
  }
  // 实际广播光标
  broadcastCursorPayload(channelRef.current, payload)
  lastSentAtRef.current = frameTime
}, [throttleMs, ...])
```

`throttleMs` 默认是多少？从 hook 调用方传入，默认通常是 **~50ms**（每帧约 16ms，所以约每 3 帧发一次）。

#### 接收端（批量合并 + rAF）：

```typescript
const scheduleRemoteCursorFlush = useCallback(() => {
  if (flushRemoteFrameRef.current !== null) return  // 已在队列中
  flushRemoteFrameRef.current = requestAnimationFrame(flushRemoteCursors)
}, [])

const flushRemoteCursors = useCallback(() => {
  flushRemoteFrameRef.current = null
  const batch = Object.values(pendingRemoteCursorsRef.current)
  pendingRemoteCursorsRef.current = {}
  if (batch.length === 0) return
  // 批量更新状态，合并同一帧内的多次光标移动
  setCursors(prev => upsertRealtimeCursorBatch(prev, batch))
}, [])
```

**时序总结：**
```
PointerMove → requestAnimationFrame → 节流检查 → 广播
                                           ↓
                                    对方接收 → requestAnimationFrame → 批量合并 → 渲染
```

## 二、离线优先架构

### 7. 离线优先怎么设计和实现的？

**三层存储架构：**

```
内存（Zustand 状态）
    ↓ 变化时延迟写入
IndexedDB（Automerge 文档 + 离线简历）
    ↓ 在线时异步同步
Supabase（云端持久化）
```

**两条独立路径：**

**路径 A — 在线简历（Supabase 关联）：**

```
编辑 → Zustand → Automerge Document → 延迟 3s → Supabase
                                   → IndexedDB（Automerge Repo 自动持久化）
```

**路径 B — 离线简历（纯本地）：**

```
编辑 → Zustand → 延迟 3s → IndexedDB（offline-resume-manager）
```

**关键代码（`src/store/resume/form.ts`）：**

```typescript
function applyResumeChange(set, get, stateUpdate, docUpdate?) {
  // 1. Optimistic UI - 立即更新本地状态
  set({ ...stateUpdate, pendingChanges: true })

  // 2. 离线模式 → 写 IndexedDB
  if (freshState.mode === 'offline' || isOfflineResumeId(resumeId)) {
    scheduleOfflinePersist(() => get().syncToSupabase())
    return
  }

  // 3. 在线模式 → 更新 Automerge + 延迟同步 Supabase
  if (docUpdate) {
    freshState.docManager?.change(doc => docUpdate(doc))
  }
  scheduleOnlinePersist(() => get().syncToSupabase())
}
```

**离线简历管理器（`src/lib/offline-resume-manager.ts`）：**

使用 `idb` 库（IndexedDB 封装）实现离线简历 CRUD：

```typescript
const DB_NAME = 'offline-resumes'
// 结构：{ resume_id, display_name, description, type, created_at, updated_at, data }
```

### 8. 本地保存的是全量还是增量？

**两者结合：**

- **IndexedDB 中的 Automerge 文档**：保存的是**全量快照**（`Automerge.save(doc)` 序列化的完整二进制）
- **Supabase 中的 `automerge_documents` 表**：也是**全量快照**（`document_data` 字段存 Base64 编码的完整二进制，`heads` 字段记录当前 Heads）

```typescript
// src/lib/automerge/document/persistence.ts:113
const binary = Automerge.save(doc)      // 全量序列化
const heads = Automerge.getHeads(doc)   // 当前 heads（用于增量同步判断）
```

Automerge Repo 本身支持增量 sync（只同步 diff），但本项目的 Supabase Network Adapter 实现的是**完整二进制广播**，没有使用 Automerge 的增量 sync 协议。

### 11. 联网了就需要上传吗？

**不是立即上传，而是延迟批量同步。**

编辑产生变化后，通过 `scheduleOnlinePersist()` 延迟 3 秒执行上传：

```typescript
// form.ts:81
export const ONLINE_SYNC_DELAY = 3000

function scheduleOnlinePersist(flushFn) {
  if (onlineSyncTimer) clearTimeout(onlineSyncTimer)
  onlineSyncTimer = setTimeout(flushFn, ONLINE_SYNC_DELAY)
}
```

即：用户停止编辑 3 秒后，才触发 Supabase 持久化。如果用户在 3 秒内继续编辑，计时器被重置。

---

### 12. 离线没有编辑也会同步吗？

**离线场景下，即使没有编辑，也会将当前状态持久化到 IndexedDB。**

但不会主动上传云端（因为离线）。上线后：
- 如果有 pending changes，会在 `syncToSupabase()` 时上传
- 如果无编辑变化，则没有需要同步的内容

**自动恢复机制**：重新打开已离线编辑的简历时，会从 IndexedDB 恢复最后一次保存的状态。
