## 1. 滴滴测试环境隔离怎么做的

## 答案
**测试环境隔离方案**

- **1.开发阶段隔离：**
针对不同的需求，开发初期会拉取独立的Git分支，确保各需求在各自分支上独立开发，互不干扰。
- **2.联调后测试环境：**
联调完成后，会在线下仿真环境（osim）进行测试。osim环境区别于线上和预发环境，采用流量隔离机制。
- **3.流量隔离原理：**
只需部署本次需求改动的服务，即可将标记的流量转发到自己的服务上，其他服务则复用基准环境。
- **4.流量转发机制：**
基于wings-agent实现流量转发，具体规则如下：

- .若流量的traceid未染色，则转发至本机服务。
- .若traceid已染色：
- .存在对应分支环境时，转发至该分支环境。
- .无对应分支环境时，转发至本机服务（通过127.0.0.1）。

- **5.隔离效果：**
通过上述机制，可有效隔离线上环境与其他需求开发环境，确保测试的独立性和准确性。
> 在开发初期，针对不同的需求会拉取不同的分支，各自开发在独立的分支上互不干扰，联调结束后，将会在线下仿真的osim环境中进行测试，有别于线上和预发环境，osim设计采用的是【流量隔离】的方式，只需部署本次需求改动的服务，就能够将标记的流量转发到自己的服务上，其他服务则复用基准环境【流量转发基于wings-agent，转发规则：若该流量的traceid未染色，转发至本机服务；若染色，有对应的分支环境时转发至服务对应的分支环境，否则转发给本机服务通过127.0.0.1】,因此就可以很好的隔离线上环境和其他的需求开发环境。

## 2. 整个需求开发的全流程

## 答案
# 整个需求开发的全流程

### 必要前提
- **所需必须文档：** 在启动任何开发工作前，必须准备好以下核心文档，以确保项目有章可循：
- **产品需求文档（PRD）：** 明确业务目标、功能细节和验收标准。
- **研发流程规范文档：** 统一团队的开发、测试、上线标准。
- **监控规范文档：** 定义线上服务的监控指标、报警阈值和处理流程。

### 启动阶段
- **需求评审：** 产品经理组织，研发、测试等关键角色参与，对PRD进行评审。
- **项目可行性：** 评估需求的商业价值和必要性。
- **技术可行性：** 研发团队初步评估技术实现难度和风险。
- **参与人员确定：** 明确项目负责人、开发、测试等核心成员。
- **输出：** 【技术方案文档】，包含系统架构、模块设计、接口定义等。

### 开发阶段
- **1）阶段一规划：**
- **功能点评估+排期：** 开发团队根据技术方案，拆解功能点并评估工作量，制定详细的开发排期。
- **技术方案评审：** 对于工期较长（如3日以上）或复杂度高的需求，需组织技术方案评审，确保方案的合理性与可行性。
- **2）阶段二开发：**
- **代码开发：** 开发人员依据排期和设计方案进行编码实现。
- **代码评审（CR）：** 代码合并前必须经过同事的Code Review，以保证代码质量和可维护性。
- **3）阶段三联调：**
- **前后端接口联调：** 前后端开发人员协同工作，确保接口数据交互正确。
- **自测：** 开发人员完成联调后，需进行充分的自测，确保主流程功能正常，减少提测后的低级Bug。

### 测试阶段
- **修复：** 开发人员需及时响应并修复测试人员提交的Bug。
- **监控P0级Bug：** 重点关注和优先解决影响核心功能的P0级Bug，确保产品质量底线。
- **输出：** 【测试准入报告】，由测试团队出具，确认产品已达到可上线的质量标准。

### 灰度阶段
- **灰度观察：** 产品在小范围用户中发布，期间需密切观察各项监控指标。
- **观察Sentry异常：** 关注前端和后端的错误日志，及时发现代码异常。
- **天眼监控报警：** 留意业务监控平台的报警信息，确保业务指标稳定。
- **要求：** 灰度时间需**至少经历一个业务高峰期且总时长大于24小时**，以充分验证新版本在真实环境下的稳定性。

### 上线阶段
- **上线确认：** 全量发布前，进行最后的检查和确认。
- **回归功能和主流程：** 对核心功能和主业务流程进行快速回归测试。
- **监控报警认领：** 确认所有监控报警规则已配置且相关人员已认领，确保问题能被及时发现和处理。
- **要求：** 严格遵守**封线期**规定，即**周一到周四20点以后，以及周五全天**禁止进行非紧急的上线操作，以保障线上服务的稳定性。

### 收尾阶段
- **复盘总结：** 项目上线稳定后，组织项目复盘会议。
- **Bug原因分析：** 基于【测试准入报告】和线上问题记录，分析Bug产生的原因，总结经验教训。
- **监控是否到位：** 回顾本次需求的监控覆盖情况，评估监控是否能有效发现问题。
- **线上问题总结：** 汇总并分析所有线上问题，形成知识库，避免同类问题再次发生。

## 3. openSpec规范如何制定，开发者主要参与的环节是怎样？

## 答案
### 1. OpenSpec 规范的制定

OpenSpec 的核心不是简单的文档，而是**结构化的“行为契约”**。它通过强制 AI 将模糊的需求转化为具体的、可验证的结构化文件。

*   **核心定位：** 管理 Spec 的“文件结构”和“行为定义”，确保输出结构可审核。
*   **制定方式：**
    1.  **基于上下文生成：** 利用 `/opsx:propose` 命令，输入需求描述或参考 Superpowers 生成的设计文档（Design Doc）。
    2.  **标准化结构：** OpenSpec 会自动生成一套标准文件结构，包含：
        *   `proposal.md`：变更的背景（Why）、具体内容（What）及影响评估（Impact）。
        *   `specs/*.md`：**最关键部分**，使用 `WHEN/THEN`（给定条件/预期结果）的形式定义具体的行为规格。
        *   `design.md`：数据结构、API 定义、时序图等技术实现细节。
        *   `tasks.md`：将实现过程拆解为具体的、有序的子任务清单。
    3.  **明确边界：** 规范中必须包含 **IN SCOPE**（包含范围）和 **OUT OF SCOPE**（明确不做），防止 AI 出现“过度优化”或“幻觉重构”。

### 2. 结合 Superpowers 将“想法”变“代码”的全流程

两者是互补关系，结合逻辑为：**思维过程有纪律（Superpowers） + 输出结构可审核（OpenSpec）**。

| 阶段 | 使用工具 | 核心动作 | 产出物 | 目的 |
| :--- | :--- | :--- | :--- | :--- |
| **1. 探索阶段** | **Superpowers** | `/brainstorm` | 设计文档 (Design Doc) | 厘清需求边界，探索技术可行性，不急着写代码。 |
| **2. 规格阶段** | **OpenSpec** | `/opsx:propose` | 结构化 Spec 文件 (specs/, tasks.md) | 将设计转化为可执行的“契约”，明确每一步怎么做。 |
| **3. 实现阶段** | **Superpowers** | `/execute-plan` | 代码 (Code) | 读取 tasks.md 严格按步执行，每步验证。 |
| **4. 归档阶段** | **OpenSpec** | `/opsx:archive` | 知识库资产 | 将本次变更沉淀为主规格库，供后续需求复用。 |

**流程详解：**
1.  **先想后做：** 用 Superpowers 的 `/brainstorm` 引导 AI 提问（如“核心场景是什么？”“生效条件复杂吗？”），产出设计文档。
2.  **生成契约：** 将设计文档喂给 OpenSpec，生成 `tasks.md`。
3.  **严格执行：** 开发者审核通过后，Superpowers 读取 `tasks.md`，按步执行 `/execute-plan`。AI 必须在每步完成后验证（如跑通测试），不能一次性写完所有代码。
4.  **知识沉淀：** 完成后用 `/opsx:archive` 归档，形成团队的知识资产。

### 3. 开发者主要参与的环节

在这个流程中，开发者从“写代码的机器”转变为**流程指挥官**和**质量守门员**。主要参与环节如下：

*   **需求引导与确认（Initiation）：**
    *   发起 `/brainstorm`，回答 AI 提出的关键问题（如场景、限制条件），确保 AI 理解正确。
    *   **关键动作：** 判断 AI 提出的方案是否符合业务预期。

*   **规格审核（The Crucial Gate）：**
    *   这是**最有价值**的环节。在 AI 生成代码前，开发者必须人工审核 OpenSpec 产出的文件。
    *   **审核清单：**
        *   `proposal.md`：Why/What 是否完整？影响评估是否准确？
        *   `specs/*.md`：`WHEN/THEN` 是否覆盖了边界情况（如空值、错误码）？
        *   `tasks.md`：任务顺序是否正确？有无循环依赖？
        *   `design.md`：数据结构是否合理？（例如：审核时发现漏了 `used_by` 字段并修正）。

*   **流程控制与异常处理（Control & Exception Handling）：**
    *   决定何时执行 `/opsx:apply` 和 `/execute-plan`。
    *   当 AI 在执行过程中遇到无法解决的偏离或报错时，介入进行人工干预或调整 Spec。

*   **最终复盘（Retrospective）：**
    *   使用 `/opsx:archive` 前，确认所有任务完成且测试通过。
    *   定期回顾主规格库，删除过时内容，保持文档的时效性。

通过这种方式，开发者将精力集中在了**思考“做什么”**（Specification）和**把控“质量”**（Review）上，而将重复性的编码工作交给了 AI，极大地提高了开发的可靠性和效率。

## 4. JSON Schema 动态表单方案设计怎么实现？

## 答案
### 0. 何为 JSON Schema

**JSON Schema** 是一种基于 JSON 的**数据描述语言**，用于定义 JSON 数据的结构、类型、格式和校验规则。它本质上是一个“元数据”文件，用来描述另一个 JSON 数据应该长什么样。

*   **核心作用**：
    *   **描述数据结构**：定义对象、数组、字符串、数字等数据类型。
    *   **定义约束条件**：设置字段是否必填、字符串长度限制、数值范围、正则匹配等。
    *   **数据校验**：基于定义的规则，自动验证一份 JSON 数据是否合法（Valid）。
*   **在此方案中的角色**：它是**“契约”**。前端渲染器（RJSF）读取这份契约，就知道该画出什么样的表单，以及用户输入的数据是否符合要求，从而实现了“配置驱动”。
### 1. 核心架构与流程
整个系统分为四层，通过解析器将静态配置转化为动态 UI：

1.  **Schema 定义层**：由服务端或配置中心提供 JSON Schema，定义字段类型、校验规则和 UI 属性。
2.  **解析器层**：使用 `react-jsonschema-form` (RJSF) 作为核心解析引擎。
3.  **渲染器层**：将解析后的指令映射为具体的 React 组件。
4.  **UI 组件层**：基于 Ant Design 的实际输入控件（Input, Select 等）。

### 2. 核心实现步骤

#### 2.1 Schema 规范定义 (配置源)
这是实现动态化的基础。你需要定义一套包含数据结构和 UI 指令的 JSON Schema。

*   **数据结构定义**：定义字段类型（string, number, array 等）。
*   **UI 扩展字段 (ui:)**：这是实现 Ant Design 样式的关键，用于指定组件类型、占位符、禁用状态等。

| 扩展字段 | 说明 | 示例 |
| :--- | :--- | :--- |
| **ui:widget** | 指定渲染组件 | `textarea`, `radio`, `date` |
| **ui:placeholder** | 输入框提示语 | `"请输入政策名称"` |
| **ui:disabled** | 是否禁用 | `true` / `false` |
| **ui:options** | 传递额外属性 | `{ allowClear: true }` |
| **ui:rules** | 自定义校验 | `[{ required: true }]` |

**示例 Schema：**
```json
{
  "type": "object",
  "properties": {
    "policyName": {
      "type": "string",
      "title": "政策名称",
      "maxLength": 50,
      "ui:placeholder": "请输入政策名称"
    },
    "airline": {
      "type": "string",
      "title": "航空公司",
      "enum": ["CA", "MU", "CZ"],
      "enumNames": ["国航", "东航", "南航"],
      "ui:widget": "radio"
    }
  },
  "required": ["policyName", "airline"]
}
```

#### 2.2 核心渲染器封装 (React 组件)
创建一个通用的 `JsonSchemaForm` 组件，负责接收 Schema 和数据，并注入自定义的 Ant Design 主题。

**实现要点：**
1.  引入 `rjsf-antd` 核心库。
2.  注入自定义的 `Widgets`（组件映射）和 `Fields`（字段映射）。
3.  支持只读模式（`readOnly`）。

```jsx
import React from 'react';
import Form from 'rjsf-antd';
import { Widgets } from './widgets'; // 引入自定义组件映射

const JsonSchemaForm = ({ schema, formData, onChange, onSubmit, readOnly = false }) => {
  return (
    <Form 
      schema={schema} 
      formData={formData} 
      onChange={onChange} 
      onSubmit={onSubmit} 
      widgets={Widgets} // 注册 Antd 组件
      readOnly={readOnly} 
    />
  );
};
export default JsonSchemaForm;
```

#### 2.3 Ant Design 主题适配 (Widgets)
这是实现 UI 统一的关键。你需要将 JSON Schema 中的通用类型映射为具体的 Ant Design 组件。

**主要映射关系：**
*   **TextWidget** -> `Input`
*   **TextareaWidget** -> `Input.TextArea`
*   **NumberWidget** -> `InputNumber`
*   **SelectWidget** -> `Select`
*   **DateWidget** -> `DatePicker`
*   **RadioWidget** -> `Radio.Group`

**代码示例 (SelectWidget)：**
```jsx
import { Select } from 'antd';

// 下拉选择组件映射
SelectWidget: (props) => (
  <Select 
    value={props.value} 
    onChange={props.onChange} 
    placeholder={props.placeholder}
  >
    {props.options.enum.map((value, index) => (
      <Select.Option key={value} value={value}>
        {props.options.enumNames?.[index] || value}
      </Select.Option>
    ))}
  </Select>
)
```

### 3. 实施与应用流程

1.  **环境准备**：
    安装依赖包 `react-jsonschema-form` 和 `antd`。
2.  **组件开发**：
    按照上述 2.2 和 2.3 步骤开发核心渲染组件和适配器。
3.  **页面接入**：
    *   **旧页面改造**：将原有的硬编码表单替换为 `<JsonSchemaForm />`，并从后端接口获取 Schema。
    *   **新页面开发**：直接编写 JSON Schema 配置，无需编写表单代码。
4.  **动态加载**：
    前端请求服务端接口获取 Schema，动态渲染表单。

### 4. 预期收益
*   **效率提升**：新增字段无需修改前端代码，仅需修改 JSON 配置，开发效率提升 60% 以上。
*   **维护性增强**：表单逻辑集中在配置文件中，版本可追溯，逻辑清晰。
*   **一致性保障**：所有表单自动遵循 Ant Design 设计规范，避免样式不统一。

## 5. 实习中性能优化怎么做的？什么指标来衡量的？

## 答案
### 🚀 前端应用性能优化方案

#### 1. 路由级代码分割 (Code Splitting)
采用 **`@loadable/component`** 实现路由组件的按需加载，解决首屏体积过大问题。

*   **实现方式**：
    ```js
    // routeConfig.js
    const ROUTETOCOMPONENTS = {
      '/ebk': loadable(() => import(`./pages/ebk`), {
        fallback: <div>Loading...</div> // 配合骨架屏使用
      }),
      // ... 共配置 57 个页面懒加载
    };
    ```
*   **优化收益**：
    *   **首屏 JS 按需加载**：仅加载当前路由所需代码。
    *   **弱网体验**：用户点击导航时才触发对应 Chunk 加载，显著降低首屏阻塞。

#### 2. 预加载与加载策略
通过多层级的预加载策略，消除用户感知的“白屏”时间。

| 策略 | 实现方案 | 核心价值 |
| :--- | :--- | :--- |
| **骨架屏 Fallback** | 结合 Loadable 的 `fallback` 属性 | 提供视觉反馈，降低用户对等待的焦虑感 |
| **Webpack Prefetch** | `<link rel="prefetch">` | 浏览器空闲时预取后续可能访问的 Chunk |
| **预判式加载** | 监听 `MouseEnter` (鼠标悬停) 导航事件 | 在用户点击前预加载目标页面资源 |

#### 3. 构建配置优化 (Webpack/Vite)
精细化构建配置，从源头减少资源体积。

*   **JS 压缩**：启用 `TerserPlugin` 进行多进程压缩与混淆。
*   **CSS 提取**：使用 `mini-css-extract-plugin` 提取 CSS，避免 CSS in JS 导致的 FOUC（无样式内容闪烁）。
*   **库的按需引入**：
    *   **Ant Design**：配置 `babel-plugin-import`，仅打包使用的组件。
    *   **日期库**：将 `moment.js` (200KB+) 替换为 `dayjs` (2KB)，实现轻量化。
*   **Tree Shaking**：确保 ES Module 语法，移除未引用代码。

#### 4. 依赖与工程化治理
建立统一的开发规范，消除冗余代码。

*   **依赖审计**：
    *   工具：`npm audit` / `depcheck`
    *   动作：定期清理未使用的依赖（DevDependencies）。
*   **统一封装**：
    *   请求层：`src/utils/request.js` (统一拦截器、Token 处理)。
    *   工具库：`src/utils/util.js` (复用通用函数)。
*   **路径别名**：配置 `@` 别名，减少深层级相对路径引用，提升代码可读性。

## 6. 重构任务参与度如何？基于什么样的目的以及具体重构解耦的方案？

## 答案
#### 重构任务参与度

**角色定位**：模块化重构的具体执行者（导师统筹架构，我负责具体实现）

#### 重构背景与目的

**1. 原有痛点**

滴滴企业版商旅平台存在 55+ 业务模块，遗留系统问题突出：

| 问题 | 表现 |

## 8. 为什么要选择CRDT算法而不是OT算法，有什么区别吗？

## 答案
**CRDT vs OT 核心区别**

| 维度 | CRDT | OT (Operational Transformation) |
|------|------|----------------------------------|
| 冲突处理 | 数学证明最终一致性 | 依赖中心服务器转换 |
| 离线支持 | 完全支持，离线合并 | 需要服务器参与，难离线 |
| 服务器依赖 | 只需要广播通道 | 需要中心服务器做转换 |
| 复杂度 | 库实现复杂，应用简单 | 应用需要处理复杂转换逻辑 |
| 扩展性 | 容易扩展到多用户 | 用户增多复杂度指数增长 |

**为什么选择 CRDT (Automerge)：**

1. **离线优先架构**
   - 用户可能在地铁、飞机等无网络环境编辑
   - CRDT 可以在本地先合并，网络恢复后自动同步

2. **无中心依赖**
   - Supabase 只需充当广播通道，不需要复杂转换逻辑
   - 即使 Supabase 暂时不可用，本地编辑不受影响

3. **多端同步简单**
   - 同一个文档可以在手机、平板、电脑同时编辑
   - 每个设备都是对等的，不用担心谁先谁后

**Automerge 的特点：**
```typescript
// 文档变更被编码为操作列表
const [doc, patch] = Automerge.change(doc, (d) => {
  d.name = "张三"
})
// 合并时只需要应用 patch，自动解决冲突
```

## 9. 核心功能中的冲突处理如何解决的？有哪些情况对应处理机制是怎样的？

## 答案
**冲突来源分类**

在简历编辑中，冲突主要分为三类：

1. **同时编辑同一字段**
   - 用户 A 和用户 B 同时修改"工作经历"中的同一公司
   - **处理机制**：Last-Writer-Wins (LWW) + 语义合并
   - Automerge 会保留两个操作，合并时按时间戳或自定义规则选择

2. **结构冲突（增删改同时发生）**
   - 用户 A 删除了一个模块，用户 B 在编辑该模块
   - **处理机制**：
     - 删除优先于编辑（被删除的内容不显示）
     - 删除操作本身会被记录，可追溯

3. **光标位置冲突**
   - 多人协作时，光标位置可能因为他人插入内容而偏移
   - **处理机制**：光标位置用相对位置表示（如"第5行后"而非"第100个字符"）

**CRDT 的冲突处理原理**

```typescript
// Automerge 的冲突解决示例
const [doc, conflict] = await repo.merge(doc)

// conflict 包含所有冲突的版本
// 应用层可以选择如何呈现（如显示"有冲突，需要手动合并"）
```

**UI 层处理**

- **冲突提示**：红色标记有冲突的内容块
- **手动合并**：用户可以选择保留哪个版本
- **版本历史**：所有变更都有记录，可回滚

## 15. 什么是安全流式调用？怎么基于 SupabaseEdgeFunctions 实现

## 答案
**安全流式调用概念**

安全流式调用 = **安全传输** + **流式响应**

- **安全传输**：敏感信息（API Key）不暴露在客户端
- **流式响应**：AI 生成内容逐步返回，用户立即看到结果

**为什么需要流式？**
- 简历优化建议可能很长
- 用户不用等待全部生成，可实时看到进度
- 体验接近"打字"效果

**Supabase Edge Functions 实现**

```typescript
// supabase/functions/ai-optimize/index.ts
import { serve } from 'https://deno.land/std@0.177.0/http/server.ts'
import OpenAI from 'https://deno.sh/openai@v4.20.0/mod.ts'

const openai = new OpenAI()

serve(async (req) => {
  // 1. 安全验证：检查 Supabase 认证
  const authHeader = req.headers.get('Authorization')
  if (!authHeader) {
    return new Response('Unauthorized', { status: 401 })
  }

  // 2. 解析请求
  const { resumeText, jobDescription } = await req.json()

  // 3. 构建 Prompt
  const systemPrompt = `你是一个专业的简历优化顾问...`
  const userPrompt = `简历内容：${resumeText}\n\n岗位要求：${jobDescription}`

  // 4. 流式返回
  return new Response(
    new ReadableStream({
      async start(controller) {
        const stream = await openai.chat.completions.create({
          model: 'gpt-4',
          messages: [
            { role: 'system', content: systemPrompt },
            { role: 'user', content: userPrompt }
          ],
          stream: true
        })

        for await (const chunk of stream) {
          const text = chunk.choices[0]?.delta?.content || ''
          if (text) {
            controller.enqueue(new TextEncoder().encode(text))
          }
        }
        controller.close()
      }
    }),
    {
      headers: {
        'Content-Type': 'text/plain; charset=utf-8',
        'Transfer-Encoding': 'chunked',
      }
    }
  )
})
```

**前端调用**

```typescript
// React 中调用流式 API
const response = await fetch('https://xxx.supabase.co/functions/v1/ai-optimize', {
```

## 16. React的fiber怎么理解？如何实现可中断？

## 答案
### React Fiber 架构理解

**Fiber 的本质**：Fiber 是 React 16 引入的新的调和引擎，核心目标是**可中断的异步渲染**。

**为什么需要 Fiber**：
- 旧版 React 使用同步递归调和，渲染一颗大树可能需要几十甚至几百毫秒
- 在这期间，浏览器主线程被阻塞，无法响应用户交互（点击、输入等）
- Fiber 通过将渲染工作拆分成**可中断的小单元**，让浏览器可以在帧间抽空处理高优先级任务

**Fiber 的数据结构**：

```typescript
type Fiber = {
  // 链表结构
  child: Fiber | null;      // 第一个子节点
  sibling: Fiber | null;    // 下一个兄弟节点
  return: Fiber | null;    // 父节点

  // 工作单元
  pendingProps: any;
  memoizedProps: any;
  memoizedState: any;

  // 更新队列
  updateQueue: UpdateQueue<any>;

  // 替代（旧同步调和）
  // alternate: Fiber;
}
```

### 可中断的实现原理

**1. 双缓存池 (Double Buffering)**
```
current Fiber Tree (正在渲染)
        ↕ 切换
workInProgress Fiber Tree (内存中构建)
```
- `current` 是已经渲染完成的树
- `workInProgress` 是在内存中构建的新树
- 完成后通过 `alternate` 指针切换，两者角色互换

**2. 优先级队列 ( lanes / expirationTime )**

```typescript
// React 17+ 使用 lanes 模型
export type Lanes = number;
export type Lane = number;

const lanes: Lanes = 0b0000000000000000000000000001001;
//                └───│───│───│───│───└─── Sync Lane
//                    └───│───│─── Continuous Non-Urgent
//                        └───│─── Discretionary Deferrable
```

**3. 工作单元拆分**

```
reconcile (调和阶段)
├── beginWork()    → 向下遍历，构建 Fiber 节点
│   └── 更新 props、state
│   └── 根据 diff 算法标记 effect (Placement, Update, Deletion)
│
└── completeWork() → 向上归并，收集 effect 列表

commit (提交阶段) - 不可中断
├── DOM 变更
├── useLayoutEffects
└── 调度 useEffects
```

**4. 中断与恢复机制**

```typescript
function workLoopConcurrent() {
  while (workInProgress !== null && !shouldYield()) {
    // shouldYield() 检查是否有更高优先级任务
    workInProgress = performUnitOfWork(workInProgress);
  }
}

// shouldYield() 实现
function shouldYield() {
  const deadline = getCurrentTime() + YIELD_THRESHOLD; // 约 5ms
  return getCurrentTime() >= deadline ||
         Scheduler.unstable_needsYieldscheduler();
}
```

**5. 优先级级别**

| 优先级 | Lane | 场景 |
|--------|------|------|
| Sync | 同步 | 用户输入、点击 |
| InputContinuous | 连续输入 | 拖拽、滚动 |
| Default | 默认 | 数据获取 |
| Background | 后台 | 预渲染 |
| Idle | 空闲 | 不重要更新 |

### 总结

> **Fiber = 链表 + 双缓冲 + 优先级调度**
> - 链表替代递归，实现深度优先的增量遍历
> - 双缓冲实现无锁的树切换
> - 优先级队列确保高优先级任务优先执行

## 17. 输入URL到渲染出页面全过程？

## 答案
### 完整渲染流程

```
用户输入 URL
    ↓
1. DNS 解析
    ↓
2. TCP 连接 (三次握手)
    ↓
3. TLS 握手 (HTTPS)
    ↓
4. HTTP 请求发送
    ↓
5. 服务器处理请求
    ↓
6. HTTP 响应
    ↓
7. 浏览器接收响应
    ↓
8-15. 浏览器渲染 pipeline
    ↓
16. JavaScript 框架渲染
```

### 详细步骤

#### 网络层

**1. DNS 解析**
```
浏览器缓存 → 系统缓存 → 路由器缓存 → ISP DNS缓存 → 递归查询
```
- 浏览器先检查缓存（DNS Cache）
- 约 65ms - 120ms

**2. TCP 三次握手**
```
Client: SYN=1, seq=x          → Server
Server: SYN=1, ACK=1, seq=y    → Client
Client: ACK=1                  → Server
```
- 约 56ms - 84ms

**3. TLS 握手 (HTTPS)**
```
Client: 发送支持的加密套件 + Random1
Server: 发送证书 + Random2 + 选择加密套件
Client: 验证证书 + 发送 PreMasterSecret
两者: 生成对称密钥
```
- 约 100ms - 300ms

#### 浏览器渲染 Pipeline

**8. 构建 DOM 树**
```
HTML → Tokenizer → Tokens → Lexer → DOM Nodes → DOM Tree
```

**9. 构建 CSSOM 树**
```
CSS → Parser → CSSOM Tree
```

**10. Render Tree (渲染树)**
```
DOM + CSSOM → Render Tree (只包含可见节点)
```

**11. Layout (布局/回流)**
- 计算每个节点的几何信息（位置、大小）
- **触发条件**：首次渲染、窗口 resize、元素位置变化

**12. Paint (绘制)**
- 将渲染树转换为屏幕像素
- 分解为多个图层（Layers）

**13. Composite (合成)**
- 将不同图层合成为最终画面
- 使用 GPU 加速

#### JavaScript 框架渲染 (React 为例)

**14. React Fiber 调和**
```
setState() → 创建 Update → 调度 Lane
    ↓
ReactDOM.render() → beginWork() → completeWork()
    ↓
标记 Side Effects (Placement, Update, Deletion)
```

**15. Commit 阶段**
```
DOM 操作 → useLayoutEffects → 浏览器 Paint
```

**16. 浏览器最终绘制**
- 像素写入屏幕

### 关键指标

| 阶段 | 耗时 (4G) | 耗时 (弱网) |
|------|-----------|-------------|
| DNS | 20-120ms | 200ms+ |
| TCP | 56-84ms | 300ms+ |
| TLS | 100-300ms | 500ms+ |
| TTFB | 40-80ms | 1000ms+ |
| DOMContentLoaded | 500ms-2s | 5s+ |
| FCP | 1-2s | 5s+ |
| LCP | 1.5-2.5s | 6s+ |

### 性能优化点

| 优化手段 | 针对阶段 |
|---------|---------|
| DNS 预解析 `<link rel="dns-prefetch">` | DNS |
| HTTP/2 多路复用 | TCP + TLS |
| CDN 加速 | 网络传输 |
| 关键 CSS 内联 | DOM/CSSOM |
| 代码分割、懒加载 | JS 执行 |
| SSR/SSG | 首屏渲染 |

## 21. GResume项目采用的哪种设计模式？

### **1. 适配器模式 (核心 - 协作层)**

```typescript
// src/lib/automerge/collaboration/supabase-network-adapter.ts
// 将 Supabase Realtime Channel 适配为 Automerge 的 NetworkAdapter 接口
export class SupabaseNetworkAdapter extends NetworkAdapter {
  connect(peerId: PeerId, peerMetadata?: PeerMetadata) {
    this.channel = supabase.channel(this.channelName)
    this.registerSyncBroadcast()
    this.subscribeToChannel(peerMetadata)
  }

  send(message: Message) {
    // 将 Automerge 消息格式转换为 Supabase broadcast 格式
    this.channel.send({
      type: 'broadcast',
      event: 'automerge-sync',
      payload: { senderId: this.peerId, message: encodeBytesToBase64(message.data) },
    })
  }
}
```

#### **作用**：适配器是整个实时协作的枢纽，让 Automerge 的 P2P 同步能力与 Supabase 的广播能力对接。

```
└── 代理模式 → 拦截变更、元数据更新

持久层
└── 适配器模式 → AutomergeDocumentPersistence (存储适配)
```

## 22. HTTPS传输过程的【对称加密】和【非对称加密】的作用

## 答案
### HTTPS 加密体系概述

HTTPS = HTTP + TLS/SSL，其核心目标是：在不安全的网络上建立安全通信。

```
┌─────────────────────────────────────────────────────────────┐
│                     HTTPS 握手流程                            │
├─────────────────────────────────────────────────────────────┤
│  1.  TCP 三次握手 (建立连接)                                  │
│  2.  TLS 握手 (协商加密算法 + 交换密钥)                        │
│  3.  对称加密通信 (传输数据)                                  │
└─────────────────────────────────────────────────────────────┘
```

### 非对称加密 (Asymmetric Encryption) - 密钥交换阶段

**作用**：在握手阶段安全地协商出对称密钥

| 特性 | 说明 |
|------|------|
| **算法** | RSA、ECDHE |
| **密钥对** | 公钥 (公开) + 私钥 (保密) |
| **特点** | 加密/解密使用不同密钥 |
| **性能** | 计算慢，适合短数据 |

**工作流程**：
```
Client                              Server
  │                                    │
  │  ─── 发送支持的加密套件 + Random1 ──→  │
  │  ←── 发送证书 + Random2 + 选择加密套件 ──  │
  │  ─── 验证证书 + 发送 PreMasterSecret ─→  │
  │      (用公钥加密，只有服务器能用私钥解密)    │
  │                                    │
  └── 双方用 Random1 + Random2 + PMS 生成对称密钥 ──┘
```

**为什么用非对称加密？**
- 公钥可以公开传输，不怕被窃取
- 只有服务器用私钥能解密 PreMasterSecret
- 解决了对称加密"密钥传输"的安全问题

### 对称加密 (Symmetric Encryption) - 数据传输阶段

**作用**：实际业务数据的加密传输

| 特性 | 说明 |
|------|------|
| **算法** | AES-128-GCM、AES-256-GCM、ChaCha20 |
| **密钥** | 双方共享的同一个密钥 |
| **特点** | 加密/解密使用相同密钥 |
| **性能** | 计算快，适合大量数据 |

**工作流程**：
```
Client                              Server
  │                                    │
  │  ←────── 用对称密钥加密 HTTP 数据 ──────→  │
  │  ←────── 用 MAC 验证数据完整性 ──────→  │
  │                                    │
  (每次请求使用独立的 IV + 认证标签防止重放攻击)
```

### 完整 TLS 握手 + 加密通信流程

```
┌──────────────────────────────────────────────────────────────┐
│                     完整 HTTPS 流程                            │
├──────────────────────────────────────────────────────────────┤
│ 1. TCP 三次握手                                               │
│    SYN → SYN-ACK → ACK                                       │
├──────────────────────────────────────────────────────────────┤
│ 2. TLS 握手 (非对称加密阶段)                                   │
│    • 协商 TLS 版本、加密套件                                   │
│    • 验证服务器证书 (CA 证书链)                               │
│    • 交换密钥材料 (ECDH/RSA)                                  │
│    • 生成 session key (对称密钥)                               │
├──────────────────────────────────────────────────────────────┤
│ 3. 对称加密通信 (应用数据阶段)                                 │
│    • HTTP 请求/响应全部用 session key 加密                     │
│    • AES-GCM 提供加密 + 完整性校验                             │
│    • 每次请求使用独立 IV (防止重放)                            │
└──────────────────────────────────────────────────────────────┘
```

### 为什么不全程使用非对称加密？

| 维度 | 非对称加密 | 对称加密 |
|------|-----------|----------|
| **计算速度** | 慢 100-1000 倍 | 快 |
| **适用数据量** | 短数据 (密钥交换) | 大量数据 (业务传输) |
| **密钥管理** | 只需保护私钥 | 双方需共享密钥 |

**结论**：
- **非对称加密** = 解决"如何在不安全通道中安全传递密钥"
- **对称加密** = 利用已安全协商的密钥高效传输数据
- 两者结合 = 安全 + 高效

### 实际面试加分点
> **现代 TLS 1.3 优化**：
> - 握手从 2-RTT 降为 1-RTT
> - 废除了不安全的加密套件 (RSA、3DES)
> - 0-RTT 恢复会话 (但有重放攻击风险)

## 23. 浏览器缓存机制

## 答案
### 缓存分类总览

浏览器缓存分为两大类：**强缓存**和**协商缓存**，命中顺序依次递进。

```
┌─────────────────────────────────────────────────────────────┐
│                    浏览器缓存命中流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   请求资源                                                   │
│       ↓                                                      │
│   ┌─────────────────┐                                       │
│   │  强缓存 (本地)    │  命中 → 直接使用缓存 → 返回 200 (from cache) │
│   └────────┬────────┘                                       │
│            ↓ 未命中                                           │
│   ┌─────────────────┐                                       │
│   │  协商缓存 (服务端) │  命中 → 返回 304 (Not Modified)        │
│   └────────┬────────┘                                       │
│            ↓ 未命中                                           │
│   ┌─────────────────┐                                       │
│   │   返回新资源      │  → 更新缓存 → 返回 200                   │
│   └─────────────────┘                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1. 强缓存 (Strong Cache)

**特点**：不发请求到服务器，直接使用本地缓存

#### Cache-Control (优先级高)

| 指令 | 作用 |
|------|------|
| `max-age=3600` | 资源在 3600 秒后过期 |
| `no-cache` | 跳过强缓存，但进行协商缓存 |
| `no-store` | 完全不使用缓存 |
| `private` | 只能在浏览器缓存，不能被 CDN 缓存 |
| `public` | 可以被任何节点缓存 (浏览器 + CDN) |

```
Cache-Control: max-age=3600, private
```

#### Expires (HTTP/1.0 兼容)

```
Expires: Wed, 21 Oct 2026 07:28:00 GMT
```

**注意**：因为是绝对时间，可能存在客户端与服务端时间不一致的问题，所以 `Cache-Control` 优先。

### 2. 协商缓存 (Conditional Cache)

**特点**：发请求到服务器，由服务器判断是否使用缓存

#### Last-Modified / If-Modified-Since

```
响应头: Last-Modified: Wed, 21 Oct 2026 07:28:00 GMT
请求头: If-Modified-Since: Wed, 21 Oct 2026 07:28:00 GMT
```

- 服务器对比资源修改时间
- 未修改 → 返回 `304 Not Modified`
- 已修改 → 返回新资源 + `200 OK`

#### ETag / If-None-Match (优先级高)

```
响应头: ETag: "33a64df551425fcc55e4d42a148795d9"
请求头: If-None-Match: "33a64df551425fcc55e4d42a148795d9"
```

- 服务器对比资源的内容指纹 (Hash)
- 精确到内容级别
- 解决了 Last-Modified 秒级精度问题

### 3. 缓存实践策略

| 资源类型 | 策略 | 原因 |
|---------|------|------|
| **HTML** | `Cache-Control: no-cache` | 必须及时获取最新版本 |
| **静态资源 (CSS/JS/图片)** | `max-age=31536000` + 文件指纹 | 长期缓存 + 变更后自动更新 |
| **API 接口** | `Cache-Control: no-store` | 数据敏感，不缓存 |
| **用户图片/上传** | `Cache-Control: private` | 只有浏览器可缓存 |

### 4. Service Worker 缓存 (PWA)

```
┌─────────────────────────────────────────────────────────────┐
│                   Service Worker 缓存策略                    │
├─────────────────────────────────────────────────────────────┤
│ CacheFirst:    缓存优先 → 网络回退 (静态资源)                  │
│ NetworkFirst:  网络优先 → 缓存回退 (API 数据)                 │
│ StaleWhileRevalidate: 先用缓存，同时后台更新 (常用策略)        │
└─────────────────────────────────────────────────────────────┘
```

### 5. 面试加分点

> **缓存失效机制**：
> - 静态资源：文件名加 Hash (`app.a1b2c3d4.js`)，变更后 URL 变 -> 自动获取新资源
> - HTML：设置 `no-cache`，确保每次拉取最新版本
>
> **CDN 缓存**：CDN 节点作为"中间缓存"，减少回源次数
>
> **启发式缓存**：`Cache-Control` 和 `Expires` 都没有时，浏览器会"启发式"缓存一段时间

## 24. vue3核心语法的总结（从react迁移学习vue3)

## 答案
### React → Vue3 对照表

| 概念 | React | Vue 3 |
|------|-------|-------|
| **状态管理** | `useState` | `ref` / `reactive` |
| **副作用** | `useEffect` | `watchEffect` / `watch` |
| **组件通信** | Props + Context | Props + Emits + Provide/Inject |
| **模板语法** | JSX | SFC (`<template>`) + 组合式 API |
| **响应式原理** | 不可变 + 触发更新 | 响应式代理 (Proxy) |
| **生命周期** | useEffect 清理 | `onMounted` / `onUnmounted` |

### 1. 响应式系统

**React 方式 (不可变更新)**：
```jsx
const [count, setCount] = useState(0)
// 必须创建新对象/数组
setCount(count + 1)
setItems([...items, newItem])  // 扩展运算符创建新数组
```

**Vue 3 方式 (响应式代理)**：
```typescript
// 基础类型用 ref
const count = ref(0)
count.value++  // .value 访问

// 对象/数组用 reactive
const state = reactive({
  count: 0,
  items: []
})
state.count++
state.items.push(newItem)  // 直接修改，Vue 自动追踪
```

### 2. 组合式 API (Composition API) - 核心变化

**React Hooks vs Vue Composition API**：

```tsx
// React: 自定义 Hook 逻辑复用
function useCounter() {
  const [count, setCount] = useState(0)
  const increment = () => setCount(c => c + 1)
  return { count, increment }
}

// Vue 3: composable 函数
function useCounter() {
  const count = ref(0)
  const increment = () => count.value++
  return { count: readonly(count), increment }
}
```

**关键区别**：
- Vue 的 `ref` 需要 `.value`，但模板会自动解包
- Vue 的响应式是"自动追踪"，React 需要"不可变约定"

### 3. 生命周期对比

| React | Vue 3 | 时机 |
|-------|-------|------|
| `componentDidMount` | `onMounted` | 挂载后 |
| `componentWillUnmount` | `onUnmounted` | 卸载前 |
| `componentDidUpdate` | `onUpdated` | 更新后 |
| `useEffect(() => {}, [])` | `onMounted` | 仅首次 |
| `useEffect(() => {}, [dep])` | `watchEffect` / `watch` | 依赖变化 |

### 4.  computed vs useMemo

```typescript
// React
const doubled = useMemo(() => count * 2, [count])

// Vue 3
const doubled = computed(() => count.value * 2)
```

### 5. v-if vs 条件渲染

```vue
<!-- Vue 3 -->
<template>
  <div v-if="show">内容</div>
  <div v-else>备选</div>

  <!-- key 复用问题 -->
  <div v-if="type === 'A'" key="a">A</div>
  <div v-else key="b">B</div>
</template>
```

### 6. v-for 和列表渲染

```vue
<template>
  <!-- 必须加 key -->
  <div v-for="item in items" :key="item.id">
    {{ item.name }}
  </div>

  <!-- 支持解构 + 索引 -->
  <div v-for="(item, index) in items" :key="index">
    {{ index }}: {{ item.name }}
  </div>
</template>
```

### 7. 双向绑定

**React**：受控组件模式
```jsx
<input value={value} onChange={e => setValue(e.target.value)} />
```

**Vue 3**：v-model
```vue
<!-- 单个值 -->
<input v-model="value" />

<!-- 组件上 -->
<ChildComponent v-model:title="pageTitle" />
```

### 8. 槽位 (Slots)

**React**：children + render props
```jsx
<Modal>
  {() => <div>内容</div>}
</Modal>
```

**Vue 3**：slot / 具名 slot
```vue
<Modal>
  <template #header>标题</template>
  <template #default>内容</template>
</Modal>
```

### 9. Vue 3 新特性 (React 没有的)

| 特性 | 说明 |
|------|------|
| **Teleport** | 传送组件到 DOM 任意位置 (模态框) |
| **Suspense** | 异步组件加载状态 |
| **Fragment** | 模板多根节点 |
| **setup script** | `<script setup>` 语法糖 |
| **provide/inject** | 跨层级数据传递 |
| **响应式调试** | `computed` / `watchEffect` 调试 |

### 10. 关键差异总结

| 维度 | React | Vue 3 |
|------|-------|-------|
| **学习曲线** | JSX 统一，概念少 | 模板 + API，概念多 |
| **灵活性** | 高度自由，一切皆 JS | 约定更多，模板有限制 |
| **性能** | 需要手动优化 (memo/useCallback) | 自动追踪，粒度更细 |
| **TypeScript 支持** | 初期复杂 | 内置 TS 支持 |
| **生态** | 更多选择 | 更统一 (官方方案) |

## 26. AI幻觉是什么？怎么检测？

## 答案
### 什么是 AI 幻觉 (Hallucination)

**定义**：LLM 生成的内容听起来合理、有条理，但实际上是**错误的、虚构的或与事实不符**的。

```
┌─────────────────────────────────────────────────────────────┐
│                    AI 幻觉类型                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ 1. 事实性幻觉 (Factual Hallucination)                        │
│    模型生成的内容包含错误的事实或统计数字                      │
│    例: "截至 2024年，重庆邮电大学有 3 万名学生" (实际是 ~2.5万) │
│                                                             │
│ 2. 归因性幻觉 (Attribution Hallucination)                   │
│    模型引用了不存在的来源、论文、网址                         │
│    例: "根据《Nature》2023年发表的研究..." (实际上没有这论文)  │
│                                                             │
│ 3. 逻辑性幻觉 (Logical Hallucination)                       │
│    推理过程看起来合理，但结论是错的                          │
│    例: 数学证明、代码逻辑错误                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 幻觉产生的原因

| 原因 | 说明 |
|------|------|
| **知识截止** | 训练数据有截止日期，无法感知最新信息 |
| **训练数据偏差** | 数据分布不均衡，某些领域知识不足 |
| **注意力机制** | 模型"过度自信"，将无关信息拼接 |
| **概率生成** | 本质是预测下一个 token，无法保证事实性 |
| **长上下文** | 上下文过长时，远距离信息被稀释 |
| **指令冲突** | 复杂指令导致模型"迎合"而非"回答" |

### 幻觉检测方法

#### 1. 基于置信度 (Confidence Score)

```python
# OpenAI API 返回的 logprob 不等于置信度
response = openai.ChatCompletion.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "..."}]
)

# 检查 logprob / finish_reason
# finish_reason == "length" 可能表示被截断
```

**局限性**：模型对自己答案的置信度 ≠ 答案的正确性

#### 2. 交叉验证 (Cross-Reference)

```python
# 同一个问题，多个模型分别回答
# 或者同一模型多次采样 (Temperature > 0)
# 对比答案一致性

def detect_by_consistency(prompt, model="gpt-4"):
    answers = [
        call_model(prompt, temperature=0.7)  # 采样多次
        for _ in range(3)
    ]
    # 一致性高 → 置信度高
    consistency = calculate_similarity(answers)
    return consistency > threshold
```

#### 3. RAG + 事实核查 (Fact-Checking)

```
用户问题
    ↓
RAG 检索相关文档
    ↓
LLM 生成答案 + 附带引用
    ↓
用检索到的文档验证 LLM 答案的事实性
    ↓
检测到不一致 → 标记为"不确定"
```

```python
# 验证步骤
def verify_answer(question, answer, context_docs):
    verification_prompt = f"""
    问题: {question}
    答案: {answer}
    参考文档: {context_docs}

    请判断答案是否与参考文档一致。如果不一致，指出具体错误。
    """
    return call_model(verification_prompt)
```

#### 4. 不确定性估计 (Uncertainty Estimation)

| 方法 | 说明 |
|------|------|
| **Self-Consistency** | 多路径推理取多数答案 |
| **Chain-of-Thought** | 让模型展示推理过程 |
| **Constitutional AI** | 让模型自我批评 |
| **P(True)** 估计 | 训练一个"真值探测器" |

#### 5. 结构化输出 + Schema 约束

#### 6. 实用工程实践

| 策略 | 实施方法 |
|------|----------|
| **不确定时坦白** | Prompt 要求"不确定时说不知道" |
| **引用 + 溯源** | RAG 强制要求答案附带文档引用 |
| **拒答机制** | 置信度低于阈值时返回"无法回答" |
| **人机协同** | 关键场景人工复核 |
| **持续监控** | 生产环境收集用户纠正反馈 |

### 面试加分点

> **前沿方法**：
> - **LLM-Decoder** 方案：训练专门的幻觉检测器
> - **Semantic Entropy**：基于语义熵的不确定性估计
> - **FreshPrompt**：将最新搜索结果注入 Prompt 减少过时幻觉
>
> **核心观点**：
> - 幻觉无法 100% 消除，只能降低概率
> - RAG 是目前最有效的工程缓解方案
> - 结合置信度 + 溯源 + 人工审核是生产级做法

#### 2. 文档分块 (Chunking)

| 策略 | chunk_size | overlap | 适用场景 |
|------|-----------|---------|----------|
| **固定长度** | 500 tokens | 50 tokens | 通用场景 |
| **语义分块** | 可变 | 可变 | 按段落/句子 |
| **层级分块** | 嵌套 | - | 长文档 |

```python
# 固定长度分块
def chunk_text(text: str, chunk_size: int = 500, overlap: int = 50):
    tokens = text.split()  # 简单 tokenize
    chunks = []
    for i in range(0, len(tokens), chunk_size - overlap):
        chunk = ' '.join(tokens[i:i + chunk_size])
        chunks.append(chunk)
    return chunks

# 5K 文档 → ~50K-100K chunks (按平均每文档 10-20 chunks)
```

### 二、多轮对话设计 (Multi-Turn RAG)

#### 1. Conversation History 管理

```python
class ConversationManager:
    def __init__(self, max_turns: int = 10):
        self.history: list[Message] = []
        self.max_turns = max_turns

    def add_user_message(self, content: str):
        self.history.append(Message(role='user', content=content))

    def add_assistant_message(self, content: str, context_docs: list):
        self.history.append(Message(
            role='assistant',
            content=content,
            context=context_docs  # 保存引用用于审计
        ))

    def get_context_window(self) -> list[Message]:
        # 只保留最近 N 轮，避免上下文膨胀
        return self.history[-self.max_turns * 2:]
```

#### 2. Query 改写 (Query Rewriting)

```python
def rewrite_query(original_query: str, history: list[Message]) -> str:
    """
    将多轮对话中的指代消解 + 意图补全
    例: "它的原理是什么？" → "React Fiber 的工作原理是什么？"
    """
    rewrite_prompt = f"""
    对话历史:
    {format_history(history)}

    当前问题: {original_query}

    请将当前问题改写为一个独立、完整的问题，包含足够的上下文信息。
    直接输出改写后的问题，不要解释。
    """
    return call_llm(rewrite_prompt)
```

#### 3. Hybrid Search (混合检索)

```python
# 向量检索 + 关键词检索结合
def hybrid_search(query: str, top_k: int = 5):
    # 1. 向量检索
    query_vector = embed_query(query)
    vector_results = collection.search(
        'embedding',
        query_vector,
        top_k=top_k * 2
    )

    # 2. BM25 关键词检索
    keyword_results = bm25_search(query, top_k)

    # 3. RRF 融合 (Reciprocal Rank Fusion)
    fused_results = rrf_fusion([
        vector_results,    # rank by vector similarity
        keyword_results    # rank by BM25 score
    ], k=60)

    return fused_results[:top_k]
```


### **核心代码实现**

```typescript
// src/lib/supabase/user/auth.ts
export async function SignInWithEmailAndPassword(email: string, password: string) {
  const { error } = await supabase.auth.signInWithPassword({ email, password })
  if (error) throw error
}

// src/hooks/use-current-user.ts
export default function useCurrentUser() {
  const [auth, setAuth] = useState<User>()

  useEffect(() => {
    // 1. 首次加载时获取 session
    supabase.auth.getSession().then(({ data: { session } }) => {
      setAuth(session?.user)
    })

    // 2. 订阅认证状态变化（登录/登出/Token刷新）
    const { data: { subscription } } = supabase.auth.onAuthStateChange((_event, session) => {
      setAuth(session?.user)
    })

    return () => subscription.unsubscribe()
  }, [])

  return auth
}
```

## **业界登录方案调研对比**

| 方案 | 代表技术 | 优点 | 缺点 |
|------|---------|------|------|
| **自建 JWT** | 传统 Session/JWT | 完全可控 | 实现复杂，需要处理刷新、过期、黑名单 |
| **Supabase Auth** | Supabase JWT | 开箱即用，集成简单 | 依赖第三方 |
| **Auth0/Clerk** | SaaS Auth | 功能完善，UI 丰富 | 成本高，有厂商锁定 |
| **NextAuth.js** | Next.js 生态 | 适配 React 生态 | 仅限 Next.js |

### **选择 Supabase Auth 的原因**

1. **与 Supabase 生态深度集成**
   - 数据库、实时功能、存储统一认证
   - 一个 SDK 管理所有后端交互

2. **安全性保障**
   - JWT 自动刷新
   - 过期 Token 自动处理
   - RLS (Row Level Security) 与 JWT 绑定

3. **开发效率**
   - 登录/注册/登出一行代码
   - 内置邮箱验证、密码重置
   - 支持 OAuth 第三方登录扩展
