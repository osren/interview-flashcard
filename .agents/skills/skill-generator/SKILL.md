# Skill 生成器

帮助用户按照 Claude Code Skill 规范快速创建、修改和优化 skill 文件。

## 核心流程

创建 skill 的整体流程：

1. **理解意图** — 明确 skill 要做什么、何时触发、输出什么
2. **采访调研** — 深入了解边界情况、输入输出格式、成功标准
3. **编写 SKILL.md** — 根据规范生成 skill 文件
4. **测试验证** — 创建测试用例并运行验证
5. **迭代改进** — 根据反馈优化 skill

---

## 一、Skill 目录结构

每个 skill 必须包含以下结构：

```
skills/
└── {skill-name}/
    ├── SKILL.md           # 主规范文件（必需）
    └── references/        # 参考文档目录（可选）
        ├── xxx.md
        └── ...
```

### 三级加载机制

Skill 使用三级加载系统，理解这个机制有助于设计更好的 skill：

| 层级 | 内容 | 加载时机 | 建议 |
|------|------|----------|------|
| **Metadata** | name + description | 始终加载 (~100 词) | description 要精准、pushy |
| **SKILL.md body** | Markdown 正文 | skill 触发时加载 | 控制在 500 行以内 |
| **Bundled resources** | references/ 下的文件 | 按需加载 | 无限制，用于详细文档 |

**关键原则**：
- SKILL.md 控制在 500 行以内，过长时拆分到 `references/`
- 在 SKILL.md 中明确指引何时读取哪个 reference 文件
- 大型 reference 文件（>300 行）应包含目录

## 二、编写 SKILL.md

### Frontmatter（YAML 格式）

```yaml
---
name: skill-name              # skill 标识符，小写，用 - 连接单词
description: skill 描述，说明何时使用、触发关键词、核心能力
---
```

**description 编写要点**（非常重要！）：

1. **要"pushy"一些**：Claude 倾向于"undertrigger" skill，即该用的时候不用。所以 description 要主动一些。
   
   ❌ 太被动：
   > "如何构建一个简单的仪表盘来显示内部数据。"
   
   ✅ 更主动：
   > "如何构建一个简单的仪表盘来显示内部数据。只要用户提到仪表盘、数据可视化、内部指标，或想展示任何公司数据，都应该使用此 skill，即使没有明确说『仪表盘』。"

2. **包含触发场景**：说明在什么上下文、什么关键词下应该触发
3. **包含核心能力**：简要说明 skill 能做什么
4. **包含同义词/变体**：用户可能使用的不同表述方式

### 正文结构

```markdown
# Skill 标题

简短描述 skill 的作用（1-2 句话）。

## 核心能力

- 能力 1
- 能力 2

## 工作流程

### Step 1: 步骤名称

具体操作说明...

### Step 2: 步骤名称

具体操作说明...

## 输出格式

ALWAYS use this exact template:
# [标题]
## 章节 1
## 章节 2

## 注意事项

- 注意点 1（解释为什么重要）
- 注意点 2

## References 说明（如有）

| 文件 | 说明 | 何时读取 |
|------|------|----------|
| xxx.md | 说明 | 何时需要 |
```

### 编写风格指南

1. **使用祈使句**：「执行 X」「检查 Y」「输出 Z」
2. **解释"为什么"**：不仅告诉做什么，还要解释为什么重要
3. **避免过度约束**：少用「MUST」「ALWAYS」「NEVER」等强制词汇，用解释代替命令
4. **提供示例**：在关键步骤提供代码片段、curl 命令等示例

**示例模式**：

```markdown
## 提交信息格式

**示例 1：**
输入：添加了基于 JWT 的用户认证
输出：feat(auth): implement JWT-based authentication

**示例 2：**
输入：修复了登录页面的样式问题
输出：fix(login): resolve style issues on login page
```

## 三、创建流程

### Step 1: 捕获意图

如果当前对话已包含用户想要的 workflow（例如用户说「把这个转成 skill」），从对话历史中提取：
- 使用的工具
- 步骤顺序
- 用户的修正
- 观察到的输入输出格式

用户需要确认的信息：

1. **这个 skill 要让 Claude 做什么？**
2. **何时应该触发？**（什么用户短语/上下文）
3. **预期的输出格式是什么？**
4. **是否需要设置测试用例？**
   - 有客观可验证输出的 skill（文件转换、数据提取、代码生成、固定流程）受益于测试用例
   - 主观输出的 skill（写作风格、艺术创作）通常不需要
   - 根据类型建议合适的默认值，但让用户决定

### Step 2: 采访调研

主动询问关于以下方面的问题：

- **边界情况**：哪些特殊情况需要处理？
- **输入输出格式**：期望什么样的输入？输出应该是什么样？
- **示例文件**：有没有现成的示例可以参考？
- **成功标准**：怎么判断 skill 工作正常？
- **依赖项**：是否依赖特定的 MCP、工具或环境？

**在编写测试用例之前先搞清楚这些问题。**

### Step 3: 编写 SKILL.md

根据收集的信息填充以下组件：

- **name**：Skill 标识符
- **description**：何时触发、做什么。这是主要的触发机制，要写得主动一些
- **compatibility**：必需的工具、依赖（可选，很少需要）
- **正文内容**：按照上面的正文结构编写

### Step 4: 创建 references（如需要）

如果 SKILL.md 超过 500 行，或包含可独立加载的详细文档：

```
skill-name/
├── SKILL.md (workflow + selection)
└── references/
    ├── topic-a.md
    └── topic-b.md
```

在 SKILL.md 中明确指引何时读取哪个文件：

```markdown
如需详细了解 AWS 部署流程，请读取 `references/aws.md`。
如需详细了解 GCP 部署流程，请读取 `references/gcp.md`。
```

## 四、测试与验证

### 创建测试用例

编写 2-3 个真实用户会说的测试提示。分享给用户确认：

> 这里是几个我想尝试的测试用例，这些看起来对吗？还是想增加一些？

测试用例格式：

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示",
      "expected_output": "预期结果的描述",
      "files": []
    }
  ]
}
```

### 验证检查清单

生成后，检查以下内容：

- [ ] Frontmatter 格式正确（YAML 语法）
- [ ] description 包含触发关键词和场景
- [ ] description 足够"pushy"（防止 undertriggering）
- [ ] 工作流程步骤完整，每步有明确输入输出
- [ ] 提供了示例（代码片段、命令等）
- [ ] 注意事项解释了"为什么"
- [ ] SKILL.md 控制在 500 行以内
- [ ] 如有 references，在正文中明确指引何时读取

## 五、迭代改进

根据测试结果和用户反馈改进 skill：

### 改进原则

1. **从反馈中泛化**：不要针对具体测试用例打补丁，要理解背后的通用问题
2. **保持精简**：删除不必要的内容，减少 token 浪费
3. **解释"为什么"**：用解释代替命令，让模型理解意图
4. **识别重复工作**：如果多个测试用例都在做类似的辅助工作，考虑将其打包成脚本

### 迭代循环

1. 应用改进到 skill
2. 重新运行测试用例
3. 等待用户评审
4. 阅读新反馈，继续改进
5. 重复直到用户满意

## 六、Description 优化（高级）

如果 skill 触发不准确，可以优化 description：

### 创建触发评估查询

生成 20 个评估查询 — 混合"应该触发"和"不应该触发"：

```json
[
  {"query": "用户提示内容", "should_trigger": true},
  {"query": "另一个提示", "should_trigger": false}
]
```

**好的查询特点**：
- 真实、具体、有细节（文件路径、上下文、具体名词）
- 包含不同风格（正式/随意、长/短）
- 边界情况（而非显而易见的案例）

**应该触发 (8-10 个)**：
- 不同表述方式
- 用户没有明确说出 skill 名称但明显需要它
- 与其他 skill 竞争但应该胜出的场景

**不应该触发 (8-10 个)**：
- 共享关键词但实际需要不同的功能
- 模棱两可的表述
- 触及相关功能但其他工具更合适的场景

## 七、最佳实践总结

| 方面 | 最佳实践 |
|------|----------|
| **description** | 主动、pushy、包含触发场景和关键词 |
| **正文长度** | SKILL.md < 500 行，过长拆分到 references |
| **编写风格** | 祈使句、解释"为什么"、提供示例 |
| **输出格式** | 明确定义输出模板和格式 |
| **测试** | 创建真实测试用例，验证触发和输出 |
| **迭代** | 根据反馈泛化改进，保持精简 |
