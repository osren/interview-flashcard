## 任务：将产品文档解析为结构化 proposal.md

## 输入

调用方需传入：
- **【feature 目录】**：如 `docs/specs/active/{name}/`（由 `/oh:propose` 传入）
- **【产品文档】**：用户提供的文档路径或内容（若未提供，进入交互询问）

## 工作流

### Step 1: 获取产品文档输入

如果调用方未传入产品文档路径，使用 AskUserQuestion 询问：

> 请提供产品文档。支持以下方式（可多选组合）：
> 1. Word 文件路径（.docx）
> 2. PDF 文件路径（.pdf）
> 3. 截图/流程图路径（.png/.jpg）
> 4. 直接粘贴文字
>
> 注意：在线文档（Cooper/Confluence/飞书/Notion）无法直接访问，请先导出为 Word 或 PDF，或复制粘贴内容。

### Step 2: 按格式分别处理输入

#### 2a. Word 文件（.docx）

```bash
which pandoc
```

**pandoc 可用时**：
```bash
mkdir -p {FEATURE_DIR}/media
pandoc "<file>.docx" -o "{FEATURE_DIR}/content-raw.md" --extract-media="{FEATURE_DIR}/media"
```

- Read 转换后的 `content-raw.md` 提取文字信息
- Glob `{FEATURE_DIR}/media/**` 查找提取出的图片
- 对每张图片用 Read 读取并理解内容（见 Step 2c）

**pandoc 不可用时**：
- 提示用户：`pandoc 未安装，请执行 brew install pandoc，或将 Word 另存为 PDF 后重试。`
- 停止处理此文件

#### 2b. PDF 文件（.pdf）

- 超过 20 页时分批读取（每次 20 页），不跳过任何页面
- Read PDF 文件，提取文字信息
- 解析结果混乱时提示：`PDF 解析质量不佳，建议将关键页面截图提供，或复制粘贴文字内容。`

#### 2c. 图片文件（.png/.jpg）

用 Read 工具读取图片，按类型提取信息：

| 图片类型 | 提取内容 |
|---------|---------|
| 流程图/状态机 | 节点、转换条件、完整流程描述 |
| UI 原型/交互稿 | 页面元素、交互行为、字段列表 |
| 架构图 | 组件、调用关系、数据流向 |
| 表格/文字截图 | 文字内容 |
| 手绘白板 | 尽力识别，识别不清的明确标注 |

- 提取结果标注 `<!-- FROM_IMAGE: 源文件名，请确认 -->`
- 无法完全理解时主动告知用户具体哪部分不确定

#### 2d. 粘贴文字

直接作为原始内容处理，无需转换。

### Step 3: 合并所有来源

- 多个来源有**重复信息**：取最详细的版本
- 多个来源有**冲突信息**：列出冲突点，使用 AskUserQuestion 询问用户以哪个为准
- 保留来源标注，方便追溯

### Step 4: 结构化提取

从合并内容中提取以下信息。**提取不到的不要编造，标注"原始需求未说明"**：

a. **背景与目标**：为什么有这个需求，来源是哪里
b. **核心功能点**：逐条列出，每条一个独立功能
c. **用户场景 / 业务流程**：正常流程 + 异常流程
d. **数据模型**：涉及的实体、字段、关系
e. **接口需求**：对外暴露或需对接的接口
f. **非功能需求**：性能、安全、兼容性、数据量级
g. **边界与约束**：本次做什么、不做什么
h. **验收标准**：怎么算做完了

### Step 5: 完整性检查

用以下清单检查提取结果，缺失的收集到"待澄清项"：

- [ ] 异常场景覆盖了吗？（超时、失败、重试）
- [ ] 并发/幂等考虑了吗？
- [ ] 数据量级/性能要求明确了吗？
- [ ] 权限/安全要求明确了吗？
- [ ] 上下游系统的接口定义清楚了吗？
- [ ] 涉及的领域术语都有明确定义吗？
- [ ] 数据迁移/兼容性要求明确了吗？

待澄清项超过 3 条时，使用 AskUserQuestion 一次性询问用户。用户无法回答的保留在待澄清项中（后续在 design.md 或 spec.md 中继续收敛）。

### Step 6: 生成 proposal.md

输出到 `{FEATURE_DIR}/proposal.md`（FEATURE_DIR 为调用方传入的 feature 目录）：

```markdown
---
feature: {name}
created: {YYYY-MM-DD}
status: draft
source: {来源文件列表}
---

# {需求名称} — Proposal

> 原始文档：{来源列表}

## 一、背景与目标

{提取的背景信息}

## 二、核心功能点

1. **功能点 A**：{描述}
2. **功能点 B**：{描述}

## 三、用户场景 / 业务流程

### 正常流程
{文字描述或 ASCII 流程图}

### 异常流程
{文字描述，原始需求未说明的标注}

## 四、数据模型

| 实体 | 关键字段 | 说明 |
|------|---------|------|

> 原始需求未说明的字段标注"原始需求未说明，待 design/spec 收敛时确认"

## 五、接口需求

| 接口 | 方向 | 说明 |
|------|------|------|

## 六、非功能需求

| 维度 | 要求 |
|------|------|
| 性能 | {原始需求未说明 / 具体要求} |
| 安全 | |
| 兼容性 | |
| 数据量级 | |

## 七、边界与约束

- **本次做**：...
- **本次不做**：...
- 限制条件：...

## 八、验收标准

- [ ] {AC-001}: {具体可测量的验收条件}

## 九、待澄清项

- [ ] {问题 1}（来源：{文档节次 / 完整性检查}）

## 十、信息来源追溯

| 章节 | 来源 | 备注 |
|------|------|------|
| 业务流程 | {来源文件} | |
| 流程图 | {图片文件名} | <!-- FROM_IMAGE: 请确认 --> |
```

临时文件清理：若生成了 `{FEATURE_DIR}/content-raw.md`，处理完成后删除。

### Step 7: 输出解析报告

```
proposal.md 已生成：{FEATURE_DIR}/proposal.md

信息来源：
- {文件名}（{格式}）
- ...

提取到：
- N 个核心功能点
- M 个接口需求
- K 个待澄清项

⚠ 以下内容从图片提取，请重点确认：
- {具体章节}（来自 {图片文件名}）
```

## 约束

- 提取不到的信息标注"原始需求未说明"，**绝不编造需求**
- 从图片提取的信息必须标注 `<!-- FROM_IMAGE -->`
- Word 转换依赖 pandoc，不可用时明确提示安装方式
- PDF 超过 20 页必须分批读取，不跳过
- 多来源冲突时必须询问用户，不自行判断
- 在线文档明确告知无法访问，引导用户导出
- proposal.md 是后续 spec.md 与 design.md 生成的输入，用户可直接编辑修改
